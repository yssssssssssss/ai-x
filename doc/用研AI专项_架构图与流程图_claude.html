用研 AI 专项 · 架构图与流程图

底座搭建方案 · _claude
① 系统架构 ② 四段协作流 ③ 渐进加载 ④ 数据模型 ⑤ 功能架构

一条主原则:业务判断交给 LLM + 配置(放 Git/YAML),执行边界与质量门禁交给后端(schema/权限/风控/审计/状态流转/版本追溯),运行时数据放 DB。 下面四张图分别覆盖架构(图 ①④)与流程(图 ②③)两侧,均基于权威方案文档 doc/用研AI专项_底座搭建方案_优化版_claude.md。

01系统架构 · 分层组件与数据流

三个真实组件:agent-api(统一入口)、Orchestrator(编排运行时,全项目最重)、Research Memory DB(只存运行时数据)。 判断力全在配置层,后端不含业务 if-else。

02四段协作运行流程 · 计划→确认→执行→交付

核心是段 2 后的硬闸门:未确认不执行。段 3 每步按风险走审批 / 标注 / 阻断三分支,失败入 failures.jsonl 可断点续跑。

03能力发现与三层渐进加载 · 路由机制

YAML 只负责"找到谁",SKILL.md 负责"怎么做"。先发现 → 再理解 → 最后执行,避免一次性把全量能力塞进上下文。 

flowchart LR
  Req["用户需求"] --> Task["LLM 结构化
ResearchTask"]
  Task --> L1["① 轻量索引
skill/tool/decision-graph 摘要"]
  L1 --> Node["激活决策节点
task_type 分流·略过无关"]
  Node --> Filter["规则粗筛
候选 skill/tool"]
  Filter --> L2["② 候选能力
加载 2-5 个 SKILL.md"]
  L2 --> Plan["LLM 生成
待确认计划"]
  Plan --> Confirm{"⏸ 用户确认"}
  Confirm --> L3["③ 执行期资源
schema/examples/templates/adapter"]
  L3 --> Call["skill 经 tool-adapter 调 tool"]
  Call --> Log["写 execution_log + artifacts
留 manifest/prompt/model hash"]

05功能架构 · 需求分析 → 差异化能力组合

同一入口进,LLM 先把一句话解析为 task_type / business_domain / 完备性节点,再按不同需求类型激活不同的决策节点子集并命中对应的 skill/tool 组合——竞品任务不出现"用户人群"节点,用户之声不出现"竞品"节点。能力目录持续可增,命中逻辑全由 LLM + 配置驱动,不写死分支。

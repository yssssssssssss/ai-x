# 用研 AI 专项 · 项目完成度与可用性分析报告

> 评估日期:2026-07-18　评估口径:基于代码实测,不粉饰、不贬低
> 一句话结论:**"能跑通的骨架"已扎实建成,"能真替代团队的产品"尚未到位。钱花在了正确的地基上,但离交付真实价值还有一段明确、可规划的距离。**

---

## 〇、为什么要"双轨"评估

一个 AI Agent 产品有两条独立的完成度轴,混为一谈会误导决策:

- **A 轴 · 平台/架构完成度**:编排引擎、状态机、容错、可追溯、前端交互——"骨架和血管"。
- **B 轴 · 真实能力完成度**:工具能否取到真实数据、skill 是否为经验证的方法、结论能否被业务采纳——"血肉"。

**本项目 A 轴高、B 轴低**。这不是缺陷,是早期项目的正常形态——先修骨架再长血肉。但**对外不能只报 A 轴的分**,否则一旦被要求"真跑一个课题",会当场穿帮,反噬信任。本报告把两轴分开打分。

---

## 一、结论摘要(TL;DR)

| 维度 | 完成度 | 判断 |
|---|---|---|
| **A 轴:平台/架构** | **~75%** | 编排四段流、候选选择、步级容错、SSE 流式、schema 校验、全链留痕都已建成且有测试。工程质量扎实。 |
| **B 轴:真实能力** | **~25%** | 工具全部拿不到真实数据(o2 假数据 / 5 个 lab 服务未起);22 个 skill 实体虽在,但多数是 20–25 行薄骨架;无真实用户触达。 |
| **可交付状态** | **Demo-ready,非 Production-ready** | 能完整演示"流程与体验",不能保证"结论真实可信可采纳"。 |

**代码规模**:自研 ~4900 行(编排 2417 / 前端 1349 / DB 491 / API 415),22 个测试文件,30+ 次提交、演进脉络清晰。**这不是 PPT 项目,是有真实工程投入的项目。**

---

## 二、A 轴 · 已扎实交付的部分(给信心)

这些是真做出来、且经测试验证的能力,值得肯定:

1. **四段编排闭环**:理解 → 候选计划 → 选择 → 执行 → 报告,状态机完整。
2. **双候选方案 + HITL 闸门**:一次产出"深度/速度"两份可对比方案,用户选一份再执行;确认前不落地——决策权留给人。
3. **步级容错(本轮新增)**:某步失败不再整链崩,停在该步 `paused`,用户可"跳过续跑 / 终止";缺口如实进报告 `risks_and_open_issues`。这是"能长期运行"的关键工程能力。
4. **真实阶段流式(SSE)**:规划过程逐阶段推送(理解→激活→召回→判定→候选),用户看得见后台在做什么。实测为真流式(各阶段随真实 LLM 耗时依次到达)。
5. **可信工程底座**:所有 LLM/tool 输出过 JSON Schema 校验;幻觉能力 id 在计划阶段拦截;全链 `execution_log` + `context_manifest` 留痕可复盘;RunWorkspace 落盘可回放。
6. **知识库引导召回**:planPhase 按决策节点 `related_tags` 从知识库召回方法论喂给 LLM,让"选方法"有据。
7. **测试与类型**:全量 74 pass / 0 fail,根 + web `tsc` 干净。

> 小结:**架构判断力是这个项目最大的资产**。它没有走"硬编码 if task_type"的死路,而是数据驱动(决策节点池 + registry + schema),这决定了它未来能横向扩展。

## 三、B 轴 · 真实可用性缺口(诚实的部分)

必须如实说明——现在**没有一个工具能产出真实数据**:

### 3.1 工具层:0 个真实产出
| 工具 | 通道 | 现状 |
|---|---|---|
| o2-web-search(最常被选中) | fake | **返回 3 条硬编码假数据** |
| 5 个能力实验室(美学/注意力/体验/虚拟用户/品牌) | rest_json | **服务端口 8801–8805 全部未启动** |
| ai-spider-search | internal_api | 需内网登录,未验证 |

后果:现在真跑一个竞品研究,检索步拿到的是假数据、lab 步会失败——**报告"格式真、数据假"**,这是最危险的一类不可信。

### 3.2 方法层:skill 实体在,但"薄"
- 22 个 skill 的 SKILL.md **确实都存在**(在 `knowledge-base/skills/`,body_ref 已正确指向)——此前"实体缺失"的担忧已排除。
- 但多数是 **20–25 行的 KB 派生薄骨架**(`kb_derived`),仅 `competitive-analysis`(54行)、`design-experience-review`(65行)相对充实。
- skill 执行本质是"LLM 读 SKILL.md + 基于 tool_outputs 产出"——**骨架薄 = 方法论指导弱 = 产出靠 LLM 自由发挥**,不是经验证的用研方法。且 registry 的 `evals_ref` 尚无实际 eval,质量无回归保障。

### 3.3 真实用户触达:为零(硬边界)
用研的定性/定量核心(访谈、问卷、可用性测试)需要触达真实用户。当前完全没有,也无样本库/panel 接入。这部分存在**不可凭 agent 凭空完成的硬边界**——终态必然是"agent 编排 + 关键采集接真实源 / 人在环"。

## 四、客观定位:它现在是什么

- **它是**:一个架构扎实、可演示完整流程、工程质量在线的**用研 Agent 平台骨架 + 编排引擎**。
- **它不是**(现在还不是):一个能独立产出可被业务采纳结论、真替代团队完成课题的**成品**。

用一个比喻:**造好了一条设计合理、有安全冗余的生产线,但原料(真实数据)还没接管道,工艺卡(skill 方法)还是草稿**。生产线本身是硬功夫,不该被低估;但不能对外宣称"已能量产"。

---

## 五、风险提示(与预算决策直接相关)

1. **过度承诺风险**:若现在对领导演示时宣称"已能真替代团队",一旦被要求现场真跑课题会穿帮,信任反噬远大于诚实汇报。**建议演示口径统一为"流程与架构已通,正在接入真实数据与方法"**。
2. **"最后一公里"被低估的风险**:B 轴(接真实源 + 方法写实 + 真实用户触达)工作量不小于已完成的 A 轴,且部分依赖组织提供的资产(样本库、内部数据接口)。预算规划要为这段留足。
3. **registry 虚高风险**:22 个 skill 中多数薄骨架却标 `active`,规划器会把它们排进方案,执行时产出质量参差。需要"写实一个、激活一个"的治理纪律。

---

## 六、结论与预算建议

**该不该继续投:该。** 理由:
- 架构方向正确(数据驱动、可扩展、可容错、可追溯),这是最难、最值钱、也最容易被外行忽视的部分,已经做对了。
- 已完成部分是真实工程投入(~4900 行 + 测试 + 清晰演进),不是空壳。

**但要投在刀刃上:先做"单场景真实打通"验证价值,再谈规模化。**

建议把下一笔预算**锚定一个可验收的里程碑(M1)**,而非泛泛"继续开发":

> **M1 · 竞品研究单场景真实打通**(理由:唯一有较充实 skill、只依赖公开信息检索、不需要真实用户源)
> - 接通 1 条真实检索通道(替换 o2 假数据)
> - 把该场景 3–4 个 skill 从薄骨架写实 + 配 eval
> - 报告结论可溯源到真实证据
> - **验收:业务专家盲评"可采纳" ≥ 70%**

M1 达成 = 用最小成本证明"这套架构能产出真实可信的用研结论"。这是从"能演示"到"能真用"、也是从"要预算"到"值预算"的关键一跃。M1 不成立,则不宜盲目铺开 7 个 lab、30 个 skill。

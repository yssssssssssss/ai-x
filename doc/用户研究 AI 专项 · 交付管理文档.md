# 用户研究 AI 专项 · 交付管理文档

一句话讲清楚这个东西

"用研专项" = 拉齐设计同学对于用研的共识 总结提炼出 一个懂用研的资深老同事，你调用他,你告诉它"我要做 X 的用研",它告诉你"你该找谁 + 用什么方法 + 参考什么"。

我们要建的不是新的工具,是把散落在各团队的用研能力,聚合成一个可以被调用的系统

## 一、为什么要建这个专项

### 1.1 现在的三个真实痛点

| 痛点 | 通俗讲 |
| --- | --- |
| 每次用研从 0 起 | 新项目要做用研,研究员翻邮件找模板 / 问同事经验 / 拼凑方法 |
| 各团队能力是孤岛 | 数字人的竞品工具、Dacong 哥的人群审查、绍安的眼动 review、雷晓鸣的用户之声 — 每家都有好东西,但没有一个入口把它们串起来 |
| 新人不知道找谁 | 一个新设计师要做用研,第一反应是"我该找谁?看哪份文档?" — 没有答案 |

### 1.2 建成之后是什么样

新设计师小 A 要为直播场域做用户体验研究。

**没有专项之前**:

- 问 3-5 个人 → 拼几个模板 → 摸索 2 周 → 才开始做正事

**有了专项之后**:

- 在 wiki 里输入"直播场域用户体验研究" → 系统输出:"你需要走 7 个决策问题,推荐用 X + Y + Z 三个工具,联系 A / B / C 三位同事,参考近期 W 项目的做法"
- 小 A 30 分钟出方案 → 直接开工

**关键**:不是替代用研老师,是让**新人不用再"从 0 摸索"**。

## 二、专项框架长什么样 · 4 个层次

分成 4 层,每层各司其职:

### 2.1 第 1 层: Wiki 知识底座 (所有过去的经验)

存"用研这件事的知识"。

| **存什么** | **举例** |
| --- | --- |
| 用研方法论 | 访谈怎么做 / 焦点小组的坑 / 日记研究啥时候用 |
| 用户人群标签 | 京东用户群分类(dacong 团队 own) |
| 正例库 | 我们做得好的案例 |
| 反例库 | 我们做错过什么(避免再犯) |
| 反模式库 | "这些做法看起来对,其实是错的" — 集中警示 |
| 术语表 | 内部黑话统一(减少误解) |
| 历史项目档案 | 过去做过的用研项目全存档 |

**这一层的价值**:任何新项目起步时,可以先"查底座"看有没有类似做过的。

### 2.2 第 2 层:能力库(Skill 库)

存"做用研的具体能力"。

每个 Skill 就是"完成一件具体事情的能力包"。**颗粒度要小**,越小越准。

| 分类 | 具体 Skill 举例 |
| --- | --- |
| 研究流程系 | 研究启动 · 访谈准备 · 数据采集 · 分析聚类 · 报告生成 |
| 竞品分析系 | UI 竞品分析 · 功能竞品分析 · 商业化竞品分析 · 数字人竞品(何云深) |
| 用户之声系 | VoC 采集 · VoC 分析 · VoC 报告(雷晓鸣) |
| 走查系 | UI 走查 · 分人群审查(dacong)· 眼动 review(绍安) |
| 设计分析系 | 用户旅程 · 可用性测试 · AB 分析 |

**关键决策**:**每个 Skill 都是一个文件夹**,里面包含 6 个固定文件(SKILL.md / README / 正例 / 反例 / 反模式 / 参考资料)。

### 2.3 第 3 层:调度中心(决策树)⭐ 本手册核心

存"如何决定用哪些能力"的判断力。

它的形态是**一棵决策树**(下一段详细讲)。

**为什么这一层最关键?** — 没有它,前两层只是散落的资料。有了它,才能真正"活起来"。

### 2.4 第 4 层:反馈机制(持续学习)

存"如何持续改进"的机制。

| 功能 | 用什么 |
| --- | --- |
| 项目进度管理 | JS多维表格 |
| Skill 迭代记录 | 每次改进都留痕 |
| 周报模板 | 让大家汇报有统一格式 |
| 自动同步 | Wiki 改了 → Js表格自动更新(减少手工填) |

## 三、决策树 · 专项系统怎么工作 ⭐ 本手册核心

这是"用研专项"的**核心调度机制**。任何一个用研任务进来,都会顺着这 **7 个问题**问一遍,决定该干什么。

### 3.1 决策树全貌

```
目标:完成一个设计相关的用研任务

├── 🎯 问题 ① 研究目标是否清晰?
│   └── 找 用研老师 / PMO(Shaka)校准
│       → 触发 skill:research-kickoff(研究启动)

├── 👥 问题 ② 用户人群是否定义清楚?
│   └── 找 dacong / 用研老师 校准
│       → 触发 skill:audience-tagging(人群标签)

├── 🔬 问题 ③ 用什么研究方法?
│   └── 找 用研老师 / 翟又仪(用研 Wiki)校准
│       → 触发 skill:method-selection(方法选择)

├── 🎨 问题 ④ 用户体验现状清楚吗?
│   ├── 设计侧走查:Shaka / dacong / 绍安老师(眼动)
│   └── 用户侧监听:雷晓鸣(用户之声)
│       → 触发 skill:design-audit + voc-monitoring

├── 🔍 问题 ⑤ 竞品参照要不要?
│   └── 找 何云深(数字人竞品) / 雷晓鸣(线上竞品)校准
│       → 触发 skill:ui-competitive / business-competitive

├── 📊 问题 ⑥ 采集分析工具到位吗?
│   └── 找 刘昭然 + 李艺(用研小工具 / 走查归档)校准
│       → 触发 skill:data-collection + analysis-clustering

└── 📝 问题 ⑦ 产出物符合标准吗?
    └── 找 PMO(报告模板) / 翟又仪(Wiki 沉淀)校准
        → 触发 skill:report-generation + wiki-sedimentation
```

### 3.2 每个问题的运作机制

| 决策问题 | 找谁校准 | 触发的 Skill | 参考哪一层的知识 |
| --- | --- | --- | --- |
| ① 研究目标 | 用研老师 + PMO | research-kickoff | 方法论 wiki + 历史项目档案 |
| ② 用户人群 | dacong + 用研老师 | audience-tagging | 人群标签库 + 正例库 |
| ③ 研究方法 | 用研老师 + 翟又仪 | method-selection | 方法论 wiki |
| ④ 体验现状 | Shaka + dacong + 绍安 + 雷晓鸣 | design-audit + voc-monitoring | 反模式库 + 人群标签库 |
| ⑤ 竞品参照 | 何云深 + 雷晓鸣 | ui-competitive / business-competitive | 竞品案例库 |
| ⑥ 采集分析 | 刘昭然 + 李艺 | data-collection + analysis-clustering | 术语表 + 方法论 wiki |
| ⑦ 产出标准 | PMO + 翟又仪 | report-generation + wiki-sedimentation | 报告模板 + 历史档案 |

### 3.3 决策树的关键特点

| 特点 | 通俗讲 |
| --- | --- |
| **不是流程图** | 7 个问题可以**同时评估**,不是"必须走完前一个才能下一个" |
| **按需触发** | 不是每个问题都要用 skill · 用不着的直接跳过 |
| **可以生长** | 未来加新问题(比如"要不要生理心理指标?")或加新 skill |
| **能力聚合器** | 让 10 个团队的能力**组合起来用**,而不是各干各的 |

## 四、真实场景演示 · 直播场域用户体验研究

**输入**:小 A 打开Agentwiki 驱动Agent, 用户 输入"我要为直播场域做一次用户体验研究"

**系统走 7 个问题**:

| 决策 | 系统的输出 |
| --- | --- |
| ① 研究目标 | 提升直播用户购买决策效率 · 已由 PMO 校准 ✅ |
| ② 用户人群 | 直播频繁用户 25-40 岁 · dacong 校准 ✅ |
| ③ 研究方法 | 焦点小组(N=8)+ 日记研究(N=15)· 用研老师推荐 ✅ |
| ④ 体验现状 | 走查现有直播设计(Shaka)+ 眼动 review(绍安)+ 近 30 天用户之声(雷晓鸣)✅ |
| ⑤ 竞品参照 | 抖音 / 快手 / 淘宝直播的 UI 竞品分析(何云深)✅ |
| ⑥ 采集分析 | 焦点小组录音转写 + 主题聚类(刘昭然 + 李艺)✅ |
| ⑦ 产出标准 | research-report.md · 沉淀至 Wiki 直播板块 ✅ |

**最终输出**:一份完整研究方案 → 涉及 N 个 skill 组合 → 5 个团队协作 → 8 周产出

**这就是"用研专项"的核心价值** — 你只需要给一句话,系统给你完整方案。

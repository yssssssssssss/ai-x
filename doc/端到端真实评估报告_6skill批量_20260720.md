## 🔧 关键 bug 发现 · domain 语义使用错误

### 病灶

上一轮我给 8 个纯用研 skill(analyze-satisfaction / build-experience-metrics / code-open-feedback / conversion-funnel-analysis / feature-adoption-analysis / issue-prioritization / structure-interview-transcript / synthesize-qualitative-insights)补了 `domain: [user_research]`。

### 而 planPhase 段2c 的过滤逻辑([orchestrator.ts:227](apps/orchestrator-runtime/src/orchestrator.ts))是:

```ts
const relevantSkills = allSkills.filter(
  (s) => (s.domain ?? []).includes(task.task_type) ||
         (s.domain ?? []).includes('cross_cutting')
);
```

**匹配的是 `task.task_type`**(合法值:`user_research_planning / voc_diagnosis / competitive_research / design_audit / a11y_audit`),不是"主题分类"。

### `user_research` 不是合法 task_type

- `[user_research]` **既不匹配任何 task_type,也不含 `cross_cutting`**
- 所以这 8 个 skill 从未真正进 `relevantSkills`
- 之前能被 activeSkills 见到,只因"空回退保底":`relevantSkills.length > 0 ? relevantSkills : allSkills`
- **一旦有其他 skill 命中(如今天的 case 3/5/6),这 8 个立刻被过滤掉**

### 修法(下一轮做,不在本报告范围)

把 8 个 skill 的 `[user_research]` 改为覆盖真实 task_type 的组合:
```yaml
# 覆盖 user_research_planning + voc_diagnosis 两个场景
domain: [user_research_planning, voc_diagnosis]
# 或者最保守:加 cross_cutting 让它们始终在池
domain: [user_research_planning, voc_diagnosis, cross_cutting]
```

同理需检查 `[design_audit, user_research]` 等混合 domain——`user_research` 都应删掉。

## ⚠️ 意外发现 2 · task_type 分类偏差

case 4/5/6 query 都是"给了访谈/评价/数据"→ LLM 判 `task_type=voc_diagnosis`(不是 user_research_planning)。这在语义上**其实是对的**:VOC = Voice of Customer,涵盖满意度/访谈/评价分析。但 8 个"纯用研"skill 里,`analyze-satisfaction / code-open-feedback` 才有 `task_types: [voc_diagnosis]`,其他没有——task_type 分类和 skill 覆盖有轻微错位。

`journey-map` 靠 `cross_cutting` domain 兜底才被选中——这**正好验证 cross_cutting 兜底机制的价值**。

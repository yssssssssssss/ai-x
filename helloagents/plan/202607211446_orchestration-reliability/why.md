# 变更提案: 编排可靠性与证据化报告

## 需求背景

当前用研任务链路可以完成工具调用、Skill 分析和报告生成，但运行时审计确认了四类会直接影响结果可信度的问题：

1. 文本模型实际只配置了 `GPT-5.4-joybuilder -> GPT-5-joybuilder` 两级顺序切换，`GPT-5.2-joybuilder` 未进入有效链路；切换策略也不是双活热备。
2. 视觉实验室的 `VLM_*` 配置为空，运行进程仍可能读取旧的 `LLM_MODEL_NAME`，导致视觉任务退化为启发式分析或出现配置漂移。
3. 22 个活跃 Skill 中仅 2 个声明了输出 Schema，六个 Skill 没有任何执行记录，最终报告难以稳定消费和追溯分析结论。
4. 主编排目前按步骤串行执行；最终汇总虽有 `actor_id=synthesis`，但不是具备受限输入和证据检查职责的独立汇总边界。

本方案将模型选择、输出契约、证据审阅和报告汇总收敛成可验证的流程，同时保持一个全局任务编排器，避免引入多套平级调度逻辑。

## 产品分析

### 目标用户与场景

- **用户群体:** 发起用户研究任务、消费研究报告的业务和研究人员，以及维护系统的工程人员。
- **使用场景:** 用户提交多源材料，系统调用检索/分析工具与 Skill，生成带来源、限制条件和建议的研究报告。
- **核心痛点:** 模型降级、工具失败、无证据推断和自由文本输出会被混入结论，用户无法判断报告哪些是事实、哪些是模型推断。

### 价值主张与成功指标

- **价值主张:** 每项任务可解释模型选择、工具状态、证据来源和报告结论的形成过程。
- **成功指标:**
  - 文本和视觉请求在受控故障下按 `GPT-5.4 -> GPT-5.2 -> GPT-5` 顺序切换。
  - 每个活跃 Skill 都有可解析的输出契约和至少一个受控执行用例。
  - 报告的关键结论 100% 标注证据引用或 `llm_inference`，无来源内容不得伪装为事实。
  - 独立工具步骤并发执行时，结果顺序、失败语义和报告内容保持确定性。

### 人文关怀

研究报告必须将“工具计算结果”“外部事实”“用户材料”和“模型推断”区分展示；不得因为模型或视觉能力不可用而给出具有误导性的确定性结论。

## 变更内容

1. 建立三模型、有能力声明和健康状态的请求级自动故障切换策略。
2. 使文本和视觉模型共享统一配置契约，但仅将已探测确认支持图片输入的模型用于视觉任务。
3. 为 Tool 和 Skill 建立统一结构化输出信封、Schema 校验、执行契约和降级状态。
4. 将报告汇总收敛为受限上下文的 `ReportSynthesisAgent`，并在生成前执行确定性的证据完整性检查。
5. 在显式依赖图基础上仅并行执行互不依赖的步骤，补充可观测性和上线验收。

## 影响范围

- **模块:** `agent-api`、`orchestrator-runtime`、三个 LLM/VLM 实验室、Skill 注册与 Schema、报告渲染、测试与启动脚本。
- **文件:** `.env.example`、`apps/orchestrator-runtime/src/runtime/gateway-llm-client.ts`、`apps/orchestrator-runtime/src/orchestrator.ts`、`apps/orchestrator-runtime/src/runtime/context-builder.ts`、`external-tools/*-lab/apps/server/src/{config,services}/`、`schemas/`、`tests/`。
- **API:** 任务执行进度、工具/Skill 执行结果及最终报告 JSON 均可能增加明确的状态、证据与追踪字段；需保持版本兼容或提供迁移。
- **数据:** 运行记录增加模型尝试、降级原因、证据引用和调度批次；不得记录 API Key、原始敏感提示词或未脱敏材料。

## 核心场景

### 需求: 三模型故障切换
**模块:** 模型路由

默认以 `GPT-5.4-joybuilder` 执行；发生可恢复错误时依次尝试 `GPT-5.2-joybuilder`、`GPT-5-joybuilder`。恢复的主模型通过半开探测重新承接后续请求。

#### 场景: 主模型暂时不可用
- 给主模型注入超时、配额 429 或 5xx。
- 当前请求转向下一个健康候选模型，并记录每次尝试与原因。
- 鉴权、请求格式和内容安全类 4xx 不切换模型，直接返回可行动错误。

### 需求: 真实视觉能力与降级透明
**模块:** 视觉实验室

视觉任务只能选择声明且经探测证实支持图像输入的候选模型。

#### 场景: 没有可用视觉模型
- `VLM_*` 未配置或所有视觉候选失败。
- 输出 `status=degraded` 与机器可读的 `reasonCode`，报告明确展示局限，不能把启发式结果标为视觉模型结论。

### 需求: 可消费且可追溯的分析结果
**模块:** Tool、Skill、报告汇总

每一个步骤输出统一信封与领域 `payload`，使下游不需要解析自由文本来判断证据和失败状态。

#### 场景: Skill 基于工具结果形成建议
- Skill 使用上游 `evidence_refs`，在 `findings` 中引用这些证据。
- 无法验证的结论标为 `llm_inference`，并列出假设与限制。
- 汇总前的校验拒绝没有来源、无状态或不符合 Schema 的关键结论。

### 需求: 隔离的最终报告汇总
**模块:** `ReportSynthesisAgent`

最终汇总使用独立角色和最小必要上下文，而非复用整个任务对话和原始日志。

#### 场景: 部分工具失败但仍可生成报告
- 汇总 Agent 接收通过校验的证据包、失败摘要、审阅意见和报告 Schema。
- 报告保留已验证的结论，同时在限制章节说明缺失来源和降级影响。

## 风险评估

- **风险:** 三个候选模型不一定都支持视觉输入，真实网关权限、配额和响应格式也可能不同。
  **缓解:** 配置提交前通过独立文本/图片探针验证每个候选；探针失败时禁用其相应能力，不做名称推断。
- **风险:** 统一 Schema 可能破坏现有自由文本消费端。
  **缓解:** 先引入版本化信封和兼容读取层，随后逐步收紧为强校验。
- **风险:** 并行化导致速率限制、结果顺序不稳定或上下文竞态。
  **缓解:** 仅调度无依赖批次，设置并发上限、稳定排序、幂等键和失败隔离测试。
- **风险:** 工作区已有未提交改动与本方案改动重叠。
  **缓解:** 开发前先逐文件审阅和合并当前改动，禁止覆盖、重置或清理现有工作区内容。

# Conversion Funnel Analysis — 分步数据 + 转化路径 → 漏斗诊断（编排 wiki 正典）

吃一个转化/流失诊断诉求与可用的分步数据，**默认直接当分析师做一次漏斗诊断**：把「完成一件事」看成一条有序事件序列，逐级衰减、定位最大流失环节，再往下追「为什么流失、怎么优化」。本 skill 不发明漏斗方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——转化漏斗分析方法本体（定口径 / 拆步骤 / 校验埋点 / 算分步转化率流失率 / 看相对流失 / 宏微转化分层 / 分群下钻 / Fogg 五类根因 / 定性求证 / 优化回测）、功能采纳分析、VOC、问题优先级、A/B 测试、体验度量——把它们用到 *这一条* 路径、*这一批* 数据、*这一个* 诊断目标上。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 分步转化率/流失率的算法、总转化率连乘、宏转化 vs 微转化的分层、Fogg 行为模型 B=MAP 的五类根因、Baymard 放弃原因清单、漏斗口径定义清单、诊断假设清单的口径，唯一出处都在 wiki 正典（`conversion-funnel.md`）里。你的工作是**读对的正典、抽对的算法与框架、用到本次数据上**，而不是凭记忆默写「Fogg 五类是哪五类」。产出的价值在于贴合 *这一条* 路径、*这一批* 真实数字、*这一个* 场景的诊断判断，不是又一份通用方法讲解。
- **定位 → 归因 → 优化，是漏斗的命根。** 正典点破：漏斗**只回答「在哪流失、流失多少」，回答不了「为什么」**；它是诊断的**入口，不是终点**。停在「第 3 步流失最大」价值有限；要害是**接 Fogg 五类根因形成归因假设、用定性方法求证、再落成可执行的优化动作并回测**。诊断里凡是只报了流失率却不给归因方向与下一步的，都要补齐或删掉。
- **看相对流失，别被绝对人数和平均值骗。** 这是漏斗最大的两个误用。① 基数大的步骤流失人数自然多，未必是病灶；病灶是**相对流失**明显高于前后步 / 历史基线 / 同类流程经验值的那一环。② 整体转化率会掩盖结构差异（NN/g 的「平均的谎言」）——整体平稳可能盖住某一渠道 / 设备 / 版本的严重退化，**必须分群下钻**，把问题收敛到「异常人群 × 异常环节」的交叉点。
- **口径先行，否则结论可能相反。** 开放漏斗 vs 封闭漏斗、转化时间窗、每用户/每会话计一次、直接跟随 vs 间接跟随——这些口径不先约定就开算，算出的人数与转化率会不同，跨期对比更会失真。缺口径要么先问，要么按流程常识合理假设并**在运行说明显式声明**，绝不闷头就算。
- **归因是「假设」，不是已证实的结论。** 按 Fogg 五类落到某一步的根因（认知不足 / 动机不够 / 操作能力门槛 / 信任顾虑 / 技术故障）是**待验证的假设**，不是原因定论。所有根因判断都以「疑似 / 假设：…」的口吻落笔，**绝不写成「就是因为字段太多才流失」这种断言**，必须接定性求证（录屏 / 可用性测试 / 深访 / VOC）才能定论。
- **分层交付：结论写干净，依据集中到后面。** 使用者要的是能直接开会的漏斗诊断，不是分析过程的实况转播。正文的「发现」（结论摘要、分步转化率表、最大流失环节、Fogg 归因假设）**先说清楚、能扫读**；而口径存疑、数据待校验、draft 背书、"归因是假设需求证"这些「依据与边界」**集中放到口径小结/数据可信度校验 +《本次运行说明》**，不逐句夹进每条发现里。判据：读者**只读发现段就应拿到一份干净、可汇报的诊断**。做法——① 结论先行；② 一句话能说清就不写一段；③ 发现层只在真正影响该结论时留一句轻量 ⚠️。
- **结论直给：写「结论 + 证据」，别写「我是怎么想出来的」。** 这是可读性的命门。使用者要的是分析结论，不是你抵达结论的思维轨迹——发现层每一条都落成「结论（括号里附关键数字/证据）」，而不是「先铺方法 → 逐一排除 → 最后得出」的推导叙事。三条戒律：① **不复述方法**——「为什么看相对流失而非绝对」「转化率 = 下一步人数÷上一步人数」这类是方法常识，直接用、别在产出里再讲一遍；连公式定义都不必印，给算好的数字即可。② **不留推导草稿**——像「0.39÷(0.68×0.93×0.94)≈65.6%」这种算式收进一个括号或脚注，正文只给结果「改版前基线 ≈ 65.6%（反算）」。③ **不写元评论**——「这是典型病灶信号」「三个信号一致指向」「看相对流失、不被绝对人数误导」这类对自己分析的旁白，一律删。判据：把某句删掉后，结论和它的证据是否还在？在，则那句是过程赘述，删。
- **两种形态，按「有没有可算的分步数据」自动切换：**
  - **有数据就直接诊断（默认、主线）。** 用户给了分步人数 / 分步转化率 / 分群明细 / 改版前后对比 / 足够具体的流程与数字，就**直接算、直接定位、直接给归因假设与优化建议**，而不是只递一套「框架」让他自己填。
  - **没数据才产设计/诊断框架。** 若用户只问「怎么做漏斗分析」或只描述了流程却没有任何数字，就产出**针对本次这条路径的漏斗设计/诊断框架**（口径清单 + 步骤拆解 + 埋点校验清单 + 分群维度 + Fogg 五类诊断假设清单 + 出图建议），并**明确索要分步数据**——拿到后即可切到「直接诊断」。
- **边界：拆客观转化/流失 → 本 skill；主观满意度 / 搭长期指标体系 / 界面可用性走查 → 隔壁。** 要诊断满意度/NSS/NPS 分数为什么掉、找关键驱动、做 IPA，用 `analyze-satisfaction`（漏斗不给「体验好不好」的主观结论）；要给产品搭长期体验指标体系/看板/北极星（先建体系那一步），用 `build-experience-metrics`（本 skill 反过来把分步转化率作为其中一个可追踪指标喂给它）；要拿尼尔森十大原则逐条走查界面找可用性问题，用 `run-heuristic-evaluation`。新功能的「发现→首次使用→持续使用」本身就是一条漏斗，与**功能采纳分析**同源、常配合，本 skill 直接处理并在正典层 cross-walk 到 `feature-adoption.md`。**用户实际走法发散、路径不固定时**，先提示做路径分析、再决定是否建线性漏斗（强塞会制造假流失）。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把分步转化率/流失率算法、总转化率连乘、宏微转化定义、Fogg 五类根因表、Baymard 放弃原因、口径清单与诊断假设清单的口径写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——框架可能已从 draft 改为 reviewed、示例可能已补充本土案例。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份诊断建立在什么之上、哪里可能不牢。结构见 `references/conversion-funnel-analysis-skeleton.md`，五个部分一个都不能少。
3. **声明局限，但集中归口、不淹没结论（分层交付）**：基于 draft（未评审）正典的部分、靠 Fogg 五类推断而非定性证据确证的根因（标「假设 / 待验证」）、因数据不足或未分群而没能覆盖的环节/人群（标「未覆盖」）、埋点可信度存疑的地方（标「数据待校验」），都要交代清楚、别让使用者误以为处处已确证——但**集中写到口径小结/数据可信度校验 +《本次运行说明》**；发现层（结论摘要、分步转化率、最大流失环节）只在真正影响该条结论时留一句轻量 ⚠️。
4. **缺内容不编造；「未找到」必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空默写一套「Fogg 五类根因」或「口径清单」冒充正典。**反过来同样重要：写下任何「未找到 / 走空链接」之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。** 绝不靠目录列举、shell 报错或印象就断定「它不存在」。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。
   - **更要命的是对数据本身的编造。** 分步人数、分步转化率、分群差异、改版前后对比——**必须基于用户提供的真实分步数据或明确事实**。**绝不脑补漏斗数字、编造某步转化率、编造分群差异或「掉了几个点」**——数据不足就说不足、索要数据，不要靠想象拼一份「看起来很详实」的漏斗诊断。埋点/数据可信度存疑时先声明「数据待校验」，别在可疑数据上强行下确定性结论。

## 第 1 步：解析 brief

从用户输入里抽出要素，决定走「直接诊断」还是「产框架」、读什么、产出长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **转化路径 + 可用分步数据** | 决定能不能直接诊断、诊断哪条链路。数据形态：埋点分步人数 / 分步转化率 / 分群明细（渠道·设备·新老·版本）/ 改版前后对比 / 足够具体的流程与数字描述 | **有任一可算的分步数据 → 直接诊断模式**；**完全没有数字（只描述了流程）→ 框架模式 + 索要数据**；连「拆哪条路径」都没有 → 先问清 |
| **转化目标与口径** | 漏斗的前置命根：宏转化目标是什么、开放/封闭、转化时间窗、每用户/每会话计一次、直接/间接跟随 | **尽量先确认**；确实拿不到时按流程常识合理假设并**在运行说明显式声明**（口径不同结论可能相反） |
| **分群维度是否可得** | 决定能否下钻拆穿「平均的谎言」（渠道/设备/新老/版本） | 缺 → 先做整体漏斗，声明「未分群，可能被平均值掩盖结构差异」，并建议补分群数据再收敛人群 |
| **诊断深度 / 场景** | 决定落点：版本监控期预警 / 明确变差但病灶未知的快速收敛 / 改版评估回测闭环 / 新功能采纳漏斗 | 缺 → 默认「定位最大流失环节 + 给 Fogg 归因假设与求证方向」，并在运行说明声明目标假设 |

判断原则：**路径与分步数据缺了优先问 / 索要；口径、分群、深度等参数缺了就合理假设 + 声明**——不要因为缺一个「时间窗多长」就卡住，也不要在没有任何真实数字时硬编一份「看起来像漏斗诊断」的数据（那违反纪律 4）。

## 第 3 步 · A：直接诊断（默认模式 —— 有分步数据时）

你现在就是漏斗分析师。按 `conversion-funnel.md` 的 8 步骨架，对用户提供的分步数据做诊断（详见 `references/conversion-funnel-analysis-skeleton.md`，**起草时打开它**）：

1. **定口径 · 先约定再算**：先落一段口径小结——宏转化目标、步骤序列（3–7 步、互斥必经、中性名）、开放/封闭、时间窗、计数口径（每用户/每会话）、顺序要求、可用分群维度。口径里用户没给的，先问或按流程常识合理假设 + 标注（呼应纪律 3）。
2. **校验数据可信度**：确认每步都有对应事件、参数完整、无漏报/重复触发、步骤互斥必经。数据存疑就标「数据待校验」，别在可疑埋点上下确定性结论。
3. **算分步转化率/流失率 + 总转化率**：逐步算分步转化率、流失率，总转化率取各步连乘；产出里给**算好的数字表**（每步进入人数 / 绝对流失 / 转化率 / 流失率）即可，**不必再印公式或列竖式推导**。
4. **定位最大流失环节（看相对，不看绝对）**：认定**相对流失**明显高于前后步 / 历史基线 / 经验值的那一环。写法**结论直给**——「最大流失 = 第 X 步（相对流失 Y%）」+ 一行证据（高于前后步 a%/b%、高于基线 c%、命中改动/某分群）；若绝对流失最多的是另一步，**一句带过**（如「第 1 步绝对流失最多但属未改动的天然分流，非病灶」）即可，**别写整段「为什么不是第 1 步」的排除式推导**。宏/微转化分层一句话点到（中段是否健康），不展开方法讲解。
5. **分群下钻**：按渠道 / 设备 / 新老 / 版本拆分（分群对比矩阵：步骤×分群），定位「异常人群 × 异常环节」的交叉点，区分**普遍问题**与**特定人群问题**（如某版本、某渠道在某步集中流失）。改版评估场景则对比改动前后同口径分步转化率、量化收益或退化。
6. **归因假设（Fogg 五类）**：对最大流失环节，按 **认知不足 / 动机不够 / 操作能力门槛 / 信任顾虑 / 技术故障** 五类列疑似根因，各配观测信号。**每条根因以「假设 / 待验证」落笔**，不写成已证实的原因。
7. **定性求证方向**：对每类假设给对应求证方法（录屏 / 可用性测试 / 深访 / VOC / 表单字段分析 / 错误监控）——把「为什么」交给定性方法，漏斗只负责把靶子收敛到具体环节与人群。要深访接 `interviews.md`、要聚合抱怨接 `voc-analysis.md`。
8. **优化建议 + 回测方案**：给可落地的优化动作（对准假设的根因，如降门槛/补提示/减顾虑/修故障），并给**同口径漏斗或 A/B 回测**方案验证收益。多个待改环节要排序时接 `issue-prioritization.md`；严格验证因果接 `ab-testing.md`。
9. **边界归口（别逐句夹在发现里）**：基于 draft 正典、靠假设的根因（标「假设/待验证」）、数据没覆盖的环节/人群（标「未覆盖」）、埋点存疑处（标「数据待校验」）——集中写到口径小结/数据可信度校验 +《本次运行说明》（呼应纪律 3 的分层交付）；发现层只在真正影响该结论时留一句轻量 ⚠️。

## 第 3 步 · B：产漏斗设计/诊断框架（无数据时）

产出一份针对本次这条路径的漏斗设计/诊断框架（结构见 skeleton），并**末尾明确索要分步数据**（「把这条路径每一步的进入人数、可拆的分群（渠道/设备/新老/版本）、以及改版前后的数据发我，我就能直接算出在哪流失最多、给你归因假设和优化建议」）：

- **口径清单**：宏转化目标、步骤序列建议（3–7 步、互斥必经、中性名）、开放/封闭、时间窗、计数口径、顺序要求——按正典「漏斗定义清单」逐项列；
- **步骤拆解建议**：针对用户这条具体路径，给一版拆步骤草案（含建议加入的微转化中间步）供确认；
- **埋点校验清单**：开工前要逐项确认的数据质量项（事件齐全/参数完整/无重复触发/无漏报）；
- **分群维度建议**：建议拆哪些维度（渠道/设备/新老/版本）以便下钻，别被平均值骗；
- **Fogg 五类诊断假设清单**：五类根因 × 观测信号 × 求证方法（口径取自正典的诊断假设清单），供拿到数据后逐环节套用；
- **出图建议**：漏斗图 / 分群对比矩阵 / 到达耗时拆解。

## 第 4 步：分析自检（拿「局限与误用」回头扫一遍）

**实时读 `conversion-funnel.md` 的「局限与误用」**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **只看绝对人数？** 最大流失环节的定位是否基于**相对流失**（对比前后步/基线/经验值），而非「哪步流失人数最多」？
- **口径没约定就开算？** 开放/封闭、时间窗、计数口径、顺序要求是否都交代清楚，跨期对比是否同口径？
- **发散路径硬塞线性漏斗？** 这条路径是否相对固定、可线性化；若用户实际走法发散，是否提示先做路径分析而非强塞？
- **时间窗设置不当？** 窗是否可能太短（把慢决策用户误判为流失）或太长（混入无关行为）？
- **忽视分群、被平均值误导？** 是否做了分群下钻（或在无分群数据时明确声明这一风险并索要）？
- **只做漏斗不做归因？** 是否接了 Fogg 五类根因假设 + 定性求证方向，而非只报了一串流失率？
- **埋点不可信仍强行解读？** 数据可信度是否校验/声明，可疑处是否标「数据待校验」？
- **把假设当结论？** Fogg 根因是否都以「假设/待验证」落笔，没写成已证实的原因？
- **脑补数据？** 分步人数/转化率/分群差异是否都有用户素材出处，没凭空编造；未覆盖处是否标「未覆盖」？

把改动落到诊断上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/conversion-funnel-analysis-skeleton.md` 的顺序输出。**贯穿始终的写法：发现层结论直给（结论 + 证据），不复述方法、不留推导草稿、不写元评论；依据与边界归口到运行说明（见北极星「结论直给」「分层交付」两条）。**

- **直接诊断模式**：① **诊断结论摘要（结论先行，放最前）**——一句话核心判断（这条路径卡在哪一步、相对流失多大、最可能是哪类根因）+ **Top 优化建议速览**（每条：环节 + 疑似根因 + 一句怎么优化 + 怎么求证/回测），让人第一眼看到「最该先动哪一步」，细节在下方展开 ② 漏斗诊断报告（口径小结 · 数据可信度 · 分步转化率/流失率表 + 总转化率 · 最大流失环节定位[相对流失] · 分群下钻矩阵 · Fogg 五类根因假设 · 定性求证方向 · 优化建议 + 回测方案）③ 分析自检结果 ④《本次运行说明》（五部分）
- **框架模式**：① 漏斗设计/诊断框架（口径清单 · 步骤拆解建议 · 埋点校验清单 · 分群维度 · Fogg 五类诊断假设清单 · 出图建议）② 对分步数据的索要 ③《本次运行说明》（五部分）

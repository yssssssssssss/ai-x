# Generate Survey — 研究目标 → 合规问卷草稿（编排 wiki 正典）

吃一个研究 brief，产出一份**这次专用**的问卷。本 skill 不发明问卷方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——问卷设计规范、标准化体验量表、满意度/定性/量表编制方法——把它们组装成贴合本次目标与人群的题目清单，并拿规范的检查清单回头自检。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 问卷规范、量表、计分口径的唯一出处都在 wiki 正典里。你的工作是按本次 brief **读对的正典、抽对的要点、填进问卷骨架**，而不是凭记忆复述「问卷应该怎么写」。产出的价值在于贴合 *这一个* 目标和 *这一类* 人群，不是又一份通用模板。
- **问卷是定制品。** 同样测「满意度」，给新用户和资深用户、放在 App 弹窗和短信链接、要 3 题还是 20 题，题目、量表点数、跳转逻辑都不同。务必让研究目标、人群、维度、约束真正改变问卷内容。
- **一道坏题污染整份数据。** 双重含义、诱导题干、答非所问的题型、模糊的程度词，会让回收的数据不可信。落笔后必须拿规范的**检查清单**逐条自检——这是本 skill 区别于「随手列几道题」的关键一环。
- **边界：结构化定量 → 本 skill；定性深挖 → 隔壁。** 本 skill 面向**可投放、可统计**的结构化问卷（封闭题、量表、跳转、计分）。若用户真正要的是 1v1 深访 / 焦点小组那种靠开放追问挖 why 的提纲，用 `generate-interview-guide`。开放题为主、用问卷形态做前瞻探索（低成本替代访谈）仍属本 skill，走 `qualitative-questionnaire` 那条。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把规范条文、量表题目、计分公式、检查清单写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——规范可能已从 draft 改为 reviewed、量表可能已扩充。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份问卷建立在什么之上、哪里可能不牢。结构见 `references/survey-skeleton.md`，五个部分一个都不能少。
3. **在问卷正文里就地声明局限**：基于 draft（未评审）正典的部分、因量表库未覆盖该维度而现写题的部分、因未找到贴合场景而用通用方法兜底的部分，都在该处直接标注，别让使用者误以为处处有正典或标准量表背书。
4. **缺内容不编造。** 读不到的正典、空的目录（如题库）、走空的链接——标「未找到 / 待补充」，绝不凭空生成看似正典的规则、看似标准量表的题目或计分公式。问卷会真的投给用户、数据会真的进分析，宁可少给、也不可误导。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/standards/questionnaire-design.md`，其所在的 `.../research-wiki/` 即根。
- **找不到 wiki**：不要硬编一份问卷糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用经验」，且**不得**伪造任何「依据规范第 X 条」的说法。

## 第 1 步：解析 brief

从用户输入里抽出四个要素，决定后面读什么、问卷长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **研究目标** | 问卷的主轴，决定要测哪些维度、问卷类型（满意度/体验评测/探索/量表自研…） | 完全缺失 → 先问清，这是不能假设的核心 |
| **目标人群** | 决定措辞、抽样/甄别题、投放假设（C 端/B 端、新/老、已购/流失…） | 缺 → 合理假设并在运行说明声明 |
| **要测维度** | 每个维度 → 一节题目；决定选哪个量表或现写题 | 缺 → 按研究目标推断维度并声明 |
| **约束（题量 / 渠道 / 时长）** | 决定量表取舍（短问卷用 UMUX-Lite）、是否分页、题型繁简 | 缺 → 默认「短问卷、移动端弹窗、3–5 分钟」，并声明 |

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**。不要因为缺一个题量上限就卡住，也不要在目标都没有时硬编一份问卷。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着「我要从这抽什么填进问卷」的目的，而不是通读。读完记下每篇的相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于问卷 |
|---|---|---|
| `methods/standards/questionnaire-design.md` | **必读（全程硬约束）** | MUST / MUST NOT / SHOULD 的题目与选项编写规则；一份问卷的构成（封面信/指导语/题目/编码）；5 步设计流程；**上线前检查清单**（第 4 步逐条对照自检的依据） |
| `assets/scales/standardized-ux-scales.md` | 测**体验 / 可用性 / 满意 / 忠诚**维度时**必读** | SUS / UMUX-Lite / SEQ / NPS / CSAT / CES 的选型口诀、题数·点数·计分·基准；**直接取标准量表的题干与口径，别自造** |
| `methods/toolbox/collection/satisfaction-survey.md` | 满意度 / 体验评分类问卷 | 量表与计分口径（CSAT/NPS/NSS/CES）、经典三段式问卷结构（核心计分→开放归因→低分二级归因）、题目编写自检清单、数据清洗 |
| `methods/toolbox/collection/qualitative-questionnaire.md` | 开放题为主、用问卷做**前瞻探索**（低成本替代访谈） | 何时用 / 不适合、开放题问卷的设计取舍、产出形态 |
| `methods/toolbox/collection/scale-development.md` | 要测的维度**无标准量表、需自研量表** | 量表编制流程、信度/效度、维度划分与计分、题库构建与正式测试前的环节 |
| `assets/question-bank/index.md` → `survey/<主题>.md` | 先取现成题再现写 | 取贴题的现成问卷题。题库按 7 主题组织（screener/behavior/needs/mindset/decision/experience/concept），题目含 `[目标品类]/[目标人群]/[本平台]` 占位符，按 brief 替换。⚠️ **当前 `survey/` 多为空**（问卷提纲需从 sheet 手动导出桥接后才入库）——空属正常，此时按规范现写题并在运行说明注明「问卷题库暂缺，本次为现写」；`interview/<主题>.md` 有货，可借其题组作**选题思路**参考，但问法须按问卷规范改写（访谈题≠问卷题） |

读法补充：先快速看相关 `index.md`（`methods/standards/index.md`、`methods/toolbox/collection/index.md`、`assets/`下索引、`models/index.md`）确认有哪些条目、再决定打开哪几篇，避免空读。正典里的 `related:` 若指向某文件，按需顺藤再读；指向的文件不存在 → 记为「走空链接」写进运行说明。

## 第 3 步：选量表 / 选范式，并起草问卷

骨架与各段写法见 `references/survey-skeleton.md`，**起草时打开它**。先做两个选择，再落题：

**① 按维度选量表**（依据第 2 步读到的量表库口诀，库未覆盖则现写）：

- 要**全面**评估可用性 → SUS；问卷要**短**又想沿用 SUS 基准 → UMUX-Lite；只评**单个任务**难易 → SEQ；要**忠诚度/口碑**、对外沟通 → NPS；**单点满意** → CSAT；**费力度** → CES；**整体满意/体检** → NSS 或 CSAT。
- 量表库**未覆盖**的维度（态度、品牌认知、心理、需求强度、价格敏感等）→ 按规范现写题（量级口径要可量化、选项穷尽互斥同层次）；维度需要可信可比、要自研一套量表时，走 `scale-development` 的编制流程。**凡现写的量表/题，就地标注「量表库未覆盖，本次按规范现写」。**

**② 按问卷类型选范式**：满意度/体验评分 → `satisfaction-survey` 的三段式；前瞻探索/开放题为主 → `qualitative-questionnaire`；自研量表 → `scale-development`。

**③ 起草问卷**（骨架见 skeleton 文件）：

- **整卷结构**：封面信 → 指导语 → 分节题目（每个维度一节） → 结尾致谢。封面信/指导语按规范「问卷构成」一节的要求写全（身份、内容、目的、抽样理由、价值、不损害、奖励细则；多选题注明最多选几项等）。
- **每节 = 一个研究维度**，节内先客观/行为题、后态度/量表题；量表题直接用选定的标准量表题干与点数，或现写并标注。
- **题型与测量目标匹配**：要测「存在哪些问题」用多选；要测「各方面态度程度」用程度量表——别让题干与选项答非所问（规范 MUST 之一）。
- **逻辑跳转写明**：如满意度三段式里「仅 0–6 分低分用户」展示二级归因多选；甄别题不符则终止；按人群分支。跳转条件写在题目旁，执行者一看就懂。
- **就地标注**：兜底/基于 draft/现写量表的部分，在该节或该题旁标一句，呼应纪律 3。

## 第 4 步：合规自检（逐条过规范的检查清单）

**实时读 `methods/standards/questionnaire-design.md` 的「检查清单」一节**，对照本次问卷**逐项**给结论——这是硬约束，不是走过场：

- 每一项给 ✅ 通过 / ⚠️ 存疑 / ❌ 不通过，并附**证据或定位**（哪道题、怎么改）。清单条目以运行时读到的为准，不要凭记忆默写条数。
- 重点扫：一题一事（无双重含义）、题干中性无诱导、题型与选项协调（无答非所问）、程度/频率有可量化锚点、抽象概念已操作化、选项穷尽互斥同层次、封面信/指导语完整、问题少而精。
- 发现 ❌ → 回第 3 步改题，再自检，直到无 ❌（剩余 ⚠️ 要在自检结果里说明为何保留）。把自检结果作为交付的一部分输出（见 skeleton）。

> 若规范当前为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审规范，定稿后需复核」。

## 第 5 步：组装交付

按 `references/survey-skeleton.md` 的顺序输出三块：

1. **问卷正文**（封面信 · 指导语 · 分节题目[题目 + 题型 + 量表 + 跳转逻辑] · 结尾致谢）
2. **合规自检结果**（对照规范检查清单逐条 ✅/⚠️/❌ + 证据）
3. **《本次运行说明》**（五部分，见骨架文件，逐项填实）

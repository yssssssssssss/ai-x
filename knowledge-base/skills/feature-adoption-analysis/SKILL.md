# Feature Adoption Analysis — 采纳行为数据 → 采纳漏斗 + 五维指标 + 流失段归因（编排 wiki 正典）

吃一个功能采纳诉求与采纳行为数据，**默认直接当分析师做一次采纳诊断**：回答"新功能上线后，有多少人知道它、试用它、并持续用它"，并定位知晓→试用→持续用链路上的流失点。本 skill 不发明采纳分析方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——功能采纳分析（四段采纳漏斗 / 五维指标口径 / cohort 留存 / 分段病因）、转化漏斗、A/B 实验、VOC、体验度量 HEART、期望确认理论 ECT、用户深度访谈——把它们用到 *这一个* 功能、*这一批* 行为数据、*这一个* 目标上。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 采纳漏斗的四段划分、五维指标的计算口径、分段病因的判读、cohort 留存的做法，唯一出处都在 wiki 正典（`feature-adoption.md`）里。你的工作是按本批数据**读对正典、抽对方法、套进产出骨架**，而不是凭记忆复述"采纳率怎么算""漏斗有哪几段"。产出的价值在于贴合 *这一个* 功能和 *这一批* 数据的真实判断，不是又一份通用方法说明。
- **"采纳"不是"点了一次"，这是全套方法的命门。** 正典开篇就点破：功能采纳是"用户开始把这个功能稳定地用来解决自己的问题"，不是"知道有这个功能"、更不是"点了一次"。所以第一件事是**定义一个有意义的"激活"事件**（体现价值实现的关键动作，而非"打开页面"）。把"曝光/点击一次"当成采纳，会让采纳率虚高而留存惨淡——分析里凡是拿"点击/打开"当采纳的，都要纠正或标注。
- **先分清"手里是什么数据、要答什么问题"，再选形态。** 本 skill 两种形态：**有采纳行为数据（分段人数 / 功能埋点事件 / cohort 留存表，或至少各段计数）→ 直接做采纳诊断**；**还没有埋点/行为数据 → 产出采纳度量工具包 + 索要数据**。选错形态，后面算得再漂亮也答非所问。
- **算你能算的，不编造指标。** 这是本 skill 区别于"嘴上讲方法"的关键纪律，单列见下方纪律 4。一句话：**段转化率（下一段人数 / 上一段人数）、采纳率、采纳广度、深度/频率、绝对人数**——这些你**能直接算就真算**；而**功能留存曲线**（需逐 cohort 的时间序列）、**首次采纳时间**（需首次曝光与首次规律使用的时间戳）这类依赖原始明细的，**有明细数据就真算，否则只描述方法并诚实标注"需 cohort/时间戳明细"，绝不写一个看似精确却没真算的采纳率或留存数**。
- **数据不够算，就先要数据，别硬出结论。** 采纳诊断有"最低数据要求"。若缺的是**核心依据**（要读漏斗却没有分段人数、要判留存却没有 cohort、"激活"根本没定义）→ 视情况**先产出《数据补充请求》或采纳度量工具包**，不靠假设/占位把空缺糊成结论；若缺的只是**精度/粒度**（无分群维度、无时间戳）→ 降级 + 就地声明继续。判据见第 1 步「数据准入闸」。
- **定量定位"哪一段流失"，定性回答"为什么"。** 漏斗和五维指标告诉你**哪一段掉得最狠、掉了多少**——但**为什么**那一段流失（是没触达、还是上手太难、还是用一次觉得没价值），采纳分析本身回答不了，要靠对该段人群的分段问卷 / 访谈 / VOC 来补。只给一串漏斗数字不给病因与机制，等于把诊断停在半路。**相关≠因果**：漏斗定位到病灶段后，"改了它采纳率就会升"要靠 A/B 或下一周期回测验证（在正文点明）。
- **别只盯单一采纳率。** 正典反复强调：综合**采纳率、广度、深度/频率、首次采纳时间、功能留存**五个维度一起读——只看一个采纳率，分不清"是没人用"还是"少数人狂用"；深度高既可能是刚需、也可能是绕路反复试错，必须用定性区分。
- **分层交付：结论写干净，方法与不确定性集中到后面。** 使用者要的是一份能直接拿去开会的诊断，不是分析过程的实况转播。所以正文的「发现」（结论摘要、漏斗、指标、病因、建议）**先把结论说清楚、能扫读**；而**算力声明、口径存疑、draft 背书、相关≠因果、机器事件未剔**这些「依据与边界」**集中放到文末**（《数据与口径说明》+《本次运行说明》），不要逐句夹进每一条发现里。判据：读者**只读发现段就应拿到一份干净、可汇报的诊断**，想深究再往下看依据。做法上——① 结论先行（先给答案，再给数）；② 一句话能说清就不写一段；③ 每个发现只在**真正影响该结论成立**时才带一句轻量 ⚠️（如漏斗某段"该数含机器事件、偏高"），系统性的口径/算力交代一律下沉到文末那一节；④ 病因**先给最可能的一个**，其余列为"其他可能"一笔带过，别把三四个并列假设＋机理长篇都堆在结论区。把过程/自证与结论揉在一起，是这类诊断最伤可读性的通病。
- **边界：功能级采纳漏斗的定量诊断 → 本 skill；其它走隔壁。**
  - 看**单次任务/路径的完成转化**（下单流程哪步跌、注册转化）→ `conversion-funnel-analysis`。二者都用漏斗，但口径不同：转化漏斗看"单次任务的完成转化"，本 skill 看"功能级的知晓→持续使用"，跨越多次会话与较长时间窗。（两者读的是同源的 `conversion-funnel.md` / `feature-adoption.md` 正典。）
  - 手里是**满意度 / NSS / NPS 分数**（不是行为采纳数据）要做驱动或掉分归因 → `analyze-satisfaction`。
  - 要**从零给产品搭一套体验指标体系 / 北极星 / 看板**（还没指标那一步）→ `build-experience-metrics`。本 skill 是针对某个功能、读它的采纳数据做诊断，不是搭体系（体系搭好后，采纳分析是它下游的一个归因入口）。
  - 要**设计一个对照实验判断某改动是否带来增量** → `ab-testing` 对应流程。采纳分析拆解增量来自漏斗哪一段，二者互为因果验证与机理解释。
  - 把海量**开放题 / 评价 / 工单文字**编码成痛点清单 → `code-open-feedback`。本 skill 只在归因阶段借 VOC 找"为什么"的线索，深做文本编码转那边。

## 敏感素材处理（先读再动手）

采纳数据多为聚合计数（不敏感），但常夹带**用户 ID / 设备号、低采纳用户的回访记录、订单号、联系方式**——这些是敏感原始数据：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户级明细、回访记录写进 `research-wiki/` 或落盘到 skill 目录。
- **引用用户原话做匿名化，且贯穿每个角落**（正文、代表原话、《本次运行说明》都不得出现真实姓名 / 手机号 / 订单号 / 设备号 / 单位）。要说明做了匿名化，只陈述"已匿名"，**绝不回写"真名→代称"对照**。涉及联系方式等字段尤其当心，既是隐私要求，也避免触发内容网关拦截。
- **原始表只读不改。** 你的职责是分析，不是修订埋点数据。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把四段漏斗的划分、五维指标公式、分段病因、cohort 做法写死在这里或 references 里**。每次都从当前 wiki 现读，确保是最新版——正典可能已从 draft 改为 reviewed、口径或行业基准可能已修订、可能已补充本土（电商/京东）示例。
2. **产出末尾固定附《本次运行说明》**，五部分齐全（结构见 `references/feature-adoption-skeleton.md`），让使用者知道这份诊断建立在什么之上、哪里可能不牢。
3. **在交付物里声明局限——集中归口，不淹没结论**：基于 draft（未评审）正典的部分、样本/分母口径存疑的部分、因缺 cohort/时间戳而只描述未实算的指标、机器/自动事件未剔除可能虚高的部分、相关非因果的部分，都要交代清楚，别让使用者误以为处处有正典背书、或误以为是经检验的结论——但**集中写到文末「数据与口径说明」+《本次运行说明》**，发现层只在真正影响该条结论时留一句轻量 ⚠️（呼应北极星「分层交付」）。
4. **缺内容不编造；这里的"不编造"尤其指不编造采纳指标与人数。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，且**写下"走空"之前先用 Glob/Read 核验该路径确不存在**（本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read；把真实存在的正典误报成缺失，和编造一样有害）。**更要命的是指标造假**：采纳率、各段人数、段转化率、采纳广度、深度/频率、首次采纳时间、功能留存%——凡不是你**真实从给定数据算出来**的，绝不写一个看似精确的数。给了各段计数就真算段转化率与采纳率；没给 cohort/时间戳就**只描述该指标怎么算、标注"需明细数据/未实算"**，不要凭空写一条留存曲线。采纳诊断会真的拿去决定功能"继续投入还是下线"，宁可少给、也不可拿假数误导。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若已在该树内，向上找到含 `methods/ models/ assets/` 的那层；③ 不确定时用 glob 搜锚文件 `**/methods/toolbox/analysis/feature-adoption.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与"文件是否存在"的核验**一律用 Glob/Read**，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一套分析糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用采纳分析经验」，且**不得**伪造「依据本库功能采纳正典」的说法。

## 第 1 步：接收并研判这批采纳数据

动手算之前先搞清四件事，它们决定走哪种形态、读哪些正典、能算到什么程度：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **数据形态** | 决定走「直接分析」还是「产工具包」：**分段人数漏斗**（曝光/激活/使用/复用各段人数，或够拆成这几段的功能埋点事件）→ 直接做漏斗+指标+归因；**cohort 留存表**（按批次×周/月的仍在用人数）→ 叠加功能留存曲线；**仅总量**（总用户+功能用户，无分段）→ 只能算采纳率/广度、无法读漏斗流失段（降级）；**完全没有行为数据** → 走工具包形态 | 不清楚 → 看用户给的表结构推断（有"分段/各步人数"=漏斗；有"第N周留存"=cohort；只有两三个总数=仅总量），并在运行说明声明判断依据 |
| **分析对象 + "激活"口径** | 分析哪个功能？"激活"（啊哈动作）定义成什么？这是采纳分析的地基——把"点击/打开"当激活会让采纳率虚高。**若用户把"点一次"当采纳** → 先指出、建议改用体现价值实现的关键动作定义激活 | 激活未定义 → 先问或据功能性质**建议**一个价值型激活事件（并标注是建议、待确认）；连"分析哪个功能"都没有 → 先问清 |
| **分析目标** | 效果评估（这次迭代被多少人接住、价值兑现没）？还是诊断归因（流失在哪一段、为什么）？还是判断某能力去留？决定落点与深度 | 完全缺失 → 先问清或默认「效果评估+流失诊断」并声明 |
| **分析单位 / 分群 / 分母** | 用户级还是账号级（正典建议两级都看，以过滤掉角色上本就用不到该功能的人）？分母是什么（同期登录数 / 目标分群总数）？有没有新老/版本/渠道/价值分层可建 cohort？**机器/自动事件是否已剔除**？ | 缺分群 → 只出整体结论，标"无分群维度，无法分 cohort/定位人群"；分母不明 → 标"采纳率分母口径待确认"；未剔机器事件 → 提示指标可能虚高 |

落表之前先做两个判别：

- **这批数据是不是本 skill 该接的？** 若其实是单次任务转化（下单流程哪步跌）→ 转 `conversion-funnel-analysis`；若是满意度分数 → 转 `analyze-satisfaction`；若是要从零搭指标体系 → 转 `build-experience-metrics`。
- **算力/数据够不够支撑用户想要的诊断？** 用户要"看功能留存曲线"但只给了一期总量（没有 cohort 明细）→ 明说"留存曲线需按批次的时间序列明细，当前数据只能算截面采纳率/广度"，不要假装画了条留存曲线（呼应纪律 4）。

### 数据准入闸（硬缺口 → 先补数据或先出工具包，不硬出结论）

研判完、动手算之前先过这道闸：**对照"最低数据要求"，判断现有数据能不能产出一个有效诊断。**

- **完全没有采纳行为数据**（只有"感觉没人用""想做采纳分析"）→ 走**工具包形态**（第 3 步·B）：给出激活定义 + 四段埋点方案 + 五维指标口径 + cohort 设计，并索要数据。这不是失败，而是把地基先打好。
- **有部分数据但缺核心依据**（缺了它，所选诊断的核心产出——漏斗流失段定位 / 关键指标——就没有有效依据）→ **产出《数据补充请求》**（结构见 skeleton）：逐条列「缺什么 / 为什么这条诊断离不开它 / 最低可接受形态 / 补齐后能产出什么」，并给「若暂时只能提供 X，我可降级产出 Y（但给不了 Z）」的退路。**绝不用假设、占位把硬缺口糊成一个看似成立的采纳率或流失结论。**
- **仅影响深度 / 稳健性**（核心产出仍有效，缺的只是精度或下钻粒度）→ 不打断用户，按"合理假设 + 就地声明"继续（如无分群、无时间戳、样本偏小）。
- 判据一句话：**能不能给出一个"有效结论"——哪怕结论是"曝光段就塌了，先解决告知，其余段样本太小暂不判"（这本身有效）。给得出 → 继续；给不出、只能靠编 → 先要数据或先出工具包。**

| 最低数据要求（做完整漏斗诊断） | 硬缺口 → 先补数据 / 先出工具包 | 仅影响深度 → 降级继续 |
|---|---|---|
| 采纳漏斗**各段人数**（能拆成曝光/激活/使用/复用，或够还原这几段的事件明细）+ 一个**有意义的激活定义** | 完全无行为数据 → 出**工具包**；有总量但无任何分段 → 无法定位流失段，请补分段人数（或事件明细）；"激活"未定义或定义为"点一次" → 先定义价值型激活再算，否则采纳率失真 | 无分群维度（只出整体漏斗）、机器事件未剔（标可能虚高）、样本偏小（标不稳） |
| 判**功能留存**：按批次的 **cohort 留存明细**（第 N 周/月仍在用人数） | 无 cohort、只有一期截面 → 留存曲线无从画，请补时间序列明细，或说明先按周对齐版本采集 | 有 cohort 但周期短（只够看前 1–2 期）→ 可画但标"观察期短、趋势未定" |
| 算**采纳率**：功能活跃用户 + **明确的分母**（同期登录/活跃数或目标分群总数） | 分母完全不明、且给不出目标分群总数 → 采纳率算不了，请明确分母口径 | 分母口径可选（登录数 vs 目标分群）→ 两种都算并标口径 |

> 这一闸的意义：采纳诊断会真的拿去决定一个功能**继续投入还是下线**。**一个建立在缺失数据上的"采纳率"，若被当成结论用，比不给更危险。** 注意分寸：**别因小缺口频繁打断**（精度类降级即可），也**别对硬缺口硬算**——界线就是"能否给出有效结论"。

## 第 3 步 · A：直接分析（默认模式 —— 有采纳行为数据时）

你现在就是采纳分析师。按 `feature-adoption.md` 的九步骨架，对用户提供的数据做诊断（详见 `references/feature-adoption-skeleton.md`，**起草时打开它**）：

1. **核对/定义"激活"事件**：确认分析对象是哪个功能、"激活"定义是否为价值型动作。若用户把"点击/打开"当采纳，先纠正、建议改用体现价值实现的关键动作（并标注是建议、待确认）。
2. **搭采纳漏斗**：把数据串成**曝光→激活→使用→复用**四段（用户给的分段可能粗细不一，就近映射并说明），逐段算**绝对人数 + 段转化率（下一段/上一段，简单除法务必真算）**。缺某一段就留空并标"该段无数据"，不要脑补。
3. **读五维指标**：按正典口径算**能算的**——采纳率（标清分母口径）、采纳广度、深度/频率；**首次采纳时间、功能留存**若有时间戳/cohort 明细就真算，否则只描述口径并标"需明细、未实算"。综合五维一起读，别只报一个采纳率。
4. **（有 cohort 则）读功能留存曲线**：用同期群看不同批次用户的功能留存随时间衰减，判断"真价值 vs 一时新鲜"。无 cohort 就明说做不了、给采集建议。
5. **定位最大流失段并按病因归因**：找出段转化率最低的一段，套正典分段病因给**假设**——曝光段塌=告知/通知问题；曝光→激活塌=上手摩擦大、啊哈时刻太晚；**使用→复用低=最值得警惕**（价值或习惯没形成，可用 ECT 解释）。病因是**假设**，措辞用"疑似/推断"，不写成定论。
6. **定性补"为什么"**：对流失段人群（"试过一次不再用""从未触达"）**建议**触发分段问卷/访谈/VOC 回答动因；用户给了定性线索就用上，没有则标"建议补定性验证病因"。
7. **迭代建议 + 回测方案**：针对病因给可落地改进（告知 / 引导降摩擦 / 价值传达 / 习惯钩子），并**明确用 A/B 或下一周期采纳数据回测**（相关≠因果，写进正文）。
8. **可选·判功能定位**：用 Kano 判断该功能是基本/期望/魅力型——边缘增值功能天然低采纳不必恐慌，核心功能低采纳才是警报。
9. **边界归口（别逐句夹在发现里）**：基于 draft 正典、机器事件未剔可能虚高、无 cohort/无分群、病因属推断、相关非因果、行业基准仅供量级参考——这些**集中写到文末「数据与口径说明」+《本次运行说明》**（呼应纪律 3 与北极星「分层交付」）；发现层只在真正影响某条结论时带一句轻量 ⚠️。

> 这步的产出是**采纳漏斗（逐段人数+段转化率）+ 五维指标 +（可选）留存曲线 + 最大流失段病因 + 迭代建议与回测**，每条改进项点明"针对哪一段的什么病因"，并把"相关≠因果、需 A/B/下期回测"写在结论。

## 第 3 步 · B：产工具包（无采纳行为数据时）

产出一份针对本次功能与目标的**采纳度量工具包**（结构见 skeleton），并**末尾明确索要采纳行为数据**（「把这个功能的分段人数漏斗、或功能级埋点事件、或按批次的留存表发我，再确认下'激活'算哪个动作，我就能直接做一遍诊断」）：

- **"激活"事件定义建议**：针对该功能**建议**一个价值型激活事件（体现价值实现的关键动作，而非"打开页面"），并说明为何不能拿"点击"当激活；
- **四段漏斗埋点方案**：曝光/触达→激活→使用→复用各段该埋什么事件、记版本号与时间戳、口径一致；
- **五维指标口径清单**：采纳率（含分母口径）、广度、深度/频率、首次采纳时间、功能留存各自怎么算、主要回答什么；
- **cohort 分群设计**：按新老/版本/渠道/价值分层建 cohort，用户级 vs 账号级两级都看（过滤掉角色上用不到该功能的人）、剔除机器/自动事件的提醒；
- **数据索要清单**：拿到哪些数据后即可切到"直接分析"。

## 第 4 步：分析自检（拿"局限与误用"回头扫一遍）

**实时读 `feature-adoption.md` 的「局限与误用」**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **把"曝光/点击一次"当采纳？** "激活"是否定义为体现价值的关键动作，而非打开/点击？
- **只盯整体采纳率？** 是否综合了广度、深度、时间、留存，而不是单报一个采纳率？
- **直接套用行业基准？** 引用基准时是否声明"仅供量级参考、强依赖功能定位、跨产品横比意义有限"，没拿它当达标线？
- **深度高就当成好？** 高频是否区分了"刚需"与"绕路/反复试错"，用定性佐证？
- **机器/自动事件未剔除？** 是否提示先剔除自动化/机器事件与非目标角色再算分母，否则指标虚高？
- **只看数据不问原因？** 流失段是否都配了"为什么"的定性补充（或明确建议补）？
- **埋点缺失/口径漂移？** 跨版本"激活"定义是否一致、漏斗能否对齐？
- **编造了没算的数？** 每个采纳率/段转化率/留存%是否都从给定数据真算，没有凭空数字？

把改动落到分析上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

> **若第 1 步数据准入闸判定为"完全无数据"→ 交付工具包（3·B）**；**判定为"部分数据、硬缺口"→ 交付《数据补充请求》**（按骨架对应模板），不是诊断报告。下面的诊断产出待数据补齐后再做，不要先硬出一份。

按 `references/feature-adoption-skeleton.md` 的顺序输出**本次实际做到的**块（没做的不强行凑）。交付**分两层**——先「发现层」写干净，再「依据与边界层」集中收口（呼应北极星「分层交付」）：

- **直接分析模式**：
  - 顶部一行**caption**（功能 / 目标 / 数据形态 / 激活定义 / 有无 cohort），一句带过，不展开。
  - **【发现层，写干净、能扫读】** ① **采纳诊断结论摘要（结论先行，放最前）**——加粗一句话判断（这次迭代被多少人接住 / 最大流失段是哪段·最可能什么病因 / 去留建议）+ 3–4 条**关键发现要点** + Top 改进建议速览；**不要写成一大段**。② 采纳漏斗（逐段人数+段转化率，最大流失段标红）③ 五维采纳指标（表格；真算 vs 需明细未实算用一列区分，不在正文重复解释）④（可选）功能留存曲线/cohort ⑤ 最大流失段病因（**先给最可能的一个**，其余"其他可能"一笔带过；配一句怎么定性验证）⑥ 迭代建议 + 回测方案（表格）
  - **【依据与边界层，集中放后面】** ⑦ **数据与口径说明**（把原"数据研判"挪到这里：数据形态、激活口径是否价值型、采纳率分母口径、分群/cohort 有无、**算力声明**——真算了什么 vs 因缺明细未实算什么、机器事件是否剔除）⑧ 分析自检结果 ⑨《本次运行说明》（五部分）
- **工具包模式**：① 采纳度量工具包（激活定义建议 · 四段埋点方案 · 五维指标口径 · cohort 分群设计）② 数据索要清单 ③《本次运行说明》（五部分）
- **数据补充请求模式**：① 一句话说明为何先停 ② 待补数据清单（缺什么/为什么必需/最低形态）③ 降级退路 ④ 简版运行说明

**局限归口，不逐句夹进发现里**：基于 draft 正典、无 cohort/无分群、行业基准仅供量级参考、机器事件是否剔除、哪些指标未实算——这些系统性的口径与边界**集中写在⑦数据与口径说明 + ⑨运行说明**。发现层只在**真正影响某条结论成立**时带一句轻量 ⚠️（如漏斗某段人数含机器事件偏高、病因属推断需回测、相关≠因果需 A/B）——一处一句，点到为止，别把整套口径复述在每个发现下。

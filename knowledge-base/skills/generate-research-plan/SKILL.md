# Generate Research Plan — 研究需求 →（对齐）研究 brief →（展开）可执行研究方案

吃一个研究需求，交付一份**这次专用**的研究方案。真人研究员接到需求不会立刻写方案，而是**先把需求问清、对齐成 brief，再展开成方案**——本 skill 复刻这套「一份方案到底」的工作方式：一个 skill、两个内部阶段，中间留一道「需求不清就停下来」的闸门。本 skill 不发明研究方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——需求沟通规范、项目流程规范、采集/分析方法本体、场景打法、理论模型——把它们组装成贴合本次决策的执行方案。

## 北极星与边界（先理解，再动手）

- **先对齐，再展开。** 研究做废，多半不是方法选错，而是一开始就没问清「到底要回答什么、拿去做什么决策」。第一阶段的对齐不是走过场，是方案质量的地基。
- **模糊不是要绕过的障碍，是第一阶段的入口。** 用户给来一句话、很碎、说不清，恰恰是本 skill 该接住的常态。别因为需求不全就拒绝启动，也别脑补一个需求硬往下做——把模糊问清，本身就是交付物的一部分。
- **编排，不重造。** 方法与规范的唯一出处在 wiki 正典里。你的工作是按本次需求去**读对的正典、抽对的要点、填进方案骨架**，而不是凭记忆复述研究流程。方案的价值在于「贴合 *这一个* 决策」，不是又一份通用模板。
- **方案是决策的脚手架，不是方法清单。** 每个方法选择都要能回答「这个决策需要什么证据、这个方法为什么能给出」。选了什么要说理由，排除了什么也要说理由。
- **不确定就停下来问，胜过带着臆断硬出方案。** 这是真人研究员最核心的价值，也是本 skill 的关键设计——见阶段一的「条件停顿」。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份方案（或 brief）建立在什么之上、哪里可能不牢。模板见 `references/run-notes-template.md`，五个部分一个都不能少。
3. **在正文里就地声明局限**：基于 draft（未评审）正典的部分、抽样段在规范读不到时给的无背书通用建议、因未找到贴合场景而用通用方法兜底的部分，都在该处直接标注，别让使用者误以为处处有正典背书。
4. **缺内容不编造。** 读不到的正典、空的目录、走空的链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法、规则或结论。宁可少给，不可误导。高风险处（尤其抽样）尤其如此。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/standards/research-project-workflow.md`，其所在的 `.../research-wiki/` 即根。
- **找不到 wiki**：不要硬编一份方案糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用经验」。

# 阶段一 · 需求拆解与对齐（先问清，再动手）

目标：把原始需求（可能很碎）梳理成一份结构化 brief，并判断**是否清晰到可以动手做方案**。

## 1.1 读需求沟通规范，建立追问逻辑

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于 brief |
|---|---|---|
| `methods/standards/requirement-elicitation.md` | **必读** | 「五个坑」作为追问框架：① 真正的提出者/关注者是谁（别只信转述人）② 产品/业务现状与背景 ③ 需求的真实原因与目标（同一句需求，不同提出者/背景下目标常不同）④ 纵向（目标逐层分解）+ 横向（按角色/工作内容）拆解研究内容 ⑤ 方法/资源/时间规划与风险。把这五点变成你向需求方追问的提纲。 |

读法：带着「拿五个坑逐条审视这条需求，缺哪一坑就追问哪一坑」的目的读，而不是通读。记下该文件 frontmatter 的 `status`（当前为 `draft`），运行说明要列。

## 1.2 借场景「聊需求清单」反推该追问什么

场景打法里常带「典型研究问题 / 聊需求要问清的清单」，是反推追问点的好素材。先看 `methods/scenarios/index.md` 的打法清单，按本次需求主题挑**最贴的 1–2 篇**读，只抽其中的「该问清什么」：

| 何时读哪篇（相对 wiki 根） | 适用需求主题 |
|---|---|
| `methods/scenarios/category-consumption/cross-category/category-research.md` | 理解某品类、做品类研究（含一份立项前要问清的清单） |
| `methods/scenarios/category-consumption/cross-category/churn-user-research.md` | 用户流失、召回、判断真假流失 |
| `methods/scenarios/product-experience/iteration/iteration-evaluation-diagnosis.md` | 某功能/改版上线后的效果评估与问题诊断 |
| `methods/scenarios/product-experience/iteration/continuous-discovery.md` | 持续发现、把用户对话并入交付节奏 |
| `methods/scenarios/comprehensive-business/cross-line-topics/b-end-experience-measurement.md` | B 端系统体验度量、定北极星指标 |

**找不到贴合本次主题的场景** → 不要硬套不相关的清单，退回 1.1 的五个坑做通用追问，并在运行说明注明「无贴合场景的聊需求清单，按通用框架追问」。

## 1.3 结构化出 brief

把读到的框架套在本次需求上，产出结构化 brief（字段与写法见 `references/brief-skeleton.md`，**起草时打开它**）。五个核心字段：

- **核心研究问题**：这次到底要回答什么（一句话能说清，且彼此不矛盾）。
- **决策用途**：结论将用来支持什么决策 / 动作——"研究完拿来干嘛"。这是判断方法、深浅、样本的总开关。
- **目标人群**：要研究谁，界定到能去招募的程度（哪类用户、什么行为特征）。
- **约束**：时间、预算、可触达的用户、合规等现实边界。
- **成功标准**：这次研究怎样算"做成了"——能支撑哪个决策、达到什么确定性。

> **术语对齐（条件降级）**：本应顺带核对业务黑话/指标口径（业务术语库 / glossary）。**本库不含业务术语库** → 跳过这步，遇到拿不准的术语就在 brief 里列为待澄清项，并在运行说明注明「无业务术语库，术语对齐降级」。

## 1.4 ⛔ 条件停顿（关键闸门——别跳过）

结构化出 brief 后，**先判断需求是否清晰到可以做方案**，再决定往下还是停下。这道闸门是本 skill 区别于「拿到需求就堆方法」的核心。

**进入阶段二的最低门槛（三者缺一即停）：**

1. **核心研究问题**能一句话说清，且不自相矛盾；
2. **决策用途**明确——说得出结论拿去支持哪个决策/动作；
3. **目标人群**可界定到能去招募的程度。

判断原则与阶段一的取舍一致：**核心要素（上面三条）缺了就停下来问；次要参数（方法倾向、具体时间、预算额度、样本量）缺了不停**——合理假设并在 brief 与运行说明里声明即可，别因为缺一个时间就卡住。

- **信息充分（三条都过）** → 直接进入阶段二。可在回复里说明"需求已清晰，继续展开方案"。
- **信息不足（任一条不过）** → **只输出「结构化 brief + 待确认问题清单」，在此停下**，请需求方补充后再续阶段二。**绝不带着臆断把缺失的核心要素脑补出来硬做方案。** 待确认问题清单要具体、可回答（针对缺的那一坑发问，见 `references/brief-skeleton.md` 的清单模板），并明确告诉用户"补充这些后我再展开完整方案"。停下时同样在末尾附《本次运行说明》。

# 阶段二 · 方案展开（需求清晰后）

目标：把已对齐的 brief 展开成一份可执行、可被信任的研究方案。

## 2.1 取方案骨架与自检清单

| 读什么（相对 wiki 根） | 何时读 | 抽什么 |
|---|---|---|
| `methods/standards/research-project-workflow.md` | **必读** | 「方案撰写」一节列出的方案结构（研究目的[探索/描述/解释] → 研究内容 → 研究方法 → 抽样及配额 → 成本 → 时间规划）作为输出骨架；文末的检查清单作为方案自检的依据（聚焦其中方案阶段的条目：是否覆盖目的/内容/方法/抽样/成本/时间；其余偏执行/交付的条目，方案里以"后续安排"体现即可）。 |

骨架与各段写法另见 `references/plan-skeleton.md`，**起草时打开它**。

## 2.2 方法选型（采集 + 分析）

先读索引看清"货架上有什么"，再按 brief 的研究目的与内容挑方法：

- 读 `methods/toolbox/collection/index.md`（采集方法条目清单：文件 / 方法 / method_family / 何时用 / status）。
- 读 `methods/toolbox/analysis/index.md`（分析技术条目清单，同上字段）。
- 按"这个研究问题需要什么证据"挑选**采集 + 分析**的组合，再打开选中的那几篇方法本体抽要点（如怎么做、注意事项）。读到的每篇都记下相对路径与 `status`。
- **克制地只开你真要用的那几篇。** 先用索引的「何时用」列筛掉不相关的，再打开——通常一次方案用 2–3 篇采集 + 3–5 篇分析就够覆盖研究内容；不必把货架读遍。读了却不会写进方案的，就别打开，省时也省得方案堆砌。

**选型必须给理由**：每个选用的方法说清"为什么它能回答本次的研究问题"；同时点名 1–2 个**排除的方法**及排除理由（如成本、时间、不匹配研究阶段）——这能体现方案是权衡过的，不是随手堆的。

## 2.3 方法编排（场景打法做范例）

把选中的方法编排成"先做什么、再做什么、各步产出什么"。优先参照最贴合的场景打法做范例（同 1.2 的 5 篇清单，按 brief 主题选最贴的 1 篇打开，看它如何把方法串成一条线）。

- **有贴合场景** → 参照其编排逻辑（但方法本体仍以 toolbox 为准，场景只学"怎么串")，并记下该场景路径与 `status`。
- **无贴合场景**（如 planning/design 子线、多数品类子线当前为空） → 用通用流程兜底（探索→采集→分析→验证的常识顺序），并在该处就地标注「无贴合场景打法，通用编排兜底」。

## 2.4 （可选）选理论透镜

若本次研究有合适的概念框架可套（如期望确认、用户成长、认知偏误等），读 `models/index.md` 挑一个理论透镜，让方案的研究内容/分析更有结构。无合适的就不强加。

## 2.5 样本与配额 —— 读抽样规范，运行时判断有无

写「样本与配额」段前，**先去读** `methods/standards/sampling.md`（抽样与样本量规范）。**它在不在、是什么内容，都以你这次实际读到的为准——不要凭记忆或本文假设它的状态**（纪律 1）。按读到的情况分两种处理：

- **能读到（正常情况）** → 按它的规则写：定量部分讲研究总体 / 抽样方法 / 样本量的估算依据（如研究目的 + 置信水平 + 可接受误差 + 群体变异性，并要求留痕），定量/定性各自的样本量与配额按规范条款来；定性部分讲信息饱和的判据与关键分层覆盖。**记下该文件路径与其 `status`**（写进运行说明①）。规范里给的经验值（如某公式因子、经验法则）可作起步参考，但要标明"执行时按本次误差/饱和要求复核"。
- **确实读不到**（该文件不存在，早期版本曾如此） → **不要凭空编造抽样规则**（别硬给一个"样本量 ≥ 384"之类看似权威的数字冒充本库规范）。可给**合理的通用建议**（定性按信息饱和、定量按可接受误差与分层配额的常识），但**必须就地标注**一句：

  > ⚠️ 抽样规范（`methods/standards/sampling.md`）未找到，本段为通用建议、未经本库规范背书。

  并在《本次运行说明》②把"抽样规范未找到"作为缺口列出。

无论哪种情况：招募侧可读 `methods/toolbox/collection/participant-recruitment.md` 抽甄别/招募做法（它不替代抽样规范）；且**绝不编造看似权威的样本量冒充本库规范**（纪律 4）。读相关 toolbox 正典时若撞见指向 `sampling.md` 的悬挂 `⚠️待补充` 标记，而你实际能读到该文件，说明那是过时标记——按真实文件写，并在运行说明里提一句建议清理。

## 2.6 起草各段 + 对照清单自检

按 `references/plan-skeleton.md` 起草方案各段，然后**拿 2.1 读到的检查清单逐条回扫**：方案是否覆盖了研究目的、研究内容、研究方法、抽样及配额、成本、时间规划；方法是否服务于研究问题；排期是否留了余地与风险预案（呼应五个坑之坑五）。自检是实质修改，把发现的缺漏补回方案文本，并在交付里附一段简短的"方案自检结果"。

# 输出结构（固定）

- **第一部分 · 研究 brief**：核心研究问题 / 决策用途 / 目标人群 / 约束 / 成功标准（结构化）。
- **若需求不足（闸门未过）**：在 brief 之后给出**待确认问题清单**，提示"补充后再出方案"，**并在此停下**（仍附《本次运行说明》）。
- **第二部分 · 研究方案**（需求充分时才出）：研究目的 / 研究内容 / 方法选型+理由（含排除项） / 样本与配额（按读到的抽样规范撰写并记 status；规范确实读不到时才标注"未背书"） / 成本 / 时间规划 / 交付物；末尾附**方案自检结果**。
- **《本次运行说明》**：固定附在最末，五部分齐全（见 `references/run-notes-template.md`）。

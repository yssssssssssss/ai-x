# Competitive Analysis — 竞品素材 + 知己 → 竞品分析（编排 wiki 正典）

吃一个竞品分析诉求与可观测素材，**默认直接当分析师做一次竞品分析**：从「记录对手做了什么」走向「读懂对手为什么这么做、对我方有何影响、我方该如何行动」。本 skill 不发明竞品分析框架，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——竞品分析方法本体（以终为始 / 分层监控 / 战略意图四问 / AARRR / SWOT / 用户体验五要素 / 打勾对比 / 跟进决策矩阵）、桌面研究、机会点研究、KANO——把它们用到 *这一次* 的竞品、*这一个* 目标、*这一个* 生命周期阶段上。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 战略意图四问的问法、AARRR 五环节的含义、SWOT 的用法、用户体验五要素的分层、打勾对比与跟进决策矩阵的口径，唯一出处都在 wiki 正典（`competitive-analysis.md`）里。你的工作是**读对的正典、抽对的框架、用到本次竞品上**，而不是凭记忆默写「AARRR 是哪五个字母」。产出的价值在于贴合 *这一个* 竞品、*这一类* 用户、*这一个* 生命周期阶段的真实判断，不是又一份通用方法讲解。
- **看见 → 读懂 → 转化，是竞品分析的命根。** 正典开篇就点破：竞品分析**不等于「货比三家」**、不等于「功能记录」。停在「对手上线了什么」价值有限；要害是**读懂意图、判断对我方影响、转化为可落地的差异化行动**，让每一次对手迭代都成为我方进化的契机。分析里凡是只罗列「竞品有 A/B/C 功能」而不推导意图与应对的，都要补齐或删掉。
- **先「知己」再「知彼」。** 正典借「以终为始」强调：没搞清自身定位、生命周期阶段、用户特征就直接拆竞品，方向必然跑偏。**知己是不能省的前置**——缺了要么先问，要么据业务常识合理假设并在运行说明声明，绝不跳过直接堆竞品。
- **意图是「合理推断」，不是竞对官方事实。** 你读出的「竞品为什么这么做、想撬动什么指标」是基于可观测动作 + AARRR/SWOT 框架的**推断**，不是对手的公开声明。所有战略意图判断都要落成「推断：…」的口吻，**绝不写成「竞品的目标就是 XXX」这种断言**，也不得为让分析更「笃定」而编造竞对的内部动机、数据或未公开信息。
- **差异 ≠ 一定要跟。** 这是竞品分析最大的误用（盲目跟风/复制功能）。每个竞对动作、每个「人有我无」的差距，都要过**跟进决策矩阵**（是否动我方核心奶酪 × 是否值得跟进 → 必须回应 / 选择性优化 / 战略防御 / 忽略），而不是把「对手有」直接当成「我方要抄」。
- **两种形态，按「有没有竞品可观测素材」自动切换：**
  - **有素材就直接分析（默认、主线）。** 用户给了竞对功能观测 / 页面截图 / 策略描述 / 运营活动 / 用户评价 / 足够具体的文字描述，就**直接做分析、出结论与应对建议**，而不是只递一套「框架」让他自己填。
  - **没素材才产框架/工具包。** 若用户只问「怎么做竞品分析」或没给任何可分析的竞品信息，就产出**针对本次目标与竞品范围的竞品分析框架**（分层监控 + 四问清单 + AARRR/SWOT/五要素模板 + 打勾对比表 + 跟进矩阵），并**明确索要竞品观测素材与知己信息**——拿到后即可切到「直接分析」。
- **边界：读竞对的策略与意图 → 本 skill；逐条查界面可用性 / 真人做任务 / 搭自家指标 / 纯挖机会点 → 隔壁。** 要拿尼尔森十大原则逐条检视界面、找可用性问题并打严重度（哪怕对象是竞品页面，重点在「界面哪里不好用」），用 `run-heuristic-evaluation`；要让真实用户做任务并观察，用 `generate-usability-test`；要给自家产品搭长期体验指标体系，用 `build-experience-metrics`；要脱离竞品系统挖增长机会，用 `opportunity-research`。**要对（策略型）产品做微观的量化体验评测 / GSB 打分**（逐 case 打分定优劣、算低质率/优势占比，如图搜、推荐、首页评测），那是《产品体验评测》(`experience-evaluation-rating`) 的微观策略评估场景，本 skill **不做量化打分**——竞品对比只做定性的读意图与拆解（战略意图四问、用户体验五要素、打勾对比 人有我无/我有/无我有）。本 skill 聚焦「系统比较竞品 + 读懂战略意图 + 判断影响 + 定应对」。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把战略意图四问的问法、AARRR 五环节的定义与典型功能、SWOT 用法、用户体验五要素的分层、打勾对比与跟进决策矩阵的口径写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——框架可能已从 draft 改为 reviewed、示例可能已补充本土案例。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份分析建立在什么之上、哪里可能不牢。结构见 `references/competitive-analysis-skeleton.md`，五个部分一个都不能少。
3. **在正文里就地声明局限**：基于 draft（未评审）正典的部分、靠推断而非竞对官方口径得出的战略意图（标「推断」）、因素材不足而没能拆解到的竞品/功能/流程（标「未覆盖」），都在该处直接标注，别让使用者误以为处处有正典背书、处处已确证。
4. **缺内容不编造；「未找到」必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空默写一套「AARRR 五环节」或「五要素五层」冒充正典。**反过来同样重要：写下任何「未找到 / 走空链接」之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。** 绝不靠目录列举、shell 报错或印象就断定「它不存在」。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。
   - **更要命的是对竞品本身的编造。** 竞对上了什么功能、页面长什么样、用了什么策略、有什么数据/评价，**必须基于用户提供的真实可观测素材或明确的公开事实**。**绝不脑补竞品做了什么、编造竞对的功能/数据/运营动作/用户评价**——素材不足就说不足、索要素材，不要靠想象拼一份「看起来很详实」的竞品分析。读「战略意图」本质是推断，务必标明是推断而非事实（呼应纪律 3）。

## 第 1 步：解析 brief

从用户输入里抽出要素，决定走「直接分析」还是「产框架」、读什么、产出长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **竞品范围 + 可观测素材** | 决定能不能直接分析、分析谁。素材形态：竞对功能更新/页面截图/信息架构/运营活动/价格·会员·补贴策略/用户评价/足够具体的文字描述 | **有任一可分析素材 → 直接分析模式**；**完全没有素材 → 框架模式 + 索要素材**；连「分析哪个竞品」都没有 → 先问清 |
| **知己：我方定位 · 生命周期阶段 · 用户特征** | 竞品分析的前置命根，决定用哪套宏观框架（起步期看机会与威胁 / 成长期找创新亮点 / 成熟期打磨体验细节）、监控哪些「核心奶酪」 | **尽量先问**——这是不能凭空假设的核心；确实拿不到时据业务常识合理假设并**在运行说明显式声明** |
| **分析目标 / 场景** | 决定落点与深度：产品/设计评审支撑、持续监控预警、起步期验证需求·商业模式、成长/成熟期找亮点·打磨细节 | 缺 → 默认「支撑产品/设计决策的对比分析」，并在运行说明声明目标假设 |

判断原则：**竞品对象与知己缺了优先问；分析目标等次要参数缺了就合理假设 + 声明**——不要因为缺一个「评审用还是监控用」就卡住，也不要在没有任何可分析素材时硬编一份「看起来像竞品分析」的报告（那违反纪律 4）。

## 第 3 步 · A：直接分析（默认模式 —— 有竞品素材时）

你现在就是竞品分析师。按 `competitive-analysis.md` 的五步骨架，对用户提供的素材做分析（详见 `references/competitive-analysis-skeleton.md`，**起草时打开它**）：

1. **以终为始 · 先知己**：先落一段知己小结——我方产品定位、所处生命周期阶段、核心用户特征、本次分析目标。据阶段选宏观框架（起步/成长/成熟对应不同打法）。知己信息不足就先问或合理假设 + 标注。
2. **分层划竞品范围**：把纳入分析的竞品按**核心（直接竞争）/ 重要（间接竞争）/ 跨界（灵感来源）**分层；按分析目标（决策预警 vs 学习借鉴）说明为何选这几个。**别贪多**——重点竞品投入重精力，其余略。
3. **读懂意图 · 战略意图四问**（核心环节）：对每个值得关注的竞对动作，逐一走四问——**Q1 时机**（为什么是现在：临近大促？顺应热点？数据承压？）、**Q2 场景**（解决谁的什么问题：新/老/会员/全量？高频/低频？体验/变现？）、**Q3 动机**（用 **AARRR** 套一下这个功能撬的是获客/激活/留存/裂变/收入哪一环）、**Q4 市场**（用 **SWOT** 摆清我方与竞对优劣势 + 判断这动作**是否威胁我方核心奶酪**）。**每条意图判断都以「推断：…」落笔，不写成竞对官方事实。**
4. **用户体验五要素深度拆解 + 打勾对比**（成长/成熟期产品）：按**战略层 / 范围层 / 结构层 / 框架·表现层**拆解竞品（颗粒度按项目定，不必都到「原子」）；在元素/功能层用**打勾对比法**——列出功能项，逐格对核心竞品打勾，用**人有我无 / 人有我有 / 人无我有**的逻辑看清双方优劣势，重点关注「人有我无」。
5. **结果解读 + 跟进决策**：把竞对动作按时间维度归纳其战略层次（**短期战术 / 中期战略 / 长期布局**）；再用**跟进决策 2×2 矩阵**（是否威胁核心 × 是否值得跟进 → 必须回应 / 选择性优化 / 战略防御 / 忽略）对每个动作定应对基调。
6. **转化为行动**：给出**差异化、可落地**的行动建议——结合自身特色（如自营/PLUS/物流）做深做透，而非复制功能；重点回答「人有我无」处我方要不要补、怎么补。要挖增长机会点时衔接 `opportunity-research`。
7. **就地标注**：基于 draft 正典、靠推断的意图（标「推断」）、素材没覆盖到的竞品/功能（标「未覆盖」），都在该处标一句（呼应纪律 3）。

## 第 3 步 · B：产框架/工具包（无竞品素材时）

产出一份针对本次目标与竞品范围的竞品分析框架（结构见 skeleton），并**末尾明确索要竞品观测素材与知己信息**（「把要分析的竞品、看到的功能/页面/策略发我，再告诉我我方的定位和所处阶段，我可以直接做一遍」）：

- **知己清单**：先要用户补齐我方定位/生命周期/用户特征（附「为何先知己」一句）；
- **竞品分层监控框架**：核心/重要/跨界三类的选取标准 + 按目标（决策预警/学习借鉴）从品牌·品类·替代品·参照品选品的建议；
- **战略意图四问清单 + AARRR 映射表 + SWOT 模板**：四问的问法（取自正典）、AARRR 五环节-典型功能对照表、SWOT 四象限模板；
- **用户体验五要素拆解清单 + 打勾对比表模板**：五层各看什么 + 打勾对比表表头（功能项 × 各竞品）；
- **跟进决策 2×2 矩阵模板**：威胁核心 × 值得跟进 → 四类应对。

## 第 4 步：分析自检（拿「局限与误用」回头扫一遍）

**实时读 `competitive-analysis.md` 的「局限与误用」**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **只记录不读懂？** 每个竞品动作是否都推导了意图 + 对我方影响，而非停在「对手上线了 XX」的功能罗列？
- **盲目跟风/复制功能？** 每个「人有我无」的差距是否都过了跟进决策矩阵（是否动核心奶酪 / 是否值得跟进），而非默认「对手有我就要抄」？
- **贪多求全？** 是否有深度、有逻辑的结论 + 可落地建议，而非堆竞品数量、把拆解颗粒度做到「原子」却没结论？
- **脱离自身？** 是否先「知己」（定位/阶段/用户场景）再「知彼」，分析方向未跑偏？
- **意图当事实？** 战略意图判断是否都以「推断」口吻落笔，没写成竞对官方声明、没编造竞对内部动机/数据？
- **脑补竞品？** 竞品功能/页面/策略/数据是否都基于真实观测素材，没凭想象编造？素材没覆盖的是否标了「未覆盖」？

把改动落到分析上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/competitive-analysis-skeleton.md` 的顺序输出：

- **直接分析模式**：① **竞品分析结论摘要（结论先行，放最前）**——一句话核心判断（竞对在下什么棋、对我方最大的威胁/机会是什么）+ **Top 3–5 应对建议速览**（每条：动作 + 跟进基调 + 一句怎么做），让人第一眼看到「最该先应对的是什么」，细节在下方展开 ② 竞品分析报告（知己小结 · 竞品分层范围 · 战略意图四问[含 AARRR/SWOT] · 用户体验五要素拆解 + 打勾对比 · 结果解读[短/中/长期] + 跟进决策矩阵 · 差异化行动建议）③ 分析自检结果 ④《本次运行说明》（五部分）

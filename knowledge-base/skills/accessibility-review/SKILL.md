# Accessibility Review — 评估对象 + 范围 → 无障碍走查（编排 wiki 正典）

吃一个评估对象与评估范围，**默认直接当无障碍评估员做一次无障碍走查 / 审查**：借读屏视角逐屏、逐流程地检视界面是否满足无障碍规范，找出违反点、定 A/B/C 等级、给**分平台整改方式**。本 skill 不发明无障碍准则，而是**实时编排**既有正典——无障碍适配规范（查什么：POUR 四原则、9 类控件规范、A/B/C 定级、checklist）、无障碍走查方法本体（怎么查：开读屏 / 三遍法走查节奏 / WCAG 量化阈值 / 汇总）、问题分层与优先级（怎么定级排序）——把它们用到 *这一个* 对象、*这一个* 范围上，产出的核心是一份能直接派给研发的**整改清单**。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** POUR 四原则的内涵、9 类控件的规范要点、A/B/C 定级的问题类型口径、读屏走查的节奏、对比度 / 触达区等 WCAG 阈值，唯一出处都在 wiki 正典里。你的工作是**读对的正典、抽对的标尺、用到本次对象上**，而不是凭记忆默写"无障碍四原则是哪四条""对比度要几比几"。产出的价值在于贴合 *这一个* 界面、*这一类* 视障用户、*这一条* 核心流程的真实适配问题与可落地整改，不是又一份通用规范讲解。
- **两种形态，按"有没有界面素材"自动切换：**
  - **有素材就直接审（默认、主线）。** 无障碍走查是**专家检视法、不需要先招真实视障用户**，所以你（LLM）可以亲自当评估员。用户给了截图 / 原型链接 / 可访问页面 / 读屏录音录屏 / 流程或文字描述，就**直接做走查、出问题清单与整改清单**，而不是只递给他一套"工具包"让他自己评。
  - **没素材才产工具包。** 若用户只问"怎么做无障碍审查"或没给任何可评的界面，就产出**可执行的无障碍走查工具包**（交付前 checklist + A/B/C 评级卡 + 读屏三遍法走查脚本 + 量化阈值速查 + 记录 / 整改汇总模板），并**明确索要界面素材**——拿到后即可切到"直接审"。
- **违反一条 ≠ 一定是问题。** 这是无障碍走查最大的误用。装饰性图像本就该被标记为忽略朗读、层级弹窗焦点本就该锁定——把这些"当成问题"就是伪问题。每条疑似问题都要**结合场景与规范里的例外（装饰性图像 / 敏感信息朗读策略等）判断**，并落到 **A/B/C 等级**——别把"提到某条规范"当成"违反"、堆一长串伪问题。低等级也要如实标低，别为凑数注水。
- **替代不了真人验证，要说清楚。** 你做的是"专家视角 + 读屏可达性的疑似问题扫描"。真实视障用户的辅助设置、读屏熟练度、操作习惯千差万别，且**不开真机读屏时你对"实际会怎么朗读"只能推断**。凡是需要真机读屏复测或真实视障用户行为才能定论的，**标"需真机 / 真人验证"**，并在运行说明里建议接 `usability-testing`（真实视障用户任务测试）——绝不把走查结论包装成"已通过无障碍验收"。
- **边界：拿无障碍规范 / 读屏视角查界面 → 本 skill；通用可用性 / 真人做任务 / 竞品意图 / 反馈打标 → 隔壁。** 要拿尼尔森十大原则查**通用可用性**问题（好不好用、违反哪条可用性原则，不专门针对视障 / 读屏），用 `run-heuristic-evaluation`（本 skill 是它的无障碍专项姊妹）；要让真实用户（含视障用户）做任务并观察，用 `generate-usability-test`；要读竞品战略意图，用 `competitive-analysis`；要把海量用户原话归类打标，用 `code-open-feedback`。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把 POUR 四原则的内涵、各控件规范要点、A/B/C 定级的档位与问题类型、对比度 / 触达区等 WCAG 阈值写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——规范可能已从 draft 改为 reviewed、阈值或控件示例可能已补充。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份评估建立在什么之上、哪里可能不牢。结构见 `references/accessibility-review-skeleton.md`，五个部分一个都不能少。
3. **在正文里就地声明局限**：基于 draft（未评审）正典的部分、因素材不足而无法走查到的流程 / 状态 / 平台、**未开真机读屏而靠界面推断"读屏会怎么读"的判断（标"需真机验证"）**、需真实视障用户才能定论的（标"需真人验证"），都在该处直接标注，别让使用者误以为处处有正典背书、处处已实测。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空默写一套"POUR 四原则"或"A/B/C 档位""对比度 4.5:1"冒充正典。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。** 绝不靠目录列举、shell 报错或印象就断定"它不存在"。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。同理，评估发现必须基于用户提供的真实界面素材，**不得脑补界面长什么样、更不得编造"读屏朗读了 XXX"**——素材不足就说不足、索要素材或标"需真机验证"，不要凭想象编造读屏表现或问题。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/standards/accessibility-adaptation.md` 或 `**/methods/toolbox/collection/accessibility-audit.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与「文件是否存在」的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一套无障碍规范糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用 WCAG 经验」，且**不得**伪造任何「依据本库无障碍规范第 X 条 / A 级」的说法。

## 第 1 步：解析 brief

从用户输入里抽出要素，决定走"直接审"还是"产工具包"、读什么、产出长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **评估对象 + 素材形态** | 决定能不能直接审。形态：截图/录屏、可点原型链接、可访问线上页面、设计稿、读屏录音/录屏、或足够具体的流程/文字描述 | **有任一可评素材 → 直接审模式**；**完全没有可评素材 → 工具包模式 + 索要素材**；连"评什么对象"都没有 → 先问清 |
| **评估范围 / 目标** | 决定走查哪几条核心流程、聚焦哪个端/板块/任务/页面 | 缺 → 默认评"对象最核心、最高频的主流程"，并在运行说明声明范围假设 |
| **平台** | 决定**整改方式口径**（无标签在 Android 用 `contentDescription`、iOS 用 `accessibilityLabel`、H5 用 `aria-label`；取自规范 A/B/C 表"实现方式"列） | 缺 → 能从素材识别就按素材平台；否则给三平台通用整改方式并声明 |
| **是否已有读屏实测（录音/录屏）** | 有则可核**读屏实际朗读**、证据更硬；无则只能基于可见素材**推断**读屏会怎么读，须标"需真机验证" | 缺 → 基于截图/设计稿推断读屏表现，逐条标"需真机验证" |

判断原则：**对象缺了就问；范围/平台/目标等次要参数缺了就合理假设 + 声明**——不要因为缺一个"评哪个端"就卡住，也不要在没有任何可评素材时硬编一份"看起来像评估"的问题清单（那违反纪律 4）。

## 第 3 步 · A：直接审（默认模式 —— 有界面素材时）

你现在就是无障碍评估员。按走查方法本体的**三遍法 + 走查节奏**，对用户提供的素材逐屏 / 逐流程检视：

1. **第一遍 · 视觉整体浏览（先不"读屏"）**：先整体过一遍熟悉流程；同时核**不依赖读屏就能看到的**——颜色是否被当作区分信息的唯一手段、文本 / 图形**对比度**、**触达区**大小、**字号放大**可读性、有无高频闪烁 / 自动动效（对照运行时读到的 WCAG 量化基准）。
2. **第二遍 · 读屏线性遍历**：想象（或依据用户给的读屏录音 / 录屏核实）单指从左到右、从上到下逐个遍历，逐元素核：① 可交互控件有无**可读标签**；② 图片 / 图标有无**替代文本**（装饰性应忽略、承载信息必须朗读）；③ **控件类型**是否正确（按钮读"按钮"而非"图像"）；④ **状态变化**是否朗读（选中 / 禁用 / 加载 / 成功失败）；⑤ **焦点顺序**是否符合视觉阅读顺序，有无**焦点陷阱 / 底层焦点 / 大焦点覆盖 / 焦点过细**；⑥ **弹窗**焦点是否移入首元素、关闭后回到触发点、能否误聚焦到弹窗外；⑦ 朗读有无**冗余 / 不完整 / 错误**。**没有读屏录音时，把"读屏会怎么读"当作推断、逐条标"需真机验证"，不得编造"实际朗读了 XXX"。**
3. **第三遍 · 键盘 / 手势与边界**：核自定义手势是否与系统读屏手势冲突；刻意试**逻辑边界 / 异常态**（空态、错误态、加载中、无货、回退、超时）——状态朗读缺失最易在边界暴露；核读屏下有无卡顿 / 闪退 / 延迟（兼容性与性能）。
4. **对照规范逐项核**：拿规范的 **8 项 checklist + 9 类控件详规**回扫，确保各控件类型都覆盖到、别漏类型。
5. **逐条记问题**，每条落这几项（结构见 skeleton）：
   - **位置 / 证据**：在哪个页面/步骤，引用素材里的具体证据（"截图2 顶部加购按钮"）与**读屏实际表现**（有录音则引原文，无则标"推断，需真机验证"）——**不脑补界面、不编造朗读**；
   - **违反的规范条目**：对到规范里的哪条原则（POUR）+ 哪类控件规范（可多条）；
   - **等级**：按规范自带的 **A/B/C 问题定级**打级（A 级最严重、直接阻断），给出理由；
   - **分平台整改方式**：抽自规范 A/B/C 表"实现方式"列（Android / iOS / H5），具体可落地；
   - **复测验证点**：改完后怎么用读屏复测确认（如"TalkBack 聚焦该按钮应朗读'加入购物车，按钮'"）。
6. **汇总输出**：无障碍问题清单（表）+ **按 POUR 原则归类**的小结 + **按等级排序**的 Top（A 级优先）+ **整改清单**（分平台改法 + 复测点）+ 接 `issue-prioritization` 的修复优先级（P0–P3）。
7. **就地标注**：基于 draft 正典、未开真机而靠推断的读屏判断（标"需真机验证"）、素材不足没走查到的流程 / 状态 / 平台（标"未评到"），都在该处标一句（呼应纪律 3）。

## 第 3 步 · B：产工具包（无界面素材时）

产出一份可直接执行的无障碍走查工具包（结构见 skeleton），并**末尾明确索要界面素材**（"把要评的页面截图 / 原型链接 / 读屏录屏发我，我可以直接走查一遍"）：

- **交付前 checklist**：抽自规范的 8 项交付前 checklist（别默写）；
- **A/B/C 评级卡**：取自规范的 A/B/C 问题定级表（问题类型 → 等级 + 各平台实现方式）；跨问题排序取 `issue-prioritization` 的 P0–P3；
- **读屏三遍法走查脚本**：取自 `accessibility-audit`——开读屏（VoiceOver/TalkBack）、视觉遍 / 读屏遍 / 键盘手势遍的走查节奏、范围收窄、机型覆盖、独立评估；
- **量化阈值速查**：对比度 / 触达区 / 文本缩放 / 闪烁等 WCAG 基准（取自 `accessibility-audit`，运行时现读）；
- **记录表 + 整改汇总模板**：问题记录字段（位置 / 读屏表现 / 违反条目 / 等级 / 分平台整改 / 复测点）+ 亲和图聚类去重 + 按等级排序。

## 第 4 步：评估自检（拿"局限与常见坑"回头扫一遍）

**实时读 `accessibility-adaptation.md` 的"禁止（MUST NOT）/ 例外"与 `accessibility-audit.md` 的"常见坑 / 关键技巧"**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **违反 ≠ 问题**：每条疑似问题是否结合了场景与规范例外（装饰性图像忽略朗读、敏感信息朗读策略）判断？有没有把"提到某条规范"当"违反"、堆伪问题？把站不住的降级或删掉。
- **等级**：是否按规范 A/B/C 的问题类型给的，而非拍脑袋？是否如实给了低等级项、没为凑数注水？
- **覆盖**：9 类控件 + 核心流程 + 边界 / 异常态是否都走查到？素材没覆盖到的，是否标了"未评到"而非假装评过？
- **不替代真人 / 真机**：未开真机而靠推断的读屏判断，是否标了"需真机验证"？需真实视障用户才能定论的，是否标"需真人验证"并建议接 usability-testing？
- **证据**：每条是否有素材里的具体证据（截图 / 读屏录音）支撑，**没有编造"读屏朗读了 XXX"**？
- **整改可落地**：每条是否给了分平台、具体可执行的整改方式与复测点，而非空泛"加个标签"？

把改动落到问题清单与整改清单上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/accessibility-review-skeleton.md` 的顺序输出：

- **直接审模式**：① **无障碍评估结论摘要（结论先行，放最前）**——一句话总体无障碍达标判断 + **Top A 级 / 高危问题速览**（每条：问题 + 等级 + 一句整改方向），让人第一眼看到"最该先修的阻断项"，细节在下方展开 ② 无障碍走查报告（问题清单 · 按 POUR 归类 · 按等级排序 · **整改清单**[分平台+复测点] · 修复优先级 P0–P3）③ 评估自检结果 ④《本次运行说明》（五部分）
- **工具包模式**：① 无障碍走查工具包（交付前 checklist · A/B/C 评级卡 · 读屏三遍法走查脚本 · 量化阈值速查 · 记录 / 整改汇总模板）② 对素材的索要 ③《本次运行说明》（五部分）

# Synthesize Qualitative Insights — 多份定性材料 → 跨场主题 + 结构化洞察（编排 wiki 正典）

吃一批定性材料和一个研究目标，产出**这一批专用**的主题归纳与洞察。本 skill 不发明分析方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的分析正典——亲和图法、质性编码、定性洞察四框架、「结果→结论→洞察」模型、报告写作规范——把散落在多份材料里的发现先**横向聚类**成主题、再**纵向拔高**成洞察。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 分析方法的唯一出处在 wiki 正典里。你的工作是按本次材料与研究目标去**读对的正典、抽对的要点、套进分析骨架**，而不是凭记忆复述"亲和图怎么做""洞察是什么"。产出的价值在于贴合 *这一批材料* 和 *这一个* 研究目标，不是又一份通用方法说明。
- **两步连着做：聚类 → 拔高。** 先把多份材料里的散落发现自下而上归并成多层主题（这是"事实"层），再把主题提炼成能指导行动的洞察（这是"洞察"层）。两步在实操里是连续的——主题是洞察的论据，洞察是主题的升华。只给主题不拔高，等于停在描述；只给洞察不挂主题，等于没有论据。
- **跨场是关键。** 本 skill 的独特价值在"横向"：同一个痛点在几份材料里反复出现，才是高频共性；只在一份里出现的，要么是边缘个案、要么是待验证信号。务必让"出现在几份材料里"这件事真正进入你的主题频次与洞察强度判断。只有 1 份材料时仍可做单场聚类 + 拔高，但要在运行说明里说明"单场、跨场共性无从谈起、频次仅为单场内计数"。
- **亲和图原是团队协作法，你是单机复刻。** 正典里的亲和图法靠 6–8 人开数据转化会、在墙上协作归并、最终达成共识。你一个 agent 在文本上能复刻它的**编码规则、自下而上归并逻辑、复查自检**，但复刻不了"跨职能团队共识"那一环。所以你的产出是**给人去对齐、去拍板的高质量初稿**，不替代真实的跨职能评审——这一点要在产出里诚实说明，别让人以为洞察已经过团队背书。
- **两态分离：分析底稿 ≠ 交付稿。** 你在自下而上分析时产生的是**分析底稿（工作稿）**——带便签、D/Q、框架名、结果→结论推导链、《运行说明》，这些方法论痕迹利于自检与追溯，**保留**。但直接把工作稿当交付物递给业务，就是本 skill 最常见的返工来源。**交付稿默认剥离方法论词汇**（不出现"亲和图/洞察定义/框架名/D·Q/结果→结论"这类术语），**按用户的研究问题组织、结论先行**（编排 `pyramid-principle` 与 `research-report-writing`），只留"结论 + 论据 + 匿名原话"；方法论痕迹与《运行说明》**收进后置的「分析底稿」附录**。默认两者都给：交付稿在前当正文，分析底稿在后当附录。
- **下游延伸：编排优先、按需触发、弱支撑标注。** 洞察之后常还要往下走——分层定位、负向边界、落地/命名建议、可视化。这些**大多仍是编排既有正典**（结论组织→pyramid-principle；分层→means-end-laddering / user-needs / kano / user-cultivation-growth；落地→机会-解决方案树 / continuous-discovery / 优先级框架 / 命名研究），不是重造。纪律有二：① 落地/命名/信息架构类建议属**未验证的设计假设**，不是研究结论，须显式标"需原型测试 / 灰度 / A-B 验证"；② 下游各块**只在用户要时才做**（要机会才给落地桥、要图才出可视化），不为"更全"而硬塞，别喧宾夺主盖过"可信洞察"这个内核。

## 敏感素材处理（本 skill 特有，务必遵守）

研究员上传的逐字稿、访谈记录、定性笔记属**敏感原始数据**：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户材料的内容写进 `research-wiki/` 任何文件，不要落盘到 skill 目录，不要在运行说明里回贴大段原始材料。分析完即止于本次会话。
- **引用原话时做匿名化，而且要彻底。** 产出里引用用户原话来当论据是对的（增强代入感、降低沟通阻力），但要去除与论点无关的 PII（真实姓名、手机号、住址、单位全称、订单号等），在不损失含义处用"用户A""一位资深用户"等代称。**匿名化必须贯穿产出的每一个角落**——正文、原话署名、数字化亲和图表，以及最容易被忽略的《本次运行说明》，里面都不得出现真实姓名/单位/PII。有一个常见的"自我拆台"：在运行说明里为了*交代*自己做了匿名化，又把"真名 → 代称"的对照写出来（如"含真实姓名 张三/某公司，已匿名为用户A"）——这等于没匿名，原始 PII 照样泄漏进了交付物。要说明匿名化，只陈述"已对受访者姓名与单位做匿名化处理"，**绝不回写原值**。涉及姓名/联系方式等字段时尤其当心——既是隐私要求，也避免触发内容网关拦截。
- **材料只读不改。** 你的职责是分析它，不是修订它。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **固定附《本次运行说明》**（随分析底稿置于交付物末尾的附录里），让使用者知道这份分析建立在什么之上、哪里可能不牢。结构见 `references/synthesis-skeleton.md`，五个部分一个都不能少。
3. **在产出正文里就地声明局限**：基于 draft（未评审）正典的部分、因未找到贴合方法而用通用做法兜底的部分、机会点这类弱支撑的部分，都在该处直接标注，别让使用者误以为处处有正典背书。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法或结论。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先实际核验该路径确实不存在。** 把真实存在的正典误报成缺失，和编造内容一样有害（会让人误以为知识库缺了这块）。本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read。
   - **判定某个具体文件"走空 / 不存在"的唯一充分证据，是你用 Read 打开那个确切路径、它确实报错。** Glob 没匹配到、或在一次目录列举里"没看见"，都**不算**充分——pattern 可能写偏、列举可能被过滤或截断、相近文件名（如 `interview-guide-design.md` 与 `interviews.md`）容易让人误判。据一次扫描就断言某 related 链接"走空"，正是最常见的误报。凡没用 Read 逐一核验过的链接，一律在运行说明里写"未逐一核验、不在此断言"，而不是猜它走空。
   - **更要命的是对材料本身的编造。** 主题的频次、用户的原话、谁说了什么，必须逐条能在用户给的材料里找到出处。绝不为了让主题更"成立"而虚构一句原话或抬高一个频次——这是分析的生命线。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/toolbox/analysis/affinity-diagram.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与"文件是否存在"的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一套分析糊弄。如实告诉用户"未能访问 research-wiki 正典"，请其确认知识库位置后再跑；确需先给草稿时，明确声明"未基于正典、仅通用经验"。

## 第 1 步：接收并解析输入

先把"分析什么、按什么目标分析"搞清楚，这决定后面读什么、主题怎么聚、洞察往哪拔：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **研究目标** | 分析的主轴，决定聚类的着眼点与洞察的落点 | 完全缺失 → 先问清，这是不能假设的核心；漫无目的地聚类只会得到一堆平庸分类 |
| **材料份数与类型** | 几份？是访谈小结 / 逐字稿 / 定性笔记，还是开放题 / 评论 / 工单？决定补读哪些方法、用哪条聚类路径 | 缺 → 按拿到的材料如实判断并声明 |
| **目标用户 / 研究阶段** | 决定主题取舍与洞察视角（新/资深、C/B 端、规划/迭代期） | 缺 → 合理假设并在运行说明声明 |
| **研究问题清单** | 交付稿要**按用户的研究问题**组织、逐条结论先行地回应（第 6 步），不是按分析框架排 | 目标只有一句 → 主动替用户拆成几条子问题、回读确认，再据此定交付稿骨架 |
| **是否要机会 / 落地 / 命名** | 决定是否触发第 5 步「机会与落地建议桥」、要不要读命名/优先级正典 | 没提 → **默认不做**，只交付主题 + 洞察；绝不硬塞落地方向 |
| **期望交付形态** | 是否要一页纸摘要 / 可汇报 PPT / 结论示意图（第 6 步可视化） | 没提 → 默认给 markdown 交付稿，并说明可按需追加一页纸 / PPT 大纲 / 示意图 |

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**。研究问题清单即便要靠拆解也务必先拉出来——它是交付稿组织的骨架。同时按上面「敏感素材处理」对待材料。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着"我要从这抽什么用于聚类/拔高"的目的，而不是通读。先快速看相关 `index.md`（`methods/toolbox/analysis/index.md`、`models/index.md`、`methods/standards/index.md`）确认有哪些条目、再决定打开哪几篇。读完记下每篇相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于产出 |
|---|---|---|
| `methods/toolbox/analysis/affinity-diagram.md` | **必读（聚类骨架）** | 便签编码规则（左上角用户编号/假名、右上角 `D`=design idea / `Q`=question 标记、中间一条完整句子、一签一要点）；初始类目阶段先不打标签→给类目写 **level 1** 标签（详细短语或句子、第一人称描述用户行为与意愿）→类目多时再聚成 **level 2**；数据复查的自检问题（分类是否展现关注点、标签是否准确、理解有无漏洞、level1 类目能否相互联结、有无待后续研究的问题）；结果解读（从类目标签读共性痛点/需求/机会，D/Q 指向待沉淀想法与待研究问题） |
| `methods/toolbox/analysis/qualitative-insight-frameworks.md` | **必读（拔高骨架）** | **洞察的定义**（洞察 = 用户 + 具体场景 + 想完成的任务 + 希望达成的情感需求/价值 + 面临的矛盾或障碍）；四框架——用户旅程地图 / 价值主张画布 / 同理心地图（Say-Do-Think-Feel）/ ABC（情感-行为-认知）——及**选型**（梳理完整流程与情绪起伏→旅程地图；深度共情、定义一类典型用户→同理心地图；检验方案是否击中需求→价值主张画布；解构单一现象/矛盾→ABC）；过程纪律（分析非线性、框架初期就选定、信息对等原则） |
| `models/user-insight.md` | **必读（定义"什么才算洞察"）** | **结果→结论→洞察三层**（结果=直接观察/测量到的；结论=对结果做原因判断并**高度凝练**；洞察=能指导业务行动的策略建议）；从结论到洞察持续追问"如何呢、又能怎"并结合竞品/业务/后台数据做到逻辑自洽；三个认知误区（搜集痛点≠洞察、痛点是表象而需求在场景深处、停在描述性结果就收工）；"把自己作为方法"——别丢掉对数据背后那个"人"的感受力 |
| `methods/standards/research-report-writing.md` | **必读（组织产出）** | **重在事实、高在洞察**；**论点配论据**（每个结论都配数据/原话/观察/文献，交代"结论从何而来"）；一页/一段只讲一个主题；完成后能否自讲一遍**串成一条主线**（可按用户流程、或拉新-留存-促活等业务线来串）；**六股结构**（项目背景 / 执行概要 / 研究内容 / 主体 / 整体结论 / **附录**——方法论与底稿可下沉到附录）；**对照业务方原始需求逐条核对**（呼应交付稿按研究问题逐条回应）；**克服完美主义、先出初版再迭代**；引用用户原话增强代入感 |
| `models/pyramid-principle.md` | **组织交付稿 / 一页纸 / PPT 时读**（status: draft） | 结论先行、SCQA 序言（背景-冲突-疑问-回答）、以上统下、每个标题**提炼思想精华**（"只看标题就懂"，别用"现状/问题/建议"空标题）、同一层论据**演绎或归纳不可混用**——把交付稿按研究问题、结论在前地组织的核心正典 |
| `methods/toolbox/analysis/qualitative-coding.md` | 材料是**逐字稿/深度对话**、要做深读时读 | 三进程编码（开放式贴标签→主轴式按关系维度串集群→选择式抽象归纳）作为聚类的更细方法；意义凝聚五步；可预设编码库再据材料补充；证据链「结论+证据+建议」；三种效度 + **找反面例证/追踪特殊情况/多重视角**（防只挑支持既有假设的证据） |
| `methods/toolbox/analysis/voc-analysis.md` | 材料是**海量开放题/评论/工单**、偏量化收口时读 | 主题分析六步（熟悉数据→生成初始编码→归集→搜索主题→检视精炼→撰写）；演绎（套预设标签）vs 归纳（从数据长出新主题）；**编码本**（每个标签给定义+正反例，保证一致）；量化收口（各主题频次/占比/情感分布）；Top 痛点 = 高频 × 高负面 × 高影响，每条配 2–3 句代表原话 |

## 第 3 步：跨场归纳（聚类）——把散落发现聚成多层主题

用亲和图法的骨架做（详见 `references/synthesis-skeleton.md`，**起草时打开它**）：

- **拆便签（atom）**：通读每份材料，把每条独立的发现/痛点/需求/原话拆成一条"便签"，按正典规则标注——记下它来自**哪份材料/哪位用户（用代称）**，右上角标 `D`（用户提到的产品设计想法）或 `Q`（需后续研究的疑问），其余为痛点/体验。一签一要点，用完整句子。
- **自下而上归并**：把相似的便签纵向归到一起，**先不急着命名**；让分类自然浮现后，再给每个类目写 **level 1** 标签（第一人称、完整短语/句子描述用户行为与意愿，如"我担心错过连续签到的奖励"而非干巴巴的"签到"）。类目多时聚成 **level 2** 主题。
- **标频次与证据**：每个主题标"**出现在 N 份材料 / M 位用户**"（这是跨场共性的硬指标），并挂 1–3 条**代表性原话（已匿名）**当论据。频次必须真实可回溯，不得为凑高而虚标。
- **复查自检**：拿正典的复查问题回扫一遍——分类是否真的展现了对研究目标的关注点？标签是否准确？level1 类目之间能否联结？有没有冒出新的 `Q`？**主动找反面例证**：有没有与主流主题相悖的个案？别只挑支持既有印象的便签（见 qualitative-coding 的多重视角原则）。
- **按材料类型选辅助路径**：逐字稿深读 → 配合 `qualitative-coding` 的三进程编码细化；海量开放题/评论/工单 → 配合 `voc-analysis` 六步与编码本，并做量化收口（频次/占比）。**份数少但单份极长/极深**（如 5–6 份深度逐字稿）——尤其要走 `qualitative-coding` 的三进程编码（开放→主轴→选择）+ 意义凝聚五步**逐份细读**，别只在小结层面做浅层跨场归并：高密度材料里真正的洞察埋在细节和上下文里，浅读会漏。

## 第 4 步：拔高洞察——把主题升华成能指导行动的洞察

主题是论据，洞察是论点。按 `user-insight` 的三层推进，按 `qualitative-insight-frameworks` 的定义与框架成形：

- **先自检"我在哪一层"**：当前这句是**结果**（只是观察到的现象/频次）、**结论**（对原因的高度凝练判断）、还是**洞察**（可指导行动的策略建议）？停在结果或结论都不算完成。
- **从结果到结论**：别只复述"68% 的人提到 X"，要高度凝练出背后的原因判断——凝练不是形式主义，是跳出琐碎现象、放大思考范畴的关键。
- **从结论到洞察**：持续追问"如何呢、又能怎"，结合竞品/业务/后台数据（若材料或用户提供）做到逻辑自洽——能说服自己也能说服别人，再落为可行动的策略建议。警惕三误区：别把一筐痛点直接当答案；记住痛点常是表象、真需求在用户的场景里（"用户要的不是除草机，是不用自己除草的整洁草坪"）；别交付一份"全是描述性结果"的报告。
- **每条洞察结构化**：套用洞察定义把它写成——**用户** 在 **某个具体场景** 下，想完成 **某个任务**，因为希望达成 **某种情感需求/价值**，但面临 **某个具体矛盾/障碍**。再选**一个最贴的框架**承载它（按 frameworks 的选型规则），并挂上第 3 步的主题与原话当论据（呼应 report-writing 的"论点配论据"）。
- **（可选）分层 / 阶梯洞察形态**：当几条洞察之间是"逐层进阶 / 成熟度阶梯 / 需求金字塔"这类**层级关系**（而非平铺并列）时，别硬摊平——用分层形态表达：**层级 + 每层定义 + 用户当前所处层 + 目标层**。按层级的含义选**一个**透镜（第 2 步分层 models 行）：价值阶梯 → means-end-laddering；需求深浅 → user-needs；需求强度 / 发展阶段 → kano；用户养成/生命周期 → user-cultivation-growth。这类结论天然适合配示意图（阶梯/金字塔，见第 6 步可视化）。
- **负向边界（标准动作）**：除了回答"需求在哪"，也给出"**需求不在哪 / 不该硬推哪**"——这对产品决策与正向发现同等重要。抓手：`kano` 的**无差异型**（做了也没用，别投入）与**反向型**（做了反招不满，别做）；`user-needs` 的"**个性需求≠普遍需求**"（某用户特殊场景的诉求别当普适需求去推）、"痛点≠需求"。把"别把 X 当主打去硬推"这类边界显式写进洞察。
- 骨架与字段见 `references/synthesis-skeleton.md`。

## 第 5 步（可选，弱支撑，按需触发）：从洞察到机会与落地建议

**仅当**研究目标或用户明确要"机会点 / 增长方向 / 落地 / 命名"时才做，全程标弱支撑。按需要取下面几块，不必都做：

- **机会方向**（要"增长/机会点"时）：用 `opportunity-research` 的找亮点 / 学友商 / 填空白三法组织候选机会，每条按其三大局限自检（这是不是幸存者偏差？是不是难追的先发优势？补了这个空白用户真的在乎吗？）。⚠️ 该正典偏薄、无打分矩阵/结构模板，只给"候选方向 + 局限提示"，别假装有标准化打分。
- **落地桥**（要"落地/方案/怎么做"时）：用 `opportunity-solution-tree`（OST）把已确认洞察编排成一棵树——**业务结果 → 用户机会 → 候选方案 → 验证实验**，把主题/原话/机会挂进对应节点当证据。核心纪律来自 `continuous-discovery` 的"**为学习而非为确认而发现**"：**候选方案 / 命名 / 信息架构建议都是待验证的设计假设，不是研究结论**，每条显式标"需原型测试 / 灰度 / A-B 验证"，别让人当已验证结论去拍板。
- **排序**（候选多、要收敛时）：用 `priority-frameworks-overview` 按信号词选一个优先级框架；最实用是 **KANO + 四象限组合**——先用 kano 滤掉无差异型/反向型，再用 `priority-quadrant-method` 四维打分排序。
- **命名**（要"起名/命名验证"时）：走 `brand-naming-research`——命名六原则（受法律保护 / 易发音记忆 / 新颖独特 / 暗示功能 / 启发联想 / 自由延伸）+ 测评四法（问卷 / 电话 / 输入法 / A-B），是从一批候选里选优、而非验证单一名字；命名结论同属待验证假设。
- **别硬塞**：用户没要就**不做**——别为了产出更"全"而生造一堆没人要的机会/方案。落地建议再漂亮，也标清"需到更广人群 + 原型/灰度验证才能当决策依据"。

## 第 6 步：组织交付（两态：交付稿正文 + 分析底稿附录）

产出分两态，**默认都给**：交付稿在前当正文，分析底稿在后当附录。骨架分别见 `references/delivery-skeleton.md`（交付稿 + 可视化 + 落地桥）与 `references/synthesis-skeleton.md`（分析底稿）。

**A. 交付稿（正文，给业务/干系人看）—— 剥方法论、结论先行、按研究问题组织**
- 编排 `pyramid-principle` 与 `research-report-writing`：**按用户的研究问题**逐条组织，每条**结论先行**，下挂论据（数据 / 匿名原话 / 观察）。
- **剥离方法论词汇**：正文里**不出现**"亲和图 / 洞察定义 / 框架名（旅程图·画布·同理心图·ABC）/ D·Q / 结果→结论→洞察 / 便签 / level 频次"这类术语——那是你的分析过程，不是给业务看的话术。频次照用，但说"6 位里有 4 位…"而非"该 level-2 主题频次 4/6"。
- **分层结论**用阶梯/金字塔表达（第 4 步）；**负向边界**（不该推什么）显式写出。
- 主线自检：能否按用户流程、或拉新-留存-促活把交付稿讲成一条线。

**B. 分析底稿（后置附录，给要追溯的人看）—— 保留全部方法论痕迹**
- 主题归纳（level1/level2 + 频次 + 匿名原话 + D/Q，可选数字化亲和图）、结构化洞察（五要素 + 结果→结论→洞察推导链 + 所用框架 + 论据）、以及**《本次运行说明》五部分**，全部收在这里。骨架见 `synthesis-skeleton.md`。

**这是可迭代初稿，不是终稿。** 定性报告的**结构**通常会随干系人反复调（按问题重组 / 精简场景 / 合并章节 / 把画像等挪进附录）。交付时说明"这是可迭代初稿"；遇到结构性大改，**整体重写往往比局部打补丁更干净**（`research-report-writing`：克服完美主义，先出初版再迭代）。

**（可选）可视化交付**——用户要、或结论天然图形化（分层/象限/流程）时给，形态见 `delivery-skeleton.md`：
- **一页纸摘要**（markdown，结论先行）、**可汇报 PPT 大纲**（markdown，套六股）、**结论示意图**（用 Mermaid / ASCII 出阶梯·象限·流程·OST 树）。
- 真正的 `.pptx` / 图片本 skill **不直接产**，交给 `anthropic-skills:pptx` 姊妹 skill 或用户，并说明如何交接。
- **跨交付物风格一致**：多个交付物（正文 / 一页纸 / PPT / 图）的措辞、层级命名、口径要一致，别各说各的。
- **长度分层**：交付物偏长时，主动提供"**一页纸摘要 + 详版**"双层，让人先看结论再下钻。

**交付前自检**：交付稿是否已**剥净方法论术语**、按研究问题逐条回应了？每条结论是否都配了论据（匿名原话）？分层结论是否配了示意图？有没有停在描述性结果没拔上去？分析底稿与《运行说明》是否附在末尾？

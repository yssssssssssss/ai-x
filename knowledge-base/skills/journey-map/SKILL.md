# Journey Map — 研究材料 + 研究目标/场景 → 用户体验旅程图（编排 wiki 正典）

吃一批研究材料和一个研究目标，按用户达成目标的行为顺序，产出**这一次专用**的用户体验旅程图（通常是一类用户一张图）。本 skill 不发明旅程图方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——用户体验地图方法、定性洞察四框架（旅程图 / 同理心地图 / 价值主张画布 / A·B·C）、场景洞察需求、亲和图聚类、「结果→结论→洞察」模型——按本次材料与研究目标去**读对的正典、抽对的要点、填进旅程图骨架**。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 旅程图的构成（必要 / 灵活内容）、洞察绘制五步、需求评级标准、机会点共创做法、四框架的内涵，唯一出处都在 wiki 正典里。你的工作是按本次 brief **读对正典、抽对要点、填进骨架**，而不是凭记忆复述"旅程图怎么画"。产出的价值在于贴合 *这一批材料* 和 *这一个* 研究目标，不是又一张通用模板图。
- **旅程图是定制品，且"画几张图"由场景差异决定。** 同样画"网购体验"，给生鲜业务和给 3C 业务、给"全面研究型"用户和给"即时应急型"用户，阶段主线、触点、情绪曲线都不同。正典铁律：**场景不同的用户必须分开画；场景相似、行为也相似、仅需求有差异的，可在一张图里单独标出**。务必让材料和研究目标真正决定画几张、每张长什么样——别把不同场景的用户硬塞进一张图，也别把同场景用户拆成多张。
- **三条"命根"，混淆了就把旅程图做成了产品走查**（experience-map 正典「关键技巧」反复强调，落笔前先对齐）：
  - **画的是用户场景，不是产品功能模块**：一个场景可能涉及多个功能模块；要从用户视角真实发生的行为出发，而不是把产品菜单/功能列表翻译成阶段。
  - **场景主线按行为顺序，不是自然时间顺序**：行为可局部线性、前后循环（如装修里"咨询沟通 / 量房 / 出设计"在确定公司阶段会反复经历，不是一次走完）；别硬排成从早到晚的时间轴。
  - **分清"来源于调查"与"来源于洞察"**：阶段 / 行为 / 触点 / 情绪 / 原话多来自用户调研（调查）；诉求 / 机会点 / 深层需求是研究员、设计师、产品提炼的（洞察）。两者不能混为一谈，更不能把洞察包装成"用户亲口说的"。
- **你产出的是高质量初稿，不是已被共识背书的定论。** 旅程图最高的价值在于"共创中达成共识、推动决策"——机会点尤其要产品 / 运营 / 设计多角色工作坊**共创**才全面、可落地；情绪曲线高低、需求优先级也都含研究员主观判断。这几环你这个 agent 复刻不了，要在产出里诚实说明：机会点为单机初稿待团队共创、优先级待业务对齐、材料没覆盖处待补调研——别让人以为这张图已被团队与数据双重背书。
- **分层交付：发现写干净，依据集中到后面。** 使用者要的是一张能开会、能推决策的旅程图，不是方法论实况。正文**先给关键发现摘要（结论先行）与旅程图本身**、写干净能扫读；而 draft 背书、材料不足处的假设、机会点待共创、优先级待对齐这些「依据与边界」**集中到《本次运行说明》**，不逐句夹进每个阶段/单元格里淹没发现。发现层只在真正影响某条结论时留一句轻量标注。

## 敏感素材处理（本 skill 特有，务必遵守）

研究员上传的访谈记录、现场观察、用户原话、客服工单、后台明细属**敏感原始数据**：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户材料内容写进 `research-wiki/` 任何文件，不要落盘到 skill 目录，不要在运行说明里回贴大段原始材料。画完图即止于本次会话。
- **原声引用要匿名、去 PII。** 旅程图强调"尽量保留用户原声"以建立同理心——但引用的原话**必须去除真实姓名、联系方式、住址、单位全称、订单号等 PII**，署名只用代称（用户 A / 受访者 3）。"想法"行的原话、"行为/触点"里点到的人名都适用。有一个常见的"自我拆台"：在运行说明里为了*交代*自己做了匿名化，又把"真名 → 代称"对照写出来——这等于没匿名。只写"已对受访者信息做匿名化处理"，**绝不回写原值**。涉及姓名 / 联系方式等字段时尤其当心——既是隐私要求，也避免触发内容网关拦截。
- **材料只读不改。** 你的职责是基于它还原旅程、提炼洞察，不是修订它。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容（构成清单、绘制五步、需求评级标准、洞察定义、同理心地图四象限等）写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这张图建立在什么之上、哪里可能不牢。结构见 `references/journey-map-skeleton.md`，五个部分一个都不能少。
3. **声明局限，但集中归口、不淹没发现（分层交付）**：基于 draft（未评审）正典的部分、因材料不足而用假设 / 通用方法兜底的阶段或情绪、机会点为单机初稿待共创、优先级待业务对齐——都要交代清楚、别让使用者误以为处处有正典背书、处处有数据支撑，但**集中写到《本次运行说明》**；发现层（关键发现摘要、旅程图各阶段）只在真正影响某条结论时留一句轻量标注（呼应北极星「分层交付」）。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法。**写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 打开那个确切路径、确认它确实报错**——把真实存在的正典误报成缺失，和编造一样有害。本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read。
   - **更要命的是对旅程本身的编造。** 旅程图的每个阶段、每个场景、每条行为、每个触点、每段情绪曲线、每条原话、每个痛点，都必须能在用户给的材料里找到出处。**绝不为了让图更"完整"、更"好看"而虚构一个阶段、编一条原话、瞎画一段情绪曲线、补一个没人提过的触点**——这是旅程图的生命线。情绪曲线尤其危险：没有情绪证据的环节不要硬标高低。材料没覆盖到的阶段，宁可在图上留白标"待调研验证"，也不要编。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/toolbox/collection/experience-map.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与"文件是否存在"的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一张图糊弄。如实告诉用户"未能访问 research-wiki 正典"，请其确认知识库位置后再跑；确需先给草稿时，明确声明"未基于正典、仅通用经验"，且**不得**伪造任何"依据用户体验地图方法第 X 步"的说法。

## 第 1 步：接收并解析输入

先把"拿什么材料、画哪件事的旅程、给谁画"搞清楚，这决定走"绘制"还是"框架兜底"、画几张图、每张画到什么程度：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **研究目标 / 业务场景（画哪件事的旅程，从哪到哪）** | 旅程图的主轴，决定阶段主线、画几张图、要不要标杆对比，**以及阶段的边界与粒度（端到端 vs 聚焦单一环节）** | 完全缺失 → 先问清，这是不能假设的核心 |
| **材料：类型与体量** | 是访谈记录 / 观察 / 已有体验地图素材 / 客服反馈 / 已做好的 persona / 后台数据？几人/几份？——决定走「绘制」还是「框架 + 待补调研」，也决定情绪与触点证据够不够 | 缺 → 按拿到的材料如实判断并声明；**严重不足 → 退化为「地图框架 + 场景主线假设 + 待补调研清单」并指向 `generate-interview-guide`，绝不硬画一张看似完整的图** |
| **目标用户范围 / 类型差异** | 决定画几张图、要不要分型（新/老、不同品类或角色、C/B 端） | 缺 → 合理假设并声明 |
| **约束 / 期望** | 要几张图？要不要情绪量化？细到触点还是到阶段？有没有标杆产品可对比？ | 缺 → 默认：按场景差异自然定张数、情绪用"高低档位 + emoji"呈现曲线、**同时交付 Markdown 文档 + 可视化 HTML 两份**、含标杆仅当材料提供 |

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**；**材料撑不起"绘制"就别硬画**——退化为框架 + 待补调研清单，说清要补什么（指向 `generate-interview-guide`）。同时按上面「敏感素材处理」对待材料。

按下表读 wiki，**只读相关的**。读每篇时带着"我要从这抽什么填进旅程图"的目的，而不是通读。先快速看相关 `index.md`（`methods/toolbox/collection/index.md`、`methods/toolbox/analysis/index.md`、`models/index.md`）确认有哪些条目、再决定打开哪几篇。读完记下每篇相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于产出 |
|---|---|---|
| `methods/toolbox/collection/experience-map.md` | **必读（主骨架，全程对照）** | 它是什么（3 特点）；**何时用 / 不适合**（成熟产品抠细节体验 → 改做可用性测试；研究不明确、只想泛泛收集需求 → 不适合做地图）；**构成**——必要内容（场景 / 行为 / 痛点·需求 / 机会点）+ 灵活内容（用户画像 / 阶段 / 触点 / 情绪 / 诉求 / 标杆），以及"一部分来源于调查、一部分来源于洞察"；**洞察绘制五步**（创建用户角色 → 体验流程梳理（先表格汇总、合并重复、向上总结）→ 需求评级 → 机会点洞察（调研中 + 工作坊共创）→ 可视化）；**"画几张图"判据** 与"场景主线 = 行为顺序"；关键技巧（场景非功能模块、从大到小广度优先、保留原声与提及人数） |

| `methods/toolbox/collection/persona.md` | **需分型时读**（材料未给 persona、又要按类型分图） | 用户角色是抽离同类被访者**共性虚拟**出来的（≠ 单个被访者）；轻量分型取三要素——用户特征 / 产品使用习惯 / 产品目标。**注意分工：本 skill 只做"够分图用的轻量分型"；要血肉丰满、九模块完整的角色集，交给 `generate-persona`**，在运行说明里建议跑它 |

| `methods/toolbox/collection/scenario-based-need-discovery.md` | 要把"诉求 / 深层需求"挖到场景里时读 | 场景化洞察需求的做法 → 提炼每个阶段/场景下用户的**核心诉求**（想取得的核心价值，如"省心"），并据诉求扩大机会点的思考范围 |
| `methods/toolbox/collection/interviews.md` | 材料是访谈记录时可选 | 理解访谈数据要点、从原话里提炼旅程上的行为与情绪信号 |

| `methods/standards/research-report-writing.md` | 组织产出、引用原声增强代入感时可选 | 证据链结构；引用用户原话降低沟通阻力的做法 |
| `methods/scenarios/<业务线>/...` | 可选，按材料所属业务线/阶段 | 取场景化打法做定制范例。**找不到贴合场景 → 用通用方法兜底，并在该处标注** |

## 第 3 步：起草旅程图

骨架与各段写法见 `references/journey-map-skeleton.md`，**起草时打开它**。按以下顺序落地：

- **先定"画几张图"**（旅程图最关键的一步判断）：
  - **归属单位是"场景事件"，不是人。** 分图切的是"用户为达成某目标的一次具体历程"，**同一受访者可能横跨多张图**（如同一人"复购认准款"和"首次买不熟品类"是两条完全不同的路径，应拆两张）。别退化成 persona 式"1 人 = 1 类"。图名用**场景命名**（"复购认准款流程""置办新家全套采购"），不用"研究型 / 价格型"这类人物特征命名——后者会把人往"按人分"带偏。
  - **先定一个互斥的"场景主轴"再切**（避免混轴）：只用**一个**维度切场景（如"熟悉度"），其余维度（价格驱动、渠道…）作为**叠加标签 / 变体**标注，不与主轴场景并列——否则场景会重叠、一个真实行为同时落进两张图（"蹲价买熟悉款"就会既进"熟悉型"又进"价格型"）。把主轴与判断依据写进研判小结。
  - 给了 persona / 用户类型 → 可作分型输入，但仍按**场景**出图（同一 persona 在不同场景下也可拆多图）。
  - 没给且场景明显有差异 → 用**亲和图法按场景主轴轻量分型**，每个场景簇 = 一张图；要血肉丰满的角色，在运行说明里建议跑 `generate-persona`。
  - 场景与行为都相似、仅需求有差异 → **一张图**，在对应格子里单独标出差异，不拆多张。
  - 把判断依据写进研判小结，别闷头画。
- **再定每张图的阶段主线**：从大到小、广度优先——先拆几个**大阶段**（如 需求产生 → 信息收集 → 挑选对比 → 下单付款 → 收货安装），再在阶段下拆**场景**。**阶段的边界与粒度由研究目标决定**：端到端研究按大阶段走；若研究**聚焦单一环节**（如"商详页浏览"），阶段要**收敛成该环节内部的子步骤**（如 初步浏览 → 深度了解 → 信息核实），别硬套"需求 → … → 售后"全链路、把图摊得太泛。阶段/场景按**行为顺序**排（可局部循环），不是自然时间顺序；画的是**用户场景**不是功能模块。先用表格把同类用户的调研信息汇总到对应格子，**合并重复的行为/痛点/需求、向上总结**。
- **逐格填维度**（每个阶段/场景一列，逐行填；行的取舍见下）。**用完整的句子写，长短皆可，别只堆零碎短语**：每条是一句能读通的话，多个要点就分多条（HTML 自动 bullet）。**各行之间要有区分度、别互相雷同**——行为写"用户做了什么（过程）"、触点写"在哪个触点、那里发生或得到了什么"、想法写"心里怎么想"、痛点写"具体卡在哪 + 深层要什么"，同一件事别在四行里换着皮说。
  - **目标**：这一阶段用户想达成什么（一两句话）。
  - **行为（Actions）**：用户**具体做了哪些事**，按过程写得丰富、具体（如"先在小红书/B站搜'怎么选'、看拆机测评和避坑视频""把心仪的 4–5 款加进购物车反复并排比参数""专门去评价区翻差评看有没有坑"），**别只写一句笼统的"对比商品"**。行为是旅程图主干，最该丰富。真实历程若有**循环 / 跨渠道跳转 / 分支**，就写成**带箭头的步骤流**（如"看候选 A → 看候选 B → ↩ 回到 A 比价"、"本站信息不全 → ⇢ 跳外部平台 → 再回来"），别把这种过程压成并列要点。
  - **触点（Touchpoints）**：写清"**哪个触点 + 它在这一步扮演什么角色 / 用户在那里发生或得到什么**"（如"评价区：想找真实差评，但翻到的多是泛泛好评""购物车：被当成并排对比的工作台"），**别只甩触点名**，更**别把行为的动词原样照搬过来**——要换成触点视角。标清移动·数字 / 物理 / 人际。
  - **想法（Thoughts）**：用户的原话或心声（已匿名）+ 同理心地图 Think（没明说但潜藏的顾虑、期望），用第一人称写成完整的话。在匿名前提下**优先选带具体细节 / 数字的原话**（如"尺码助手推荐的码和商家尺码表对不上"胜过"尺码信息不清晰"），别用概括化转述冲淡代入感。
  - **情绪（Emotions）**：各场景情绪连成曲线（emoji + 高低档位），**只对有情绪证据的环节标**，找出低谷（痛点）与高峰（爽点）。
  - **痛点/需求（Pain Points）**：**写具体、别笼统**——把问题描述到细节（反例："评价没参考"太空泛；正例："评价大多是'还不错'这类泛泛之词，找不到能帮判断的真实细节"）。再用 `user-insight` 把表层问题挖到**深层需求**（"表层：…"与"深层：…"各一句）。
  - **诉求（可选，灵活行）**：这一阶段用户的核心价值点（如"省心""确定性"），从行为/痛点提炼。
  - **机会点（Opportunities）**：一句话写清做什么，尽量**落到具体触点**（如"在商详提供'按柜体尺寸反查可选机型'的匹配自检"）；标"单机初稿，待团队共创"。**尽量与痛点 1:1 对齐**——每条痛点对一个具体、可执行的改造（如痛点"同款不同配色未聚合、要退出重搜"↔ 机会点"同款配色聚合到同一详情页、支持一键切换"），别在一格里堆一串泛泛而谈的点。
- **一张图 = 一个"场景事件"下的一类虚拟典型用户，不是受访者拼盘（同一受访者在不同场景可出现在多张图）。** 单元格里写的是**这一类用户在该场景下的共性**，**绝不在格子里散落个体编号**（"A：…；E：…"那样既乱、又违背"旅程图面向一类虚拟用户"的本义——角色是从同类受访者抽离共性虚拟出来的，不等于某个真人）。同型受访者的小差异**综合成一句共性**（如"既看重参数也看重外观"），差异大到场景都不同就**单独成一张图**。**provenance 放图头部 `support` 或脚注**：表头可具名是哪几位受访者（如"支撑：用户 A、用户 E（已匿名）"）或写计数（"综合 2 位"）均可，**关键是别把个体编号撒进单元格**；某需求被几人提及放优先级表（"3/5 提及"）。
- **行的取舍**：必要行（阶段/场景、行为、痛点·需求、机会点）一定有；灵活行（目标、触点、想法、情绪、诉求、标杆）按项目与材料增减——截图那 8 行（阶段/目标/行为/触点/想法/情绪/痛点/机会点）是常见组合，但**以本次材料能支撑的为准，撑不起的行宁可不列也不编**。
- **边界归口（别逐句夹在发现里）**：基于 draft 正典、材料不足而用假设/通用兜底的阶段或情绪、机会点待共创——集中写到《本次运行说明》（呼应纪律 3 的分层交付）；发现层只在真正影响某条结论时留一句轻量标注。

## 第 4 步：需求/机会点评级 + 旅程图自检

- **需求与机会点评级**（用 `experience-map` 的"需求评级"标准，实时读）：从**用户角度**（是否在体验**核心链路**上、是否**无其他方式**满足；被**提及频次**）+ **业务角度**（满足它对产品目标如 GMV / 转化的影响）综合判，给每条痛点/机会点 **高/中/低** 优先级。频次必须真实，**绝不为凑数虚标**（呼应纪律 4）。
- **旅程图自检**（实时读 `experience-map` 的"关键技巧 / 构成"逐条回扫，给 ✅/⚠️/❌，是实质修改不是走过场）：
  - 画的是**用户场景**而非产品功能模块？
  - 场景主线按**行为顺序**（可循环）而非自然时间顺序？
  - **"几张图"判断对**——场景不同的用户分开画了？场景相似的没被拆成多张？
  - 每个格子**可回溯材料**、原声**已匿名**、无虚构的阶段/触点/情绪/原话？
  - **情绪曲线**只标了有证据的环节、找到了低谷/高峰？
  - 痛点挖到了**深层需求**（没停在表象）？
  - 机会点**结合了痛点/诉求**、标了"待共创"？
  - 分清了**"来源于调查" vs "来源于洞察"**？
  - 有 ❌ 回第 3 步改；正典为 draft → 自检照做，但在结果与运行说明声明"基于未评审正典，定稿后需复核"。

## 第 5 步：组装交付（Markdown 文档 + 可视化 HTML 两份）

旅程图是给需求方/产品看的**可视化产物**——纯 markdown 表格可读性差、不便直接拿去用。所以本 skill 交付**两份**：一份可分析、可追溯的 **Markdown 文档**（正文 + 自检 + 运行说明），一份**自包含的可视化 HTML 旅程图**（阶段为列、维度为行、情绪曲线，可直接在浏览器打开、打印为 PDF 分享）。

**先确认、再渲染（交付前关卡）。** 默认**分步交付、单次回复只聚焦一步**，别一口气跑到底直接甩 HTML：

1. **内容确认**：先在对话里用**表格**给出各阶段的 目标/行为/触点/情绪/痛点/机会点（精简——用表格不用长篇 prose、不复述正典原文），**显式征求确认**（"这版内容 OK 吗？要改哪儿？"），用户认可后再渲染。
2. **确认交付目录**：问清 md / html / spec 存到哪，**同一任务固定一处**，**绝不默认写进用户上传的源材料文件夹**；不确定就先问。
3. 确认后再按下面 A / B 落盘。
- **跳过开关（保自主性，别把确认写成死规矩）**：用户明说"直接出图 / 别问了 / 赶时间"，或**无人值守的自动化场景（如评测、批处理）**——**跳过确认直接产出**，并在《本次运行说明》注明"未经交互确认"。

**A. Markdown 文档**，按 `references/journey-map-skeleton.md` 顺序输出（**结论先行**：先给发现，方法论交代随后）：

1. **关键发现摘要（结论先行，放最前）**：一句话主线 + **跨旅程 Top 痛点**（情绪低谷所在）+ **Top 机会点**（高优先级几条）速览，让产品 / 设计 / 运营第一眼就看到"哪里最卡、该先做什么"，不用先读完整张图。3–6 条即可，细节在下方图里展开。
2. **旅程图研判小结**：本次目标一句话；**画几张图及判断依据**；阶段主线；走"绘制"还是"框架 + 待补调研"。（这是方法论交代，紧随摘要之后。）
3. **用户旅程图**（逐张）：每张含 persona/类型头部（角色名 + 一句心智/典型场景）+ **阶段 × 维度表格网格** + 情绪曲线行。多张图并列时像样例的「①②③」。
4. **需求与机会点优先级清单**：把图里的痛点/机会点抽出来排 高/中/低 + 依据。
5. **旅程图自检结果**：对照关键技巧/构成逐条 ✅/⚠️/❌ + 证据。
6. **《本次运行说明》**：五部分，见骨架，逐项填实。

**B. 可视化 HTML**，用本 skill **自带的渲染器**生成（**不要手搓 HTML**——保证样式统一、省 token、每次一致）：

1. 把上面画好的旅程图整理成一份 **JSON spec**（结构见 `references/journey-map-skeleton.md`「可视化 spec」一节：`project_title` / `maps[每张：type_name·mindset·scenario·support·stages·rows[目标/行为/触点/想法/痛点/机会点，key 用 goal/action/touchpoint/thought/pain/opportunity/demand]·emotion{levels 1–5 / emojis / notes} ·footnote]` / `priorities`）。**spec 内容必须与 Markdown 完全一致、同样不编造**：情绪 `levels` 只对有情绪证据的环节给 1–5，无证据填 `null`（渲染器会画成断点 + "无情绪证据"）。
   - **单元格内容尽量丰富、信息完整、分条罗列**：HTML 是给需求方直接看的，一句话容易含糊、产生歧义。当一个格子有多个要点（多种行为、多个触点、几条痛点/机会点、多个想法）时，把 `cells` 的**该元素写成字符串数组**（每条一个要点），渲染器会自动分条 bullet——别把多个点挤进一句长句。单一要点才用普通字符串。痛点/机会点尤其要把"是什么+为什么/深层需求"或"机会点+落到哪个触点"拆成清楚的几条，而不是一句带过。优先级清单的 `need`/`opportunity` 同样可用数组分条。
2. 运行渲染器：`python scripts/render_journey_map.py <spec.json> <out.html>`（用环境里可用的 python；脚本零依赖、纯标准库；spec 与 html 落在交付目录）。
3. 交付 **`journey-map.html`**（连同 `journey-map.md`）。**若本环境确实无法执行脚本**：照常交付 Markdown + 这份 `spec.json`，并在运行说明注明"HTML 未生成，可用 spec + 渲染器后续出图"——**不要假装生成了**，也不要改用手写 HTML 凑数。

# Generate Interview Guide — 研究目标 → 结构化访谈提纲（编排 wiki 正典）

吃一个研究 brief，产出一份**这次专用**的访谈提纲。本 skill 不发明访谈方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的方法正典——提纲设计三层法、ORID 提问框架、受访者偏误规范——把它们组装成贴合本次目标与受访人群的提问清单。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 访谈方法的唯一出处在 wiki 正典里。你的工作是按本次 brief 去**读对的正典、抽对的要点、填进提纲骨架**，而不是凭记忆复述访谈技巧。产出的价值在于「贴合 *这一个* 目标和 *这一类* 用户」，不是又一份通用模板。
- **提纲是定制品。** 同样问「为什么流失」，问新用户和问资深用户、做规划期探索和做迭代期诊断，模块与追问都不同。务必让目标、用户类型、研究阶段真正改变提纲内容。
- **问法即数据质量。** 一道诱导/封闭/双选的问题会污染整场访谈。落笔后必须拿偏误规范回头自检——这是本 skill 区别于「随手列几个问题」的关键一环。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份提纲建立在什么之上、哪里可能不牢。结构见 `references/guide-skeleton.md`，五个部分一个都不能少。
3. **在提纲正文里就地声明局限**：基于 draft（未评审）正典的部分、因未找到贴合场景而用通用方法兜底的部分，都在该处直接标注，别让使用者误以为处处有正典背书。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法或结论。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。**绝不靠目录列举、shell 报错或印象就断定"它不存在"——把真实存在的正典误报成缺失，和编造内容一样有害（会让人误以为知识库缺了这块）。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/toolbox/collection/interview-guide-design.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与「文件是否存在」的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一份提纲糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用经验」。

## 第 1 步：解析 brief

从用户输入里抽出四个要素，决定后面读什么、提纲长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **研究目标** | 提纲的主轴，决定主问题模块 | 完全缺失 → 先问清，这是不能假设的核心 |
| **目标用户类型** | 决定模块取舍与措辞（新/资深、C 端/B 端、已购/流失…） | 缺 → 合理假设并在运行说明声明 |
| **研究阶段** | 规划/设计/迭代，决定探索 vs 评估的侧重 | 缺 → 按目标推断并声明 |
| **访谈形式 / 时长** | 1v1 深访 / 拦访 / 焦点小组；决定补读哪些方法、问题数量 | 缺 → 默认 1v1 深访、约 45–60 分钟，并声明 |

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**，不要因为缺一个时长就卡住不动，也不要在目标都没有时硬编。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着「我要从这抽什么填进提纲」的目的，而不是通读。读完记下每篇的相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于提纲 |
|---|---|---|
| `methods/toolbox/collection/interview-guide-design.md` | **必读** | 框架/内容/问法三层骨架；追问的三个时机与「举例/对比」两种方式；按用户类型（新/资深）定制模块的思路；常见坑 |
| `models/orid.md` | **必读** | 用「客观 O → 感受 R → 理解 I → 决定 D」组织**每个主问题模块内部**的问题递进；各层在访谈中的作用与各层 Tips（敏感问题后置、还原场景拆动机而非直问为什么等） |
| `methods/toolbox/collection/respondent-bias.md` | **必读（落笔后自检）** | 逐条校验问法：剔除诱导/判断/双选式/锚定式提问；开场加「减负说明」；用中性、开放、用户能懂的措辞 |
| `methods/toolbox/collection/interviews.md` | 1v1 深访时读 | 访谈前中后技巧、情绪 ABC（喜怒哀惧）追问要点、B 端深访的划范围/控场/信任建立；用于写追问梯度与给「用法提示」 |
| `methods/toolbox/collection/intercept-interview.md` | 拦访时读 | 街头/现场冷启动的提纲适配（更短、更聚焦、降低拒访） |
| `methods/toolbox/collection/online-focus-group.md` | 焦点小组时读 | 远程群体访谈的组织、主持与防从众，问题改为群体可讨论的形式 |
| `methods/toolbox/collection/participant-recruitment.md` | 涉及甄别/招募时读 | 目标用户定义、甄别问题（若 brief 需要附筛选题） |
| `assets/question-bank/index.md` → `interview/<主题>.md` | **优先读**，先取现成题再现写 | 取贴题的现成访谈题。题库按 7 主题组织（`screener-demographics / behavior-habits / needs-scenarios / mindset-cognition / decision-journey / experience-painpoints / concept-test`）——按本次要问的主题进对应文件取题组，题目里的 `[目标品类]/[目标人群]/[场景A·B]/[本平台]` **占位符按 brief 替换**。某主题文件不存在或无贴题 → 现写，并在运行说明注明「该主题题库暂缺，本次为现写」。**注意**：`interview/` 有货 ≠ 每个主题都全，也 ≠ `survey/` 有货，分别核验 |
| `assets/templates/churn-phone-interview-script.md` | 可选，仅**形态参考** | 看「话术表 / 分流结构」长什么样。⚠️ 这是流失专用样例，**只借形态、不要把内容套死成流失场景** |
| `methods/scenarios/<业务线>/...` | 可选，按 brief 业务线/阶段 | 取场景化打法做定制（如品类消费、产品体验迭代）。**找不到贴合场景 → 用通用方法兜底，并在该处标注「无贴合场景打法，通用兜底」** |

读法补充：先快速看相关 `index.md`（`methods/toolbox/collection/index.md`、`models/index.md`、`methods/scenarios/index.md`）确认有哪些条目、再决定打开哪几篇，避免空读。**按需精读，控制读取量**：index 只看条目行，正典只深读与本次 brief 相关的段落/小节，不必逐字通读全文——必读三篇也只取要用的部分。正典里的 `related:` 指向的文件，**只有用 Glob/Read 核验确不存在时**才记为「走空链接」写进运行说明；能读到、或没核验过的，都不算走空。

## 第 3 步：起草提纲

骨架与各段写法见 `references/guide-skeleton.md`，**起草时打开它**。要点：

- **三段式**：开场/暖场 → 主问题模块（数个）→ 收尾。
- **模块来自研究目标**：把研究目标拆成几个要回答的维度，每个维度一个模块；模块的取舍与措辞按第 1 步的用户类型/阶段定制（参照 interview-guide-design 的新/资深用户拆分思路）。
- **模块内部用 ORID 递进**：每块先问客观事实/场景（O）、再问感受（R）、再挖理解/动机（I）、最后问期待/建议（D）。不必每块都强凑四层，但顺序遵循「先还原、后挖因」。
- **每个主问配追问梯度**：主问下给 2–3 级追问，标注追问时机（核心相关 / 回答不够具象 / 出现新发现）与方式（请举例、做对比、还原最近一次场景）。深访可埋情绪引导问题（挖爽点/痛点）。
- **开场含减负说明**：告知无对错、保密匿名、不会因差评受伤——这直接降低友善偏见与社会期许偏差。
- **就地标注**：兜底/基于 draft 的部分在该模块旁标一句，呼应纪律 3。

## 第 4 步：偏误自检

拿 `respondent-bias.md` 回头扫一遍每道问题，重点查并改：

- 诱导/判断式（「这个功能很方便吧？」→ 改中性开放）；
- 双选式（「好不好用？」→ 拆成「哪些方面顺手、哪些不顺」）；
- 锚定（前置抛出价格/范围/「很多人觉得…」→ 删掉锚点或随机化顺序）；
- 直问「为什么」过多 → 改「还原场景 → 拆动机」；
- 敏感/隐私问题 → 后置、说明保密、或改投射式问法。

自检是实质修改，不是走过场——把改动落到问题文本上。

## 第 5 步：组装交付

按 `references/guide-skeleton.md` 的顺序输出三块：

1. **访谈提纲**（开场/暖场 · 主问题模块[主问 + 追问梯度] · 收尾）
2. **用法提示**（追问的时机与方式、本次特别要防的几个偏误、形式相关注意点）
3. **《本次运行说明》**（五部分，见骨架文件，逐项填实）

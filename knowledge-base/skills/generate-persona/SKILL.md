# Generate Persona — 研究数据/洞察 → 结构化人物角色（编排 wiki 正典）

吃一批研究数据/已有洞察和一个研究目标，产出**这一批专用**的人物角色（俗称用户画像）。本 skill 不发明 persona 方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——人物角色创建法、亲和图聚类、定性洞察四框架、「结果→结论→洞察」模型、G7 与用户养成透镜——先把散落在材料里的发现**聚类成几类典型用户**、选定核心划分变量，再把每类用户**描摹成血肉丰满的角色**。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** persona 的方法、创建步骤、描摹模板的唯一出处都在 wiki 正典里。你的工作是按本次材料与研究目标去**读对的正典、抽对的要点、套进 persona 骨架**，而不是凭记忆复述"用户画像怎么做"。产出的价值在于贴合 *这一批数据* 和 *这一个* 研究目标，不是又一份通用模板。
- **persona 是定制品。** 同样做"网购用户"，给生鲜业务和给 3C 业务、给规划期摸底和给迭代期诊断，划分变量、核心维度、角色数量都不同。务必让研究目标、业务线、数据本身真正决定划分出哪几类角色、每类角色长什么样。
- **三个"不等于"是 persona 的命根，混淆了就做歪了**（正典《人物角色》开篇即辨析，落笔前先对齐）：
  - **人物角色 ≠ 用户细分**：用户细分是按属性把市场"切蛋糕"、各块互斥；persona 关注的是一类用户的**共性需求**，是基于共性拟合的虚拟原型。
  - **人物角色 ≠ 用户画像（user profile）**：后者是给**单个真实用户打标签**的后台数据档案；persona 代表的是**一类用户**，不是某个人的标签合集。（中文里 persona 也被叫"用户画像"，所以要看用户真正要的是"几类典型原型"还是"打标签系统"——后者不归本 skill。）
  - **人物角色 ≠ 真实用户**：角色有名有姓、有血有肉，但它是基于真实用户共性特征**拟合加工**出来的虚拟人，代表一群人在关键指标上的相似性。
- **你产出的是高质量初稿，不是已被背书的定论。** persona 的划分（尤其核心变量的取舍）受研究者主观判断影响大；量化验证、需求方对齐、真人照片授权这几环你这个 agent 复刻不了。所以核心变量是否选对、角色划分能否打动需求方、定量数据能否证明划分，都需要研究员与需求方再对齐、必要时另跑验证——这一点要在产出里诚实说明，别让人以为角色已经过数据与团队双重背书。

## 敏感素材处理（本 skill 特有，务必遵守）

研究员上传的访谈记录、定性笔记、用户原话、后台明细属**敏感原始数据**：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户材料的内容写进 `research-wiki/` 任何文件，不要落盘到 skill 目录，不要在运行说明里回贴大段原始材料。建完角色即止于本次会话。
- **角色的"基础信息""典型语录"是拟合提炼，不是某个真人的档案。** 基础信息（年龄、城市线级、收入等）取这类用户的**代表值 / 区间**（如月收入可取问卷中位数），不是照搬某个受访者；典型语录是这类角色诉求的**提炼性表达**，可基于真实原话提炼，但要去除与角色无关的 PII。**任何角落都不得出现真实姓名、联系方式、住址、单位全称、订单号等 PII**——正文、语录署名、变量清单，以及最容易被忽略的《本次运行说明》。有一个常见的"自我拆台"：在运行说明里为了*交代*自己做了匿名化，又把"真名 → 代称"对照写出来——这等于没匿名。要说明只写"已对受访者信息做匿名化处理"，**绝不回写原值**。涉及姓名/联系方式等字段时尤其当心——既是隐私要求，也避免触发内容网关拦截。
- **角色照片需真人书面授权**（正典明确）——你这个 agent **不伪造、不插入任何未经授权的真人照片**。用一句"照片：[占位 / 文字描述这类用户的形象气质]"代替，并在运行说明注明照片需线下补授权。
- **材料只读不改。** 你的职责是基于它拟合角色，不是修订它。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容（创建步骤、描摹模板、G7 公式、洞察定义等）写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这套角色建立在什么之上、哪里可能不牢。结构见 `references/persona-skeleton.md`，五个部分一个都不能少。
3. **在产出正文里就地声明局限**：基于 draft（未评审）正典的部分、因数据不足而用定性兜底（未做量化验证）的部分、因未找到贴合场景而用通用做法兜底的部分，都在该处直接标注，别让使用者误以为处处有正典背书、处处有数据支撑。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法。**写下任何"未找到 / 走空链接"之前，先用 Read 打开那个确切路径、确认它确实报错**——把真实存在的正典误报成缺失，和编造一样有害。本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read。
   - **更要命的是对数据本身的编造。** persona 的每一类划分、每个特征标签、每条典型语录、每个比例/频次，都必须能在用户给的材料里找到出处。**绝不为了让角色更"丰满"、划分更"成立"而虚构一句语录、编一个行为、抬高一个占比**——这是 persona 的生命线。材料撑不起的细节，宁可留白标"待补充"，也不要编。

- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/toolbox/collection/persona.md`，其所在的 `.../research-wiki/` 即根。

## 第 1 步：接收并解析输入（含选定创建方法）

先把"拿什么数据、按什么目标建角色"搞清楚，这决定走哪条创建路径、怎么聚、描摹到什么程度：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **研究目标 / 业务问题** | 角色的主轴，决定按什么角度划分、描摹哪些维度 | 完全缺失 → 先问清，这是不能假设的核心 |
| **数据：类型与体量** | 是访谈记录 / 定性发现 / 已有洞察，还是带了问卷 / 后台定量数据？几人/几份？——决定走哪条创建路径 | 缺 → 按拿到的材料如实判断并声明 |
| **目标用户范围 / 业务线** | 决定划分边界与描摹口吻（新/资深、C/B 端、哪条业务线） | 缺 → 合理假设并在运行说明声明 |
| **约束 / 期望** | 要几个角色？需不需要量化依据？是否已有划分变量假设？ | 缺 → 默认"先出定性角色、数量随数据自然浮现"，并声明 |

**据数据选定创建方法**（正典《人物角色》三法，运行时读细节）：

- 只有**定性材料** → **定性人物角色**（开展定性研究→细分用户→描摹）。最常见，明确声明"未做量化验证、划分依据偏主观"。
- 定性材料 + **能拿到问卷/后台定量数据来验证划分** → **经定量验证的定性人物角色**（定性三步 + 量化验证）。前提是定性的核心划分指标能在定量侧被清晰定义识别。
- 需求方只认**统计依据**、要用聚类等定量分析定划分 → **定量人物角色**（定性收集变量→定量数据→聚类划分）。注意聚类分析等需专业统计与更多数据，agent 通常只能给出"建议这样设计定量验证/聚类"的方案而非真跑统计——要讲清这一点。

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**；**数据撑不起的创建方法不要硬上**（没有定量数据就别假装做了量化验证）。同时按上面「敏感素材处理」对待材料。

## 第 3 步：收集划分变量 + 聚类浮现用户类型

这是 persona 的"细分用户"环节，用亲和图法的骨架做（详见 `references/persona-skeleton.md`，**起草时打开它**）：

- **拆便签、抽变量**：通读每份材料，把每条独立的发现拆成一条"便签"，按亲和图规则标注来源（哪份/哪位，用代称）与 `D`/`Q`。重点抽 persona 三类划分变量：**目标**（用户用产品真正想达成什么）、**行为**（频次/常用功能/路径/熟练度/生命周期阶段）、**观点**（怎么看产品、看自己、看风险/消费）。其他你认为会影响划分的变量（需求强度、价格敏感、品牌态度等）也纳入变量集。
- **自下而上归并**：把相似便签纵向归到一起，**先别急着命名**；让用户类型自然浮现后，给每个簇写 **level 1** 第一人称标签（"我网购只认准便宜、愿意花时间凑券"而非干巴巴的"价格敏感"）。
- **频次与证据可回溯**：记下每个候选类型"出现在哪几位用户"，挂 1–3 条**代表性原话（已匿名）**当论据。频次必须真实，绝不为凑数虚标（呼应纪律 4）。
- **主动找反面例证**：有没有跨在两类之间、或与主流相悖的个案？别只挑支持既有印象的便签。

## 第 4 步：选核心变量、划分用户 +（如适用）量化验证

把浮现的类型收敛成正式的用户划分（详见 persona.md 第二节实操）：

- **区分核心变量与辅助变量**：逐个变量自问"它真的把用户分出了不同类型，还是只是同类用户的差异展示？"最终**确定 1–2 个核心变量即可，1 个也行**；其余作辅助变量（仅用于展示各类差异）。变量多、难权衡时用**手动归类法**：把重要变量画成轴，把每位用户标到轴上，在 6–8 个变量上聚集成团的就是显著的一类。
- **据核心变量划分**：用 1–2 个核心变量把用户切成几类（如按"商品了解意愿 × 网购时长"切 4 类）。类型数量随数据自然浮现，别硬凑整数。
- **（如适用）量化验证**：
  - 走"经定量验证的定性"或"定量"路径时，按 persona.md 设计验证——核心变量设对应的问卷单选/多选题（每个核心变量 1 道），或用后台可识别指标；验证重点是**核心变量划分出的用户在辅助变量上是否呈现明显且可解释的差异**。
  - 划分涉及**品牌态度** → 可直接用 `g7` 的四道题与 7 群法做量化细分。
  - **没有定量数据** → 就老实走定性路径，**在正文与运行说明里声明"未做量化验证、划分依据偏主观、需后续验证"**，并可给出"建议如何设计验证"。绝不假装验证过。
- **划分自检**（用 persona.md 的质量标准里与划分相关的几条）：这套划分能解释受访用户间的关键差异吗？各类是否足够不同？能覆盖到每个受访用户吗？

## 第 5 步：描摹每个人物角色

对划分出的每一类，按 persona 九模块模板**讲故事式**描摹（骨架见 `references/persona-skeleton.md`），并用其余正典深化：

- **九模块**：角色类型名称（朗朗上口、可记忆）· 照片（占位/文字描述，见敏感素材处理）· 基础信息（代表值）· 特征标签（体现各角色关键差异）· 典型语录（一语道破核心诉求）· 用户故事（家庭/生活/消费概况）· 使用行为（场景/动机/常用功能/行为模式）· 需求及痛点 · 核心建议。
- **用同理心地图立体化**：描摹"使用行为/需求痛点"时，对照 Say-Do-Think-Feel 想清这类用户**说的/做的/想的/感受的**，提炼 Pains/Gains，避免角色扁平。
- **用 user-insight 把需求拔到深层**："需求及痛点"别停在表象痛点（"嫌贵"），按"结果→结论→洞察"追问场景里的深层需求与真实动机（除草机案例的精神）；"核心建议"要可行动，不是复述痛点。
- **（如适用）用 user-cultivation 透镜**：若按生命周期/养成阶段划分，点明每类角色当前卡在养成链路哪一环，核心建议据此给运营方向。
- **就地标注**：基于 draft 正典、或证据偏薄只能谨慎描摹的模块，在该处标一句（呼应纪律 3）。

## 第 6 步：质量自检 + 组装交付

- **质量自检**：实时读 persona.md 的**质量评估 6 条标准**逐条给 ✅/⚠️/❌（能否解释关键差异 / 各类是否够不同 / 能否覆盖每个受访用户 / 能否让人联想到真实用户 / 看完能否快速说出有哪几类 / 能否产生落地价值）——正典说**满足 4–5 条即算成功**，达不到回第 4 步调整划分或第 5 步补描摹。
- **组装交付**，按 `references/persona-skeleton.md` 的顺序输出：
  1. **人物角色集**：逐个角色按九模块描摹（开头一句话点明"本次分出 N 类、用了哪种创建方法、按什么核心变量分"）。
  2. **用户划分依据**：变量集合、核心/辅助变量、（如用）手动归类、（如适用）量化验证结论与待验证项。
  3. **质量自检结果**：6 条标准逐条 ✅/⚠️/❌ + 证据。
  4. **《本次运行说明》**：五部分，见 skeleton，逐项填实。

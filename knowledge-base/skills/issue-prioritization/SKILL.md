# Issue Prioritization — 一堆问题/需求 + (可选)量化证据 → 分层·定级·排序（编排 wiki 正典）

吃一批已识别的问题或需求（可附受影响用户量、业务损失、研发工时），**默认直接当分析师做一次分层·定级·排序**：把散落在各渠道的问题归类、定严重度、按"价值÷成本"排出先后，用统一标准回答"先修/先做哪个"。本 skill 不发明优先级框架，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——问题分层与优先级方法本体（分层 P0–P3 / 严重度定级 / RICE·ICE / 影响×难度矩阵 / 八步流程）、优先级框架选型综述（该用 RICE 还是 MoSCoW / KANO / 四象限 / 执行收益矩阵 / CD3）、四象限优先级决策法、KANO——把它们用到 *这一批* 问题、*这一个* 决策场景上。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** P0–P3 的分层定义、0–4 严重度的档位口径、RICE / ICE 的公式与档位、四象限的四类应对、各框架的适用信号词，唯一出处都在 wiki 正典里。你的工作是**读对的正典、抽对的框架、用到本次这批问题上**，而不是凭记忆默写"RICE 是哪四个字母"或"P0 怎么定义"。产出的价值在于贴合 *这一批* 问题、*这一个* 团队、*这一个* 迭代资源约束的真实排序，不是又一份通用框架讲解。
- **三个递进动作，缺一不可：分层 → 定级 → 排序。** 分层（P0–P3）统一团队语言、便于沟通处置；定级（严重度 0–4 或京东 ABCD）刻画"问题有多糟"；排序（RICE / ICE / 矩阵）叠加成本与触达才决定"先修哪个"。**别把三者混为一谈**——尤其别只做定级就当排序交差。
- **严重度 ≠ 优先级（本方法最大的误用）。** 严重度只说"问题有多糟"，优先级还要叠加 **Effort（成本）** 与 **Reach（触达范围）**。一个 4 分但要 8 人月的问题，很可能要让位于多个 3 分的快赢。凡是把严重度分直接当修复顺序、不看成本与影响面的，都要纠正。
- **两种形态，按"有没有问题清单"自动切换：**
  - **有清单就直接排（默认、主线）。** 用户给了一份问题/需求清单（哪怕只是散在一段话里的几条），就**直接做分层·定级·排序、出优先级表与"先做哪个"结论**，而不是只递一套"工具包"让他自己排。
  - **没清单才产工具包。** 若用户只问"怎么排优先级"或没给任何可排的问题，就产出**可执行的优先级评估工具包**（分层标准卡 + 严重度评分卡 + 框架选型指南 + RICE/ICE 与矩阵模板 + 汇总排序表），并**明确索要问题清单与量化证据**——拿到后即可切到"直接排"。
- **缺哪个框架、用哪个框架，让证据和场景决定。** 缺陷 / 体验问题 / 可用性问题 → 严重度定级 + RICE（有业务数据）或 ICE（早期、小批量）；候选功能 / 产品需求做规划期取舍 → 按信号词从"框架选型综述"里选（MVP 范围→MoSCoW；辨魅力/期望→KANO；快速制胜→执行收益矩阵；时间窗口→CD3；综合平衡→四象限）。**别对所有东西都硬套 RICE。**
- **数据缺失就如实退化，绝不编数字。** Reach（受影响用户量）、业务损失、Effort（工时）这些定量输入常常缺。缺了就**用定性严重度打底 + 用 Confidence 显式打折 + 就地标"数据待补"**，而不是凭空捏一个"8000 人/季""2 人月"让排序显得精确却虚假。定量数据缺失时最易退化为拍脑袋——这正是要防的。
- **边界：给一批已识别的问题/需求排先后 → 本 skill；发现问题 / 从分数找驱动 / 反馈打标 / 归纳洞察 → 隔壁。** 要从一个界面/原型里逐条找出可用性问题（发现问题本身），用 `run-heuristic-evaluation`（它给自己发现的问题打严重度并初排；跨多来源统一排序时再接本 skill）；要从满意度/NPS 分数里找关键驱动、判断先改哪个属性，用 `analyze-satisfaction`；要把海量原始反馈打标归类成主题，用 `code-open-feedback`；要跨多份定性材料归纳主题与洞察，用 `synthesize-qualitative-insights`。本 skill 聚焦「已有一批问题/需求 → 分层定级排序 → 先做哪个」。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把 P0–P3 的定义、0–4 严重度的档位、RICE / ICE 的公式与档位、四象限的口径、各框架的适用场景写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——档位口径可能已从 draft 改为 reviewed、可能已补充本土示例或新框架。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份排序建立在什么之上、哪里可能不牢。结构见 `references/issue-prioritization-skeleton.md`，五个部分一个都不能少。
3. **在正文里就地声明局限**：基于 draft（未评审）正典的部分、严重度靠单人主观判断而未多人校准的、Reach/Effort 靠估计而非实测（已用 Confidence 打折）的、因清单不全而可能漏掉的问题，都在该处直接标注，别让使用者误以为处处有正典背书、处处已确证。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空默写一套"RICE 公式"或"P0–P3 定义"冒充正典。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。** 绝不靠目录列举、shell 报错或印象就断定"它不存在"。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。
   - **更要命的是对量化证据的编造。** 每条问题的 Reach（受影响用户量/占比）、业务损失（转化流失、客诉量）、Effort（工时）**必须来自用户提供的真实数据或明确的估算来源**。**绝不脑补一个精确数字**让 RICE 分看起来客观——没有就标"待业务/研发补"，并在 Confidence 上打折、在运行说明里说明。同理，问题清单里没有的问题不要替用户"补全"想象。

## 第 1 步：解析 brief

从用户输入里抽出要素，决定走"直接排"还是"产工具包"、用哪套框架、读什么、产出长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **问题/需求清单 + 来源** | 决定能不能直接排、排的是什么。来源：可用性测试/启发式评估/访谈/VOC/漏斗/采纳分析/满意度诊断/竞品，或用户直接给的一段清单 | **有任一可排的清单（哪怕散在一段话里的几条）→ 直接排模式**；**完全没有清单 → 工具包模式 + 索要清单**；连"给什么排序"都没有 → 先问清 |
| **量化证据**：各项 Reach（受影响用户量/占比）、业务损失（转化流失/客诉量）、频率 | 决定 RICE 能否精算、严重度能否锚到定量 | 缺 → **用定性严重度打底 + Confidence 打折 + 标"数据待补"**；绝不编数字（纪律 4）。可先做纯定性的严重度分层与矩阵粗筛 |

| **决策场景 + 项目类型** | 决定选哪套框架、落点与深度。场景：评估收尾 / 版本规划 / 线上事故；类型：**缺陷·体验问题** vs **候选功能·需求** | 缺 → 缺陷/体验问题默认走**严重度 + RICE/ICE**；候选功能默认**按信号词从框架选型综述选**；并在运行说明声明假设 |

## 第 3 步 · A：直接排（默认模式 —— 有问题清单时）

你现在就是做优先级排序的分析师。按 `issue-prioritization.md` 的**八步流程**，对用户提供的清单逐条处理（详见 `references/issue-prioritization-skeleton.md`，**起草时打开它**）：

1. **汇总去重**：把各渠道的问题归并到一张清单，**合并"同因异象"的重复项**（清单杂而多时用亲和图聚类），每条写清"现象—影响的任务—证据来源"。
2. **分层打标**：按 P0–P3 给每条贴层级标签，统一团队语言（P0 阻断性 / P1 功能性 / P2 易用性 / P3 愉悦·外观性——以正典定义为准）。
3. **严重度定级**：按 NN/g 0–4（频率×影响×持续性）给每条打分并**给理由**，凭专家判断即可打、无需数据；若有影响范围数据（UV占比/点击/曝光/反馈量）可并用京东 ABCD。关键问题**建议 3–5 人独立打分取均值**降低主观偏差；若本次只能单人打，**就地标"未多人校准"**。
4. **补定量证据**：为每条补 Reach（受影响用户数/占比）与业务损失，把定性严重度**锚到定量影响**上。**没有数据的，标"待补"并在 Confidence 打折，绝不编数字**（纪律 4）。
5. **估算成本**：估 Effort（人月）或难度档（S/M/L）。无研发输入就标"待估"，别硬编精确工时。
6. **跑优先级模型**：缺陷/体验问题 → 算 **RICE**（有业务数据）或 **ICE**（早期/小批量）；候选功能/需求 → **按信号词从框架选型综述选框架**（四象限/MoSCoW/KANO/执行收益矩阵/CD3）。问题量大时，**先用价值×成本矩阵粗筛，再对高价值项做 RICE 精排**。列出每条的算分过程，别只给最终分。
7. **画矩阵与定线**：在**影响×难度矩阵**上给每条落点（快赢/大投入/顺手补/时间黑洞），**圈出"高影响×低难度"的快赢（Quick Wins）**优先排期。
8. **评审定档**：按**依赖关系**和战略**"必备项"**做人工微调（RICE 分不是铁律，别漏排前置条件），输出最终修复/开发顺序。
9. **就地标注**：基于 draft 正典、严重度未多人校准、Reach/Effort 靠估计（已用 Confidence 打折）、清单可能不全，都在该处标一句（呼应纪律 3）。

## 第 3 步 · B：产工具包（无问题清单时）

产出一份可直接执行的优先级评估工具包（结构见 skeleton），并**末尾明确索要问题清单与量化证据**（"把要排的问题/需求清单发我，最好带上各项的受影响用户量、业务损失、研发工时估算，我可以直接排一遍"）：

- **分层标准卡**：P0–P3 各层定义与判据（取自 `issue-prioritization.md`，别默写）；
- **严重度评分卡**：NN/g 0–4（档位 + 频率/影响/持续性三因素）与京东《体验问题评级标准》（广度×深度→ABCD）两套，按是否有影响范围数据择用；
- **框架选型指南**：信号词→框架对照（取自 `priority-frameworks-overview.md`）+ 该场景推荐用哪个（缺陷→RICE/严重度，功能规划→四象限/MoSCoW/KANO）；
- **RICE/ICE 评分模板 + 影响×难度矩阵模板**：字段与公式取自正典；
- **汇总排序表模板 + 评审清单**：问题记录字段（现象/影响任务/证据/分层/严重度/Reach/Impact/Confidence/Effort/得分/象限）+ 评审时叠加依赖与必备项。

## 第 4 步：排序自检（拿"局限与误用"回头扫一遍）

**实时读 `issue-prioritization.md` 的"局限与误用"**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **只数频次？** 有没有把"出现次数最多"当成"最该修"，埋没了**低频但高影响的阻断问题**（低频高影响往往是 P0）？
- **只凭主观？** 严重度是否有多人校准或定量证据支撑？单人拍脑袋打的是否**已标注"未多人校准"**？
- **严重度=优先级？** 是否**叠加了 Effort 与 Reach**，没有把严重度分直接当修复顺序？高分高成本项是否让位给了多个快赢？
- **公式当铁律？** 是否按依赖关系与战略"必备项"做了**人工微调**，没有照搬 RICE 排名漏排前置条件？
- **Confidence 缺失？** 数据薄弱的估计是否**打了折**，没让"听起来诱人但说不清"的点虚高占位？
- **分层标准飘移？** P0/P1 的定义是否统一、同一标准用到底？
- **编数字？** 每条的 Reach/业务损失/Effort 是否有真实来源，没有的是否标了"待补"而非硬编（纪律 4）？

把改动落到排序表上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/issue-prioritization-skeleton.md` 的顺序输出：

- **直接排模式**：① **优先级结论摘要（结论先行，放最前）**——一句话总判断（这批问题里最该先动的是什么、为什么）+ **Top 3–5 "先修/先做"速览**（每条：问题 + 分层/得分 + 一句为什么最先），让人第一眼看到"最该先做的是什么"，细节在下方展开 ② 优先级排序表（每条含 现象 · 分层标签(P0–P3) · 严重度(0–4，含理由) · Reach/Impact/Confidence/Effort · RICE/ICE 得分 · 影响×难度象限）+ **影响×难度矩阵**（圈出快赢）+ 分层小结 ③ 排序自检结果 ④《本次运行说明》（五部分）

# Jobs To Be Done — 需求（功能设想 / 表层诉求）→ 用户真正的 Job（编排 wiki 正典）

吃一条待定义的需求（外加可选研究材料），产出**这一次专用**的 JTBD 需求定义——把「加个什么功能 / 用户想要什么」向上翻译成「用户到底雇产品完成什么任务」。本 skill 不发明 JTBD，而是**实时编排**用研知识库（`research-wiki/`）里既有的正典——JTBD 雇佣理论、用户需求三层模型、梯子理论（手段-目的链）、Job Map 八阶段、四力模型、Job / Switch 访谈法、结果→结论→洞察、竞品分析与机会点研究——按本次需求与材料去**读对的正典、抽对的要点、填进 Job 定义骨架**。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 雇佣理论、三层需求、爬梯话术、Job Map 八阶段、四力、期待效果三铁则、跨品类竞争、机会点挖法——唯一出处都在 wiki 正典里。你的工作是按本次需求**读对正典、抽对要点、填进骨架**，而不是凭记忆复述"JTBD 怎么做"。产出的价值在于贴合 *这一条需求* 和 *这一批材料*，不是又一份通用 JTBD 科普。
- **本 skill 的核心动作是"向上翻译"。** 别人给你的十有八九是**解决方案**（"加一个一键复购按钮"）或**表层诉求**（"用户说想要更快"）——你要做的不是照单接下、也不是评审这个方案好不好，而是**顺着它往上爬一层**：用户拿这个功能 / 这句诉求，真正想完成的是什么任务？这个任务在什么情境下发生？完成得好不好，用户自己怎么衡量？爬到"一个动词 + 一个对象、且不含任何产品名"的 Job，才算落到位。功能设想只是"当前设想的一种解法"，不是 Job 本身。
- **Job 定义是定制品。** 同样一句"想要个收藏夹"，在生鲜复购场景和在 3C 决策场景下，用户雇它完成的 Job 完全不同。务必让**具体需求、情境、材料**真正决定 Job 长什么样，别套用万能模板。
- **五条"命根"，混淆了就把 JTBD 做成了功能清单**（`models/jtbd.md`、`models/user-needs.md` 反复强调，落笔前先对齐）：
  - **Job ≠ 功能 / 解决方案**：Job 是"用户要完成什么"，不是"产品提供什么"。"一键复购"是解法，"在想复购时以最小心力买回上次那批常买的东西"才是 Job。把功能名当 Job 是头号错误。
  - **Job 看情境，不看人群属性**：同一个人在不同情境会雇完全不同的产品。Job 定义要锚"什么时候 / 在什么处境下想完成什么"，而不是"25–35 岁女性白领"。人群是 persona 的事（→ generate-persona）。
  - **Job 有功能层，也有情感层 / 社会层**：只挖功能层（"快速买回"）会漏掉真需求——金佰利成人纸尿裤的 Job 不是"处理失禁"，而是"体面地、不引人注意地维持日常的正常感与自尊"。情感 / 社会层常常才是胜负手。
  - **Job 的高度要适中**：太宽（"想要美好生活"）无法行动，太窄（"点一下按钮"）退化成功能。落在"用户会为之雇佣、且能指导设计"的那一层。
  - **痛点 ≠ 需求**：痛点只说明"现状有多糟"，不告诉你"怎么做更好"（`user-insight` 除草机案例）。要顺着痛点挖到它背后场景里的**深层需求与真实动机**，别停在"嫌贵 / 太麻烦"。
- **你产出的是高质量初稿，不是被数据背书的定论。** 没有用户材料时，你给出的 Job 是**基于常识与需求描述的假设**——它有价值（能校正方向、指导下一步该问什么），但**没被真实用户验证过**；核心 Job 选得对不对、情感层是否属实、跨品类替代是否真存在，都需要用 Job / Switch 访谈去证。有材料时，也只在材料覆盖到的地方下结论。这条底线要在产出里说清楚，别让人以为 Job 已经过用户验证。

## 敏感素材处理（本 skill 特有，务必遵守）

用户可能会连同需求一起上传访谈记录、流失原声、客服反馈、后台或竞品明细，属**敏感原始数据**：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户材料内容写进 `research-wiki/` 任何文件，不要落盘到 skill 目录，不要在运行说明里回贴大段原始材料。定义完 Job 即止于本次会话。
- **原声引用要匿名、去 PII。** 引用用户原话来支撑 Job / 四力时，**必须去除真实姓名、联系方式、住址、单位全称、订单号等 PII**，署名只用代称（用户 A / 受访者 3）。有一个常见的"自我拆台"：在运行说明里为了*交代*自己做了匿名化，又把"真名 → 代称"对照写出来——这等于没匿名。只写"已对受访者信息做匿名化处理"，**绝不回写原值**。涉及姓名 / 联系方式等字段时尤其当心——既是隐私要求，也避免触发内容网关拦截。
- **材料只读不改。** 你的职责是基于它挖 Job，不是修订它。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把正典内容（雇佣理论、三层需求、爬梯话术、Job Map 八阶段、四力定义、期待效果三铁则等）写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份 Job 定义建立在什么之上、哪里可能不牢。结构见 `references/jtbd-skeleton.md`，五个部分一个都不能少。
3. **在产出正文里就地声明局限**：基于 draft（未评审）正典的部分、无材料而给出的 Job 假设、Job Map 八阶段等本身带 `🤖 草稿待审` 标记的正典内容、因材料不足只能推断的情感 / 社会层——都在该处直接标注，别让使用者误以为处处有正典背书、处处有用户验证。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的方法。**写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 打开那个确切路径、确认它确实报错**——把真实存在的正典误报成缺失，和编造一样有害。本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read。
   - **更要命的是对 Job 与证据的编造。** 每一条 Job、每一层（功能 / 情感 / 社会）、每一个跨品类替代方案、每一股力、每一条期待效果，**有材料就必须能回溯到材料里的出处，没材料就必须标成"假设 / 待验证"**。绝不为了让 Job 定义更"完整""漂亮"而虚构一句用户原话、编一个没人提过的替代方案、把常识包装成"用户亲口说的"。JTBD 最容易被滥用成"研究员自说自话的一套故事"——不编造、标清楚哪些是证据哪些是假设，是本 skill 的生命线。

## 产出写法：正文只给结果分析，理论与方法交代下沉（务必遵守）

使用者是产品经理 / 需求方，他们要的是"用户到底要完成什么、这需求的本质、该怎么办"——一份**能直接拿去开会的结论**，不是一堂 JTBD 课，也不是方法论演示。产出的可读性直接决定它有没有用。落笔守三条：

- **正文只说结果与判断，不解释理论、不复述方法、不把推理过程摆出来。** 直接给 Job、给分层、给竞争格局、给结论。**不要**在正文里科普"什么是梯子理论 / 四力 / Job Map"；**不要**写"（用 `models/user-needs` 三层模型）""（按 XX 正典这属于误区）"这类内联引用与理论旁白；**不要**把"表层→中层→深层"的爬梯当作一步步的推理秀逐级展开。你读了正典、按它分析，这个事实会体现在结论质量里，无需在正文自证。
- **方法论、调用了哪些正典、status、草稿 / 缺源等出处交代，一律只放《本次运行说明》。** 正文里不重复"这来自 draft 正典""八阶段是 ODI 通识草稿"这类话——真要在正文提醒的边界，用一句话或一个短标签，绝不成段解释。
- **假设 / 待验证只用简短标签，不展开论证。** 无材料或推断出的内容，标一个「（假设，待验证）」、或在小结里集中说明即可，**不要**每条都写一段"基于……常识推断、因为……所以……"的理由。诚实标注是硬要求（纪律 3/4 不变），但**标注 ≠ 长篇解释**。

一句话：**把"我是怎么想出来的、依据哪条理论"从正文拿掉，只留"用户要完成什么、所以怎么判断"；想追溯方法论的人自会去看《本次运行说明》。** 能用一张表讲清的就别写成大段文字，能用一个词标注的就别写成一句解释。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/models/jtbd.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与"文件是否存在"的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一套 Job 糊弄。如实告诉用户"未能访问 research-wiki 正典"，请其确认知识库位置后再跑；确需先给草稿时，明确声明"未基于正典、仅通用 JTBD 经验"，且**不得**伪造任何"依据雇佣理论第 X 条"的说法。

## 第 1 步：接收并解析输入

先把"要定义哪条需求、有没有材料、给谁在什么情境下用"搞清楚，这决定走"基于证据的 Job 定义"还是"Job 假设 + 待验证"、侧重哪套正典：

| 要素 | 作用 | 缺失怎么办 |
| --- | --- | --- |
| **待定义的需求（功能设想 / 一句业务问题 / 一个诉求）** | Job 定义的起点，也是最后要回应的对象——你要把它向上翻译成 Job | 完全缺失 → 先问清"具体要挖哪条需求 / 哪个功能 / 哪个诉求"，这是不能假设的核心；别自己编一个需求来分析 |
| **有没有研究材料，什么类型** | 有 Switch / Job 访谈、流失原声、用户反馈、后台 / 竞品数据？还是只有一句需求？——决定走「基于证据」还是「Job 假设 + 待采集」，也决定情感 / 社会层与四力能不能坐实 | 缺 → 走**Job 假设**路径：给出基于常识与需求描述的 Job，但通篇标"假设、待验证"，末尾列"要验证得采什么"，指向 `generate-interview-guide`（跑 Job / Switch Interview）。**绝不把假设当已验证的 Job** |
| **情境 / 业务线 / 目标用户范围** | Job 的情境边界（JTBD 看情境不看人群）——同一需求在不同情境是不同 Job | 缺 → 合理假设一个最可能的情境并**在正文声明**；若情境分叉很大（如复购既可能是"囤货省心"也可能是"应急补货"），分别列出候选 Job，别硬并成一个 |
| **研究目的** | 决定侧重哪套：**产品定义 / 创新机会** → Job Map 八阶段 + 期待效果；**流失 / 切换 / 采纳诊断** → 四力 + Switch 时间线；**竞品重划 / 找替代** → 跨品类竞争格局 | 缺 → 默认走"产品定义视角"（核心 Job + 功能/情感/社会层 + 简版 Job Map），并声明；若需求本身是流失 / 切换语气，默认加四力 |

判断原则：**核心（哪条需求）缺了就问，次要参数缺了就合理假设 + 声明**；**材料撑不起的层次和结论不要硬上**（没有材料就别假装情感层是用户验证过的）。同时按上面「敏感素材处理」对待材料。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着"我要从这抽什么用于定义 Job"的目的，而不是通读。先快速看相关 `index.md`（`models/index.md`、`methods/toolbox/collection/index.md`、`methods/toolbox/analysis/index.md`）确认有哪些条目、再决定打开哪几篇。读完记下每篇相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于产出 |
| --- | --- | --- |
| `models/jtbd.md` | **必读（主骨架，全程对照）** | 雇佣理论（用户买的不是产品，是雇它完成 Job）；**关注情境而非人群**、**跨品类竞争是常态**；Jobs-As-Activities（工作即活动）/ Jobs-As-Progress（工作即进步）两种解释维度；**功能层（Functional）+ 情感层（Emotional）**；需求格式 **"I want [X] in a [Y] way"**；主任务 vs 相关任务；作为透镜的问题定义 / 竞品 / 创新机会用法；金佰利、奶昔、方便面 vs 外卖等案例（可读部分）。注意其中多处 `⚠️ 待补充(缺源)`，缺的别自己补 |
| `models/user-needs.md` | **必读（"挖到用户雇产品完成什么任务"的骨干）** | **需求三层模型**：表层=显性诉求（用户直接说的功能 / 痛点）、中层=任务场景、深层=目标动机；**三条原则**——回归用户场景、Job To Be Done、痛点≠需求；三误区（只听态度不看行为 / 把用户建议当需求 / 把个性需求当普遍需求）。这是把"用户想要 X（表层）"往下挖到"Job（中层任务）+ 动机（深层）"的主尺子 |
| `models/user-insight.md` | **必读（避免停在表层痛点）** | **结果→结论→洞察三层**；"搜集痛点 ≠ 洞察、痛点常是表象、真需求在场景深处"（**除草机案例**：缺的不是除草机，是"草坪整洁又不用自己动手"）——用来把功能诉求 / 痛点挖到深层需求，是本 skill "向上翻译"的推理引擎 |

## 第 3 步：把需求"向上翻译"成 Job（本 skill 的核心动作）

骨架与各段写法见 `references/jtbd-skeleton.md`，**起草时打开它**。**注意：下面是你脑子里的思考步骤；正文产出只呈现结论与一份精简的"原始需求 → Job"对照，不把这套爬梯推理逐级搬进正文（见「产出写法」）。** 按以下顺序想清楚：

- **先记下原始需求的"形态"**：它是**解决方案层**（"加个一键复购按钮"）、**表层诉求**（"用户想要更快"）、还是**已经接近 Job**（"用户想在忙碌工作日快速搞定一顿正餐"）？形态决定要往上爬几层。
- **顺着 user-needs 三层 + 梯子理论往上爬**（这一步是灵魂）：对原始需求反复追问"用户拿它想完成什么？→ 这对用户意味着什么？→ 为什么这一点重要？"，把**表层显性诉求**还原到**中层任务场景**，再挖到**深层目标动机**。有材料就从材料里的行为 / 原话找证据爬；没材料就基于常识爬，但**每一级都标"假设"**。爬到"一个动词 + 一个对象、不含任何产品名"的 Job 为止。
- **区分核心 Job 与相关 Job**：核心 Job 是用户真正雇产品去完成的主任务；相关 Job 是完成主任务过程中连带要做的小任务（`jtbd.md` 的"主任务 / 相关任务"）。别把一堆相关 Job 平铺成并列的核心 Job。
- **锚定触发情境**（用 5W2H）：Job 不挂在人群上，挂在情境上——"**当 [什么处境 / 什么时候]，用户想 [完成什么]，以便 [达成什么结果]**"。情境写具体（"临近饭点、又累又不想做饭时"），别写成人群标签。
- **就地标注**：哪些层是材料支撑的、哪些是常识假设的，在该处标一句（呼应纪律 3、4）。

## 第 4 步：把 Job 定义完整（功能/情感/社会层 · Job Map · 竞争 · 四力 · 期待效果）

按研究目的取用下面的模块（不是每个都要，够用即可，详见骨架）：

- **写 Job Statement + 三层维度**（必做）：核心 Job 一句话；再拆**功能层**（要客观完成的操作 / 结果）、**情感层**（内在感受诉求，如安心 / 掌控感 / 不被打扰）、**社会层**（在他人眼中想成为 / 被如何看待，如适用）。可用 `jtbd.md` 的需求格式 "I want [X] in a [Y] way" 收束。**别只写功能层**——情感 / 社会层缺失是 JTBD 最常见的漏。
- **（产品定义 / 创新）用 Job Map 拆任务全流程**：按 `job-map.md` 的八阶段脚手架把核心 Job 拆成阶段，逐阶段填"用户怎么做、哪一步耗时/麻烦/易出错、什么算做好"，把痛点按阶段填回，高频痛点阶段标为机会点。**务必标注"八阶段为待核验草稿脚手架"**，且拆的是用户任务流不是产品功能流。
- **盘竞争格局（完成同一 Job 的替代方案）**：列出用户当前"雇佣"什么来完成这个 Job——**含跨品类、含'什么都不做/手动凑合'**（方便面 vs 外卖 vs 自己做 vs 不吃）。对每个替代方案，简评它在功能层 / 情感层的完成度。**比的是 Job 完成度，不是功能多少。** 要系统竞对分析→在运行说明建议跑 `competitive-analysis`。
- **（流失 / 切换 / 采纳）做四力诊断**：用 `four-forces.md` 把材料（或假设）里的**推力 / 拉力 / 焦虑 / 惯性**各列几条，套决策不等式判断"这次切换 / 采纳为什么发生或没发生、哪股力是关键"。有 Switch 材料就按时间线提取，没有就标假设。
- **提炼未被满足的期待效果 / 机会点**：用 `job-interview.md` 的**"方向+测量单位+对象+情境"**格式把关键期待效果写成可衡量语句（如"在赶时间的晚餐场景，最小化从'决定吃饭'到'吃上饭'的时间"）；用 `opportunity-research.md` 把高价值未满足处转成机会点，并**标"待验证"**（防幸存者偏差 / 填空白≠赢市场）。
- **就地标注**：draft 正典、`🤖 草稿待审` 脚手架、无材料的假设、只能推断的层，都在该处标一句。

## 第 5 步：回应原始需求 + 自检 + 组装交付

- **回到原始需求给结论**（本 skill 的落点，别漏）：那条功能设想 / 表层诉求——**命中了核心 Job 吗？命中了哪一层（功能 / 情感 / 社会）？是完成这个 Job 的最优解，还是众多替代方案之一？** 给出"该做 / 缓做 / 换个解法 / 先验证"的倾向性判断 + 一句理由，并列出"要坐实这个判断，下一步该采什么、问什么"。这一步把抽象的 Job 定义翻译回需求方能用的决策输入。
- **JTBD 命根自检**（对照"北极星与边界"的五条命根逐条给 ✅/⚠️/❌，是实质回扫不是走过场）：
  - Job 写成了"用户要完成什么"而非"产品提供什么"（没把功能名当 Job）？
  - Job 锚在情境上而非人群属性上？
  - 覆盖了功能层 + 情感层（+ 社会层，如适用）、没只停在功能层？
  - Job 高度适中（不太宽到无法行动、不太窄到退化成功能）？
  - 痛点挖到了深层需求（没停在"嫌贵 / 太麻烦"）？
  - （如列期待效果）符合三铁则（不含产品 / 可测量 / 稳定）？
  - 有材料的结论可回溯、无材料处标了"假设待验证"、无虚构的原话 / 替代方案？
  - 有 ❌ 回第 3 / 4 步改；正典为 draft / 草稿 → 自检照做，但在结果与运行说明声明"基于未评审正典，定稿后需复核"。
- **组装交付**（结论先行；且全程遵循前述「产出写法」——正文只给结果分析，理论、方法与正典交代一律下沉到《本次运行说明》），按 `references/jtbd-skeleton.md` 顺序输出：
  1. **一句话结论（放最前）**：核心 Job 一句话 + "原始需求 → Job"的翻译一句话，让需求方第一眼看到"用户到底要完成什么、这需求的本质是什么"。
  2. **需求解析（精简对照）**：一个紧凑的「原始需求（标形态）→ 核心 Job」对照，最多一两句点出关键转折（"要的不是 X，而是 Y"）。**不要把表层→中层→深层逐级展开成推理秀，不要加理论旁白**——爬梯是你脑子里做的，正文只留落点。
  3. **Job 定义**：核心 Job 与相关 Job / 功能·情感·社会层 / 触发情境 / （如适用）Job Map 八阶段表。
  4. **竞争格局**：完成同一 Job 的替代方案（含跨品类）及各自 Job 完成度。
  5. **（如适用）切换 / 采纳四力诊断**。
  6. **未被满足的期待效果 / 机会点**（+ 优先级与"待验证"标注）。
  7. **对原始需求的回应**：命中哪层 / 是否最优解 / 该做缓做换解法 / 下一步验证清单。
  8. **JTBD 命根自检结果**：逐条 ✅/⚠️/❌ + 证据。
  9. **《本次运行说明》**：五部分，见骨架，逐项填实。

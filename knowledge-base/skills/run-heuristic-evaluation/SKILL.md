# Run Heuristic Evaluation — 评估对象 + 范围 → 启发式评估（编排 wiki 正典）

吃一个评估对象与评估范围，**默认直接当专家评估员做一次启发式评估 / 体验走查**：拿用研知识库（`research-wiki/`）里的尼尔森十大启发式（评分标尺）逐条检视界面，找出违反点、定严重度、给改进建议。本 skill 不发明评估准则，而是**实时编排**既有正典——十大启发式（评什么）、启发式评估方法本体（怎么评：两遍法 / 走查节奏 / 评估员数 / 汇总）、严重度评级与问题优先级技术（issue-prioritization 唯一出处）——把它们用到 *这一个* 对象、*这一个* 范围上。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 十大启发式的条目内涵、严重度评级的档位口径、3–5 名评估员的依据，唯一出处都在 wiki 正典里。你的工作是**读对的正典、抽对的标尺、用到本次对象上**，而不是凭记忆默写"尼尔森十条是哪十条"。产出的价值在于贴合 *这一个* 界面、*这一类* 用户、*这一个* 任务的真实问题，不是又一份通用原则清单。
- **两种形态，按"有没有界面素材"自动切换：**
  - **有素材就直接评（默认、主线）。** 启发式评估是**专家检视法、不需要真实用户**，所以你（LLM）可以亲自当评估员。用户给了截图 / 原型链接 / 可访问页面 / 流程或文字描述，就**直接做评估、出问题清单**，而不是只递给他一套"工具包"让他自己评。
  - **没素材才产工具包。** 若用户只问"怎么做启发式评估"或没给任何可评的界面，就产出**可执行的评估工具包**（检查清单 + 严重度评分卡 + 评估员任务书 + 两遍法脚本 + 记录/汇总模板），并**明确索要界面素材**——拿到后即可切到"直接评"。
- **违反一条 ≠ 一定是问题。** 这是启发式评估最大的误用。每条疑似问题都要**结合场景与替代方案判断**，并落到**严重度评级**（口径取自 `issue-prioritization.md`）——而不是把"提到某启发式"当成"违反"、堆一长串伪问题。低严重度也要如实标低，别为凑数注水。
- **替代不了真人验证，要说清楚。** 你做的是"专家视角的疑似问题扫描"，覆盖面与权威性都有限。凡是需要真实用户行为/数据才能定论的，**标"需真人验证"**，并在运行说明里建议接 `usability-testing` 做正式测试——绝不把启发式评估的结论包装成"已验证可用性"。
- **边界：拿准则检视界面 → 本 skill；真人做任务 / 大规模问卷 / 选方法 / 反馈打标 → 隔壁。** 要让真实用户做任务并观察其操作，用 `generate-usability-test`；要可统计问卷，用 `generate-survey`；需求在"整体该用什么方法"，用 `generate-research-plan`；要把海量用户原话归类打标，用 `code-open-feedback`。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把十大启发式的条目文字、严重度评级的档位定义、评估员人数、走查节奏写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——准则可能已从 draft 改为 reviewed、内涵可能已补充本土示例。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份评估建立在什么之上、哪里可能不牢。结构见 `references/heuristic-evaluation-skeleton.md`，五个部分一个都不能少。
3. **在正文里就地声明局限**：基于 draft（未评审）正典的部分、因素材不足而无法走查到的流程（如只给单张截图却要评跨页链路）、靠推断而非界面证据得出的判断（标"需真人验证"），都在该处直接标注，别让使用者误以为处处有正典背书、处处已确证。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空默写一套"尼尔森十条"或"0–4 档位"冒充正典。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。** 绝不靠目录列举、shell 报错或印象就断定"它不存在"。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。同理，评估发现必须基于用户提供的真实界面素材，**不得脑补界面长什么样**——素材不足就说不足、索要素材，不要凭想象编造问题。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/models/nielsen-heuristics.md` 或 `**/methods/toolbox/collection/heuristic-evaluation.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与「文件是否存在」的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一套启发式糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用经验」，且**不得**伪造任何「依据本库启发式正典第 X 条 / 标准 0–4 评级」的说法。

## 第 1 步：解析 brief

从用户输入里抽出要素，决定走"直接评"还是"产工具包"、读什么、产出长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **评估对象 + 素材形态** | 决定能不能直接评。形态：截图/录屏、可点原型链接、可访问线上页面、设计稿、或足够具体的流程/文字描述 | **有任一可评素材 → 直接评模式**；**完全没有可评素材 → 工具包模式 + 索要素材**；连"评什么对象"都没有 → 先问清 |
| **评估范围 / 目标** | 决定走查哪几条核心流程、聚焦哪个端/板块/任务/用户态 | 缺 → 默认评"对象最核心、最高频的主流程"，并在运行说明声明范围假设 |
| **启发式集** | 默认**尼尔森十大启发式**（读 `models/nielsen-heuristics.md`）；用户若指定别的框架，按其要求并声明 | 缺 → 默认 Nielsen 10 |
| **是否竞品对比 / 要不要量化打分** | 决定要不要叠加分档评分 / GSB（读 `experience-evaluation-rating.md`） | 缺 → 默认只做单对象问题清单 |

判断原则：**对象缺了就问；范围/目标等次要参数缺了就合理假设 + 声明**——不要因为缺一个"评哪个端"就卡住，也不要在没有任何可评素材时硬编一份"看起来像评估"的问题清单（那违反纪律 4）。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着「我要从这抽什么用到本次评估」的目的。读完记下每篇相对路径与 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于评估 |
|---|---|---|
| `models/nielsen-heuristics.md` | **必读（标尺主线）** | 十大启发式**每条的准确内涵 + 电商示例**——作为逐条检视的评分准则；"违反≠问题"的判断边界与"局限与误用" |
| `methods/toolbox/collection/heuristic-evaluation.md` | **必读（流程主线）** | **两遍法**（先熟悉、再逐条比对）；**走查节奏**（整体浏览 → 模拟使用[含逻辑边界] → 查漏补缺）；**严重度评级的施测时机与汇总**（评估后独立打分取均值、亲和图聚类；**档位口径见下行 issue-prioritization**）；三因素（频率×影响×持续性）；评估员人数；前置准备（范围收窄、机型覆盖）；常见坑 |
| `methods/standards/sampling.md` | 工具包模式定**评估员人数**、或要解释"为何 3–5 人"时读 | "少数评估员即可发现多数问题"的逻辑与人数依据；**别把经验人数当硬指标，也别凭空编一个冒充本库规范** |
| `methods/toolbox/analysis/issue-prioritization.md` | **必读（严重度评级 + 优先级的唯一出处）** | **严重度评级两套**：NN/g 0–4（频率×影响×持续性，凭专家判断即可打、无需数据）与京东《体验问题评级标准》（影响范围×严重程度→等级，需 UV占比/点击/曝光/反馈量数据）——无数据用 0–4、有则可并用京东标准；**分层 P0–P3**；**排序** RICE/ICE 与影响×难度矩阵。**别在 SKILL 默写档位，运行时现读** |
| `methods/toolbox/collection/usability-testing.md` | 需要说明**与真人测试的边界 / 建议后续验证**时读 | 启发式评估是其**补充而非替代**；哪些疑似问题应转入正式可用性测试验证 |
| `methods/toolbox/collection/experience-evaluation-rating.md` | **竞品横评 / 要量化打分分档**时读 | 分档评分标准、goodcase/badcase 阈值、GSB / 优势占比——把"孰优孰劣"量化 |

读法补充：先快速看相关 `index.md`（`models/index.md`、`methods/toolbox/collection/index.md`、`methods/toolbox/analysis/index.md`、`methods/standards/index.md`）确认有哪些条目，再决定打开哪几篇，避免空读。**按需精读，控制读取量**。正典里 `related:` 指向的文件，**只有用 Glob/Read 核验确不存在时**才记为「走空链接」。经验数字（评估员人数、严重度档位等）常在多篇正典里互相转引——运行说明里务必把每个数字归到**你这次实际读到它的那篇**。

## 第 3 步 · A：直接评（默认模式 —— 有界面素材时）

你现在就是评估员。按方法本体的**两遍法 + 走查节奏**，对用户提供的素材逐屏 / 逐流程检视：

1. **第一遍 · 整体浏览**：先把素材整体过一遍熟悉系统、看直观的内容/布局/视觉问题，**先不急着挑刺**。
2. **第二遍 · 逐条比对 + 模拟使用**：扮演真实用户按重要程度走核心流程，**逐条拿十大启发式比对**找违反点；**刻意试逻辑边界 / 异常路径**（库存边界、空态、错误态、回退、关闭等）——边界最易暴露问题。
3. **提覆盖、降单视角偏差**：有意识地**从多个评估视角各扫一遍**（如新手 vs 熟手、不同核心任务、不同机型/状态），模拟"多名独立评估员"的合并效果——单一视角会漏检。
4. **逐条记问题**，每条落这几项（结构见 skeleton）：
   - **位置 / 证据**：在哪个页面/步骤，引用用户素材里的具体证据（"截图2 顶部"/"结算第3步"）——**不脑补界面**；
   - **违反的启发式**：对到 `nielsen-heuristics.md` 里的哪一条（可多条）；
   - **严重度评级**：按 `issue-prioritization.md` 的口径打分——默认 NN/g 0–4（频率×影响×持续性，凭判断即可）；若有影响范围数据（UV/点击/曝光/反馈量）可并用京东《体验问题评级标准》（影响范围×严重程度→等级）。给出评分理由；
   - **改进建议**：具体、可落地。
5. **汇总输出**：问题清单（表）+ **按启发式归类**的小结 + **按严重度排序**的 Top 问题 + 接 `issue-prioritization` 的修复优先级。
6. **就地标注**：基于 draft 正典、素材不足无法评到的流程、靠推断而非证据的判断（标"需真人验证"），都在该处标一句（呼应纪律 3）。

## 第 3 步 · B：产工具包（无界面素材时）

产出一份可直接执行的启发式评估工具包（结构见 skeleton），并**末尾明确索要界面素材**（"把要评的页面截图/原型链接发我，我可以直接评一遍"）：

- **启发式检查清单**：从 `nielsen-heuristics.md` 抽十条 + 每条的核查要点（取自正典，别默写）；
- **严重度评分卡**：取自 `issue-prioritization.md`——NN/g 0–4（档位 + 三因素）与京东《体验问题评级标准》（影响范围×严重程度→等级）两套，按是否有影响范围数据择用；
- **评估员任务书 + 两遍法脚本**：3–5 名独立评估员、先熟悉后比对、走查节奏、范围收窄、机型覆盖；
- **记录表 + 汇总模板**：问题记录字段 + 独立打分取均值 + 亲和图聚类汇总。

## 第 4 步：评估自检（拿"局限与误用"回头扫一遍）

**实时读 `models/nielsen-heuristics.md` 的"局限与误用"与 `heuristic-evaluation.md` 的"常见坑"**，对自己的产出逐条回扫，这是实质修改、不是走过场：

- **违反 ≠ 问题**：每条疑似问题是否结合了场景/替代方案判断？有没有把"提到某启发式"当"违反"、堆伪问题？把站不住的降级或删掉。
- **严重度**：是否按频率×影响×持续性给的，而非拍脑袋？是否如实给了低分项、没为凑数注水？
- **覆盖**：核心流程与逻辑边界是否都走查到？素材没覆盖到的，是否标了"未评到"而非假装评过？
- **不替代真人**：需要真实行为/数据才能定论的，是否标了"需真人验证"、并建议接 usability-testing？
- **证据**：每条是否有素材里的具体证据支撑，没有脑补界面？

把改动落到问题清单上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/heuristic-evaluation-skeleton.md` 的顺序输出：

- **直接评模式**：① **评估结论摘要（结论先行，放最前）**——一句话总体可用性判断 + **Top 3–5 严重问题速览**（每条：问题 + 严重度 + 一句改进方向），让人第一眼看到"最该先修的是什么"，细节在下方清单展开 ② 启发式评估报告（问题清单 · 按启发式汇总 · 按严重度排序 · 修复优先级）③ 评估自检结果 ④《本次运行说明》（五部分）

# Structure Interview Transcript — 单场访谈逐字稿 → 结构化小结（降噪整理版）

把一场访谈的"原始逐字稿 / 录音转写"，**围绕本次研究要回答的问题**，整理成一份结构清晰、有证据、可被后续综合调用的访谈小结。本 skill 的定位是**降噪整理**——把散乱的对话压成有结构的发现，**不急于拔总洞察、不做跨场归纳**。

> **边界**：单场 = 本 skill；多场综合出洞察 = `synthesize-qualitative-insights`（拔洞察、找跨人模式是它的事，别在这里做）。出提纲 = `generate-interview-guide`；出问卷 = `generate-survey`。本 skill 上游吃的是"已经发生的一场访谈的逐字稿"。

## 核心原则（先理解，再动手）

- **主题骨架优先由研究目的 / 研究内容 / 研究方案驱动，理论框架只作深化透镜——绝不写死模板。** 小结是给**调研需求方 / 提出人**看的，它最该回答的是"我当初提的那几个研究问题、结论分别是什么"。所以分主题发现的**主干 = 本次研究目的与研究内容**（需求方关心的重点 / 研究问题 / 研究方案的条目 / 访谈提纲维度），这些尽量由 skill 使用者提供。理论框架（ORID / 旅程地图 / 同理心 / 价值主张 / ABC）是**在每个主题内深挖的透镜**（把"现象"推到"动机 / 情绪 / 矛盾"），**不是顶层骨架**——纯按理论框架的层级铺小结，容易陷进框架细节、却答不到需求方真正问的问题。只有**固定头部**（受访者背景 / 访谈执行信息）稳定可固定。
- **数据源 = 运行时实时读取的最新 Wiki，绝不把正典内容写死进本 skill。** 框架的定义、字段的细则、选型的判据，每次都从当前 Wiki 文档现读。本 skill 只告诉你**去读哪些路径、怎么组装**，不复制任何正典内容（复制 = 一旦 Wiki 更新就过期、就成了第二出处）。
- **缺内容不编造。** 逐字稿没提到的背景字段、研究内容没覆盖的维度，一律标"未提及 / 待补充"，绝不凭空生成看似真实的内容。原话必须是逐字稿里真实出现过的。
- **敏感素材会话内处理、不持久化。** 逐字稿是敏感原始数据：只在本次会话里用，**不写入 Wiki、不落任何正典 / 持久化文件**；产出里引用原话时做匿名化（见下"敏感素材处理"）。

## 工作流

### 1. 接稿 + 锁定研究目的与研究内容（决定主干骨架）
拿到单场逐字稿后，向 skill 使用者要齐三样：① **研究目的**；② **研究内容 / 研究问题**（需求方关心的重点，最好是研究方案里列的那几条，或访谈提纲的维度）；③（可选）**研究方案 / 访谈提纲原件**。其中"研究内容 / 研究问题"清单就是小结分主题发现的**主干骨架**——优先让使用者提供。
- 若只给了一句研究目的、没有研究内容拆解：**据研究目的 + 访谈提纲 + 逐字稿实际覆盖到的话题，先提出一份研究内容主干（3–6 条），并标注"据目的 / 逐字稿推断，待需求方确认"**，再据此组织——不要默默套理论框架当骨架。
- 若连研究目的都没有：**先问，不要自己猜一个往下做**（目的决定一切）。

### 2. 读"访谈如何被结构化记录"的正典 → 定固定头部 + 痛点/待跟进的字段来源
运行时读这两份（都是当前 Wiki 的实时内容）：
- `methods/toolbox/collection/interview-guide-design.md` —— 取**背景维度**（用户基本/社会属性、日常生活、相关行为习惯）以及"完整框架 → 访谈记录表"的结构化记录思路；它也印证"提纲框架（即研究内容维度）才是记录与小结的天然骨架"。
- `methods/toolbox/analysis/affinity-diagram.md` —— 它的"整理脚本"给出**可读脚本的字段与匿名约定**：用户编号 / 假名（保护隐私）、基本信息 / 行为习惯、使用产品的细节流程；以及便签标记约定 **D = 设计想法 / Q = 待研究问题 / 无标记 = 痛点体验**——这套标记直接对应小结里的「待跟进」「痛点」。

### 3. 选**一个**理论框架作**深化透镜**（不是骨架）
主干骨架已由第 1 步的研究内容 / 研究问题确定。这一步是**在各主题内**配一个理论框架作透镜，帮你把发现从"现象"推进到"动机 / 情绪 / 矛盾"。运行时读框架正典，按下表按研究目标类型起步选型，**最终以所读文档自带的"选型提示"为准**；选定后在《本次运行说明》写明"选了哪个作透镜、为什么"。**框架是工具不是骨架，绝不用它的层级充当顶层结构、也不内置死。**

| 本次研究目标类型 | 起步框架（作透镜） | 运行时读哪份正典 |
|---|---|---|
| 探索需求 / 动机（为什么用、为什么不用、JTBD） | ORID（O→R→I→D）或 价值主张画布 | `models/orid.md` / `methods/toolbox/analysis/qualitative-insight-frameworks.md` |
| 可用性测试 / 任务走查（卡在哪、为什么卡） | 用户旅程地图 | `methods/toolbox/analysis/qualitative-insight-frameworks.md` |
| 品类 / 消费决策调研（决策链路） | 用户旅程地图 | `methods/toolbox/analysis/qualitative-insight-frameworks.md` |
| 共情 / 建立典型用户画像 | 同理心地图（Say-Do-Think-Feel） | `methods/toolbox/analysis/qualitative-insight-frameworks.md` |
| 检验某方案 / 价值点是否击中需求 | 价值主张画布 | `methods/toolbox/analysis/qualitative-insight-frameworks.md` |
| 解构单一矛盾 / 反常行为 | A/B/C 模型（情感-行为-认知） | `methods/toolbox/analysis/qualitative-insight-frameworks.md` |

> 上表是本 skill 的**路由起点**，不是正典；框架的实际定义、各层含义、选型细则都在文档里，**读了再用**。研究目标若跨多类，选最贴主目标的一个作透镜即可，别拼几个框架。**透镜服务于主干（研究内容），不要让框架反客为主、把小结铺成框架教学。**

### 4.（单场内）发现点多而散时，先归并主题
若一场里发现点很多、很碎，参考 `methods/toolbox/analysis/affinity-diagram.md` 的自下而上归并思路，**在本场内**把单条发现聚成几个主题，再挂到第 1 步的研究内容主干（对应的研究问题）下。**只在本场内归并，不跨场**——跨场是 `synthesize-qualitative-insights` 的事。

### 5. 产出结构化小结
按 `references/output-skeleton.md` 组装：固定头部（背景 / 执行）+ **以研究内容 / 研究问题为主干的分主题发现**（框架作透镜深挖，每点配匿名关键原话）+ 痛点 + 待跟进。缺的字段标"未提及 / 待补充"，不编造。

### 6. 末尾附《本次运行说明》
见下"通用纪律"，格式见 `references/output-skeleton.md` 的 D 节。

## 敏感素材处理（本类 skill 特有，务必遵守）

研究员上传的逐字稿是**敏感原始数据**：
- **只在本次会话内临时处理**，完成即止；**不写入 Wiki、不落任何正典 / 持久化文件**。本 skill 的产物是发给研究员看的小结，不是要进知识库的文档。
- 产出中引用原话时**去除无关 PII**：真实姓名、联系方式、可定位到个人的信息（公司全称、门店地址、账号等）。用**用户编号 / 假名**（如 U1、"受访者A"）替代——这正是 `affinity-diagram.md` 整理脚本的隐私约定。
- 在**不损失原意**的前提下做匿名化：原话的口语、情绪、关键细节要保留（它们是降噪整理的证据），只替换可定位信息。

## 通用纪律（每次产出固定遵守）

1. **数据源 = 最新 master 版 Wiki，运行时实时读取，绝不把正典内容写死进本 skill 或其 references。** 本 skill 只写"去读哪些路径"，内容每次从当前 Wiki 取。
2. **产出末尾固定附《本次运行说明》**，列：① 调用了哪些正典（路径 + `status`）；② 本次主题骨架的来源（研究方案 / 研究内容 / 访谈提纲，或据目的+逐字稿推断的"待确认主干"）+ 选了哪个框架作**深化透镜**、为什么；③ 该调用但未找到的内容；④ 走空的链接；⑤ 哪些部分基于 `draft` 正典 / 哪些是未找到正典而用通用方法兜底；⑥ 反馈入口 `[http://xingyun.jd.com/codingRoot/JD_Research_Wiki/Research_Wiki/settings/issues]`。
3. **在交付物正文里声明局限**：本次依赖的 Wiki 文档均为 `status: draft`（未评审）；凡未找到正典而用通用方法兜底之处，就地标注；据推断生成的研究内容主干，标注"待需求方确认"。
4. **缺内容不编造**：找不到就标"未找到 / 待补充"，绝不凭空生成看似正典或看似原话的内容。

> **关于"Wiki 无固定访谈小结模板"**：这是**设计如此**（结构按项目动态定），不是缺口。运行说明里注明本次主干结构的来源（研究内容/方案/提纲 或 推断）与所选透镜框架即可，不要因为"没找到小结模板"而报缺口。

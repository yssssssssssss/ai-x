# Analyze Satisfaction — 满意度数据 → 关键驱动 + IPA 改进优先级 + 掉分归因（编排 wiki 正典）

吃一批满意度分数和一个分析目标，产出**这一批专用**的满意度诊断。本 skill 不发明分析方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的满意度分析正典——驱动因素分析（KDA）、满意度分析模型（四分图 / Kano / AHP / SEM）、异动归因三阶段、数据标准化、期望确认理论 ECT——按本批数据的形态（截面 or 时间序列）与目标，回答三个问题中的一个或几个：**哪些属性最撬动满意度（驱动）、先改哪个（IPA 优先级）、分数为什么掉了（归因）**。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 满意度分析方法的唯一出处在 wiki 正典里。你的工作是按本批数据**读对正典、抽对方法、套进产出骨架**，而不是凭记忆复述「驱动分析怎么做」「IPA 是什么」。产出的价值在于贴合 *这一批* 分数和 *这一个* 目标，不是又一份通用方法说明。
- **先分清"手里是什么数据、要答什么问题"，再选路线。** 本 skill 有三条主线，各吃不同形态的数据、各有产出：**截面（总分+属性分）→ 驱动因素 + IPA**；**时间序列（本期 vs 对比期）→ 掉分归因**；**上线前后期望/感知 → ECT 落差**。选错路线，后面算得再漂亮也答非所问。三条可组合（如先归因定位到掉分模块，再对该模块跑驱动分析）。
- **算你能算的，不编造统计量。** 这是本 skill 区别于"嘴上讲方法"的关键纪律，单列见下方纪律 4。一句话：相关系数、均值、Top-box、NSS/NPS 口径、IPA 象限落点、Gap=E−P、结构因素拆解、单维卡方/置信区间、Min-Max/Z-score/TGI、Kano 归类与 Better-Worse、双重标准化——这些**你能直接算就真算**；而相对权重/Shapley、SEM 标准化系数、AHP 一致性比值 CR、Box-Cox 最佳 λ 这类需要统计软件的，**有行级数据 + Python 环境就写脚本真算，否则只描述方法并诚实标注"需统计工具"，绝不写一个看似精确却没真算的数**。
- **数据不够算，就先要数据，别硬出结论。** 每条路线都有"最低数据要求"。若缺的是**核心依据**（驱动分析没有逐份行级数据、归因没有样本量或下钻维度、ECT 没有期望或感知）→ **先停下、产出一份《数据补充请求》请用户补齐再分析**，不靠假设/占位/"凭经验估"把空缺糊成结论；若缺的只是**精度/粒度**（样本偏小、无趋势元数据）→ 降级 + 就地声明继续。判据：能不能给出一个**有效结论**——哪怕结论是"数据不足以判定、建议先监控"（这本身有效）；给得出就继续，只能靠编就先要数据。详见第 1 步「数据准入闸」。
- **定量定位"哪里"，定性回答"为什么"。** 驱动权重和 IPA 告诉你哪个属性最该改、掉分归因定位到哪个群体哪个模块——但**为什么**那个属性表现差、那个群体为什么不满，要靠 ECT 的期望-感知落差、开放题/关键词、低分回访来补。只给一堆数字不给机制，等于把诊断停在半路。**相关≠因果**：派生重要性高只说明该属性与总分同向变动，不等于"改了它满意度必涨"，闭环要靠 A/B（在正文点明）。
- **分层交付：结论写干净，方法与不确定性集中到后面。** 使用者要的是一份能直接拿去开会、排迭代的诊断，不是分析过程的实况转播。正文的「发现」（诊断结论摘要、驱动排序/IPA、归因四段、Gap 表）**先把结论说清楚、能扫读**；而**算力声明、样本/统计边界、draft 背书、相关≠因果**这些「依据与边界」**集中放到文末**（数据研判小结 +《本次运行说明》的算力边界一节），不要逐句夹进每一条发现里。判据：读者**只读发现段就应拿到一份干净、可汇报的诊断**，想深究再往下看依据。做法上——① 结论先行（先给答案，再给数）；② 一句话能说清就不写一段；③ 每个发现只在**真正影响该结论成立**时才带一句轻量 ⚠️（如"该维度 n 小、权重不稳"），系统性的口径/算力交代一律下沉到文末；④ 掉分归因先给**最主要的一条**根因，其余一笔带过。把过程/自证与结论揉在一起，是这类诊断最伤可读性的通病。
- **边界：满意度分数的定量诊断 → 本 skill；其它走隔壁。**
  - 还没数据、要**设计满意度问卷**（指标选择、量表、三段式）→ `generate-survey`。本 skill 是有分数之后才用的。
  - 海量**开放题 / 评价 / 工单的文字**要编码成痛点清单（纯定性文本、不是分数）→ `code-open-feedback`。本 skill 只在归因第三阶段借开放题做"为什么"的线索，深做文本编码转那边。
  - 通用问卷**交叉表分群显著性对比**（会员 vs 非会员谁更满意这类、与满意度诊断流程无关的列联表检验）→ `survey-crosstab-analysis`。注意：掉分归因里的显著性判断（卡方/置信区间）是本 skill 异动流程**自带的一道闸**，不必外转。

## 敏感素材处理（先读再动手）

满意度数据多为数值分数（不敏感），但常夹带**开放题原文、低分用户回访记录、用户 ID / 联系方式**——这些是敏感原始数据：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户原始作答、回访记录写进 `research-wiki/` 或落盘到 skill 目录。
- **引用开放题原话做匿名化，且贯穿每个角落**（正文、代表原话、《本次运行说明》都不得出现真实姓名 / 手机号 / 订单号 / 单位）。要说明做了匿名化，只陈述"已匿名"，**绝不回写"真名→代称"对照**（那等于没匿名）。涉及联系方式等字段尤其当心，既是隐私要求，也避免触发内容网关拦截。
- **分数与原始表只读不改。** 你的职责是分析，不是修订数据。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把 KDA、四分图/Kano/AHP/SEM、异动归因三阶段、标准化公式、ECT 落差法写死在这里或 references 里**。每次都从当前 wiki 现读，确保是最新版——正典可能已从 draft 改为 reviewed、公式或口径可能已修订。
2. **产出末尾固定附《本次运行说明》**，五部分齐全（结构见 `references/diagnosis-skeleton.md`），让使用者知道这份诊断建立在什么之上、哪里可能不牢。
3. **在交付物里声明局限——集中归口，不淹没结论**：基于 draft（未评审）正典的部分、样本量不足或量表有天花板效应的部分、因未真跑统计软件而只给方向估计的部分、相关非因果的部分，都要交代清楚，别让使用者误以为处处有正典背书、或误以为是经检验的结论——但**集中写到文末的数据研判小结 +《本次运行说明》**，发现层只在真正影响该条结论时留一句轻量 ⚠️（呼应北极星「分层交付」）。
4. **缺内容不编造；这里的"不编造"尤其指不编造数据与统计量。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，且**写下"走空"之前先用 Read 核验该路径确不存在**（本环境 bash 的 `ls`/`find` 可能不可靠，核验一律用 Glob/Read；把真实存在的正典误报成缺失，和编造一样有害）。**更要命的是统计量造假**：派生权重、R²、相关系数、p 值、置信区间、AHP 的 CR、SEM 标准化系数、Kano 的 Better-Worse、Box-Cox 的 λ——凡不是你**真实算出来**的，绝不写一个看似精确的数。算力够（有行级数据 + 能跑 Python）就写脚本真算；不够就**只描述该方法、给方向性判断、明确标注"需统计工具/未实算"**。满意度诊断会真的拿去排迭代、定 P0，宁可少给、也不可拿假数误导。

## 第 1 步：接收并研判这批满意度数据

动手算之前先搞清四件事，它们决定走哪条路线、读哪些正典、能算到什么程度：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **数据形态** | **截面**（一期：总体满意度/NSS + 各属性满意度）→ 走路线 A 驱动+IPA；**时间序列**（本期 vs 对比期同口径分数 + 样本量）→ 走路线 B 归因；**期望/感知配对**（上线前后逐价值点 E 与 P）→ 走路线 C ECT。三者可组合。 | 不清楚 → 看用户给的表结构推断（有"属性列+总分列"=截面；有"上期/本期"两列=时间序列），并在运行说明声明判断依据 |
| **分析目标** | 先改哪（驱动+优先级）？为什么掉分（归因）？上线收益（落差）？决定路线与产出落点 | 完全缺失 → 先问清，这是不能假设的核心；漫无目的地跑模型只会得到一堆没人要的数字 |
| **样本量 n（每期/每组）** | 驱动估计与显著性检验的前提。**模块 n<20、系统 n<100 不对单期波动做归因，只作监控**（异动归因正典的硬门槛）；样本太小驱动权重也不可靠 | 缺 → 标注"未提供样本量，显著性/置信判断无法做，仅给方向"，并提醒补 n |
| **可下钻维度 / 元数据** | 用户分层（新老/会员/RFM/部门岗位）、模块标签、时间、版本——有则能下钻定位「谁在哪个模块」、能区分阴跌vs跳水、能拆结构因素 | 缺 → 只出整体结论，标"无下钻维度，无法定位到具体群体/模块" |
| **属性清单 & 量纲** | 驱动分析要求属性**覆盖全、尽量互斥**；各属性是否同量表（5/7/10 点）？跨平台/跨人群比较吗？决定要不要先做**数据标准化** | 属性残缺 → 提示可能把作用错配给相邻属性；量纲不一/跨平台 → 触发路线里的标准化前处理 |

落表之前先做两个判别：

- **这批数据是不是本 skill 该接的？** 若其实是要设计问卷 → 转 `generate-survey`；若是一坨开放题文字要编码 → 转 `code-open-feedback`；若是与满意度无关的通用交叉表分群对比 → 转 `survey-crosstab-analysis`。
- **算力够不够支撑用户想要的方法？** 用户要 Shapley/SEM/AHP 权重但只给了汇总均值（没有行级数据）→ 明说"这些需逐份作答的原始数据 + 统计软件，当前只能给基于均值/相关的方向估计"，不要假装算了（呼应纪律 4）。

### 数据准入闸（硬缺口 → 先补数据再分析，不硬出结论）

研判完、动手算之前先过这道闸：**对照所选路线的"最低数据要求"，判断现有数据能不能产出一个有效结论。** 这一步决定"立即分析"还是"先请用户补数据"。

- **硬缺口**（缺了它，所选路线的核心产出——驱动排序 / IPA / 归因定位 / Gap——就没有有效依据）→ **不进入分析**，先产出一份《数据补充请求》（结构见 `references/diagnosis-skeleton.md`）：逐条列「缺什么 / 为什么这条分析离不开它 / 最低可接受的数据形态 / 补齐后我能产出什么」，并给「若你暂时只能提供 X，我可以降级产出 Y（但给不了 Z）」的退路。**绝不用假设、占位或"凭经验估"把硬缺口糊成一个看似成立的结论。**
- **仅影响深度 / 稳健性**（核心产出仍有效，缺的只是精度或下钻粒度）→ 不打断用户，按"合理假设 + 就地声明"继续（如样本偏小、无趋势元数据、根因缺行为数据）。
- 判据一句话：**能不能给出一个"有效结论"——哪怕结论是"数据不足以判定显著、建议先监控"（这本身有效）。给得出 → 继续；给不出、只能靠编 → 先要数据。**

| 路线 | 最低数据要求（必需） | 硬缺口 → 先请用户补 | 仅影响深度 → 降级继续 |
|---|---|---|---|
| **A 驱动+IPA** | 总体满意度 + ≥3 属性满意度，且为**逐份作答的行级数据**（才能算相关 / 派生重要性） | 只有各属性**汇总均值**、无行级数据 → 派生重要性无从算；若连重要性 / 权重评分也没有 → 四分图也做不了，只能描述"表现高低"，**不能据此说哪个"最撬动满意度"**。请补**行级数据**，或退而求其次补各属性的**重要性 / 权重评分** | 样本偏小（n 小）、属性可能不全、量纲不一（标准化可解） |
| **B 异动归因** | 本期 + 对比期**同口径**分数 + **各期样本量 n** | 无样本量 → 判不了显著、够不够归因门槛；无任何**下钻维度**（用户分层 / 模块）→ 定位不到"谁在哪"，再问"为什么"就是生造根因。请补 n（及做卡方所需的满意 / 不满意人数）+ 可下钻字段（+ 二级归因占比 / 开放题更好） | 有 n、有下钻但无行为数据 / 开放题 → 能定位 who/where，根因层标"待行为数据 / 回访复核" |
| **C ECT 落差** | 逐价值点的**期望 E 与 感知 P 配对**（至少部分价值点两者齐全） | 只有 E 或只有 P → Gap 无从算。请补缺的一端，或说明用哪种降偏差测法补测 | 仅事后单测（近因效应）→ 可算但标偏差 |

> 这一闸的意义：满意度诊断会真的拿去定 P0、排迭代。**一个建立在缺失数据上的"方向估计"，若被当成结论用，比不给更危险。** 先花一次对话把数据补齐，胜过给一份处处打补丁、却仍可能误导决策的分析。注意分寸：**别因小缺口频繁打断**（精度类缺失降级即可），也**别对硬缺口硬算**——两者的界线就是上面的"能否给出有效结论"。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

## 第 3 步：路线 A — 关键驱动因素 + IPA 改进优先级（截面：总分 + 属性分）

骨架见 `references/diagnosis-skeleton.md`，**起草时打开它**。按 KDA 正典做：

1. **定因变量与自变量**：因变量=总体满意度/NSS/NPS（用户给的那个总分）；自变量=各属性满意度。先核对属性是否覆盖全、尽量互斥；残缺就在结论里提示"可能错配作用"。
2. **算派生重要性**——按你的算力诚实分层（呼应纪律 4）：
   - **能直接算**：每个属性与总分的**相关系数**（做体检、看方向与强度）；属性**满意度均值 / Top-box**（表现轴）。这两项**务必真算**，是 IPA 的两根轴。
   - **想要稳健权重（相对权重/Shapley/多元回归 beta）**：属性间往往高相关（多重共线性），原始 beta 会失真——KDA 明确要改用相对权重/Shapley。**有逐份作答的行级数据 + Python**：写一段脚本真算（如用 `numpy`/`sklearn` 跑回归、用 Johnson 相对权重或 Shapley 思路分摊 R²），把脚本与结果一并交付。**只有汇总均值/没有环境**：用相关系数排序作为派生重要性的**近似**，并明确标注"未做相对权重/Shapley，权重为基于相关的方向估计，稳健权重需统计工具"。**绝不凭空写一个 beta=.55 / 权重=23% 这类没真算的数。**
3. **画 IPA 四象限**：横轴=表现（属性满意度均值），纵轴=重要性（派生权重或其相关近似），用均值划十字 → **高重要·低表现=重点改进（优先排期）/ 高重要·高表现=保持优势 / 低重要·低表现=次要 / 低重要·高表现=过度投入**。逐属性落点。
4. **Kano 兜底定性**：派生重要性会低估**基本型**属性（做好无感、缺失塌方），在结论里提醒别因回归数字低就砍基本盘；若用户给了正反向成对问卷数据，可按四分图正典做 Kano 归类 + Better-Worse（能算就真算）。
5. **可选·量纲不一/跨平台**：若各属性不同量表、或要跨平台/跨人群比，先按 data-standardization 做归一化/Z-score/双重标准化，**标注"标准化值仅供比较"**。
6. **可选·重型权重**：用户明确要 AHP（判断矩阵权重 + CR 一致性检验）或 SEM（标准化系数）——按满意度分析模型正典描述步骤；CR、SEM 系数等若没真算，标"需 AHP/SEM 软件，未实算"。

> 这步的产出是**驱动权重排序 + IPA 四象限 + 重点改进清单**，每条改进项点明"为什么进重点改进区"（高重要低表现），并把"相关≠因果、需 A/B 验证"写在结论。

## 第 4 步：路线 B — 满意度异动 / 掉分归因（时间序列：本期 vs 对比期）

骨架见 `references/diagnosis-skeleton.md`。**严格按异动归因正典三阶段走，顺序不能跳**——跳过第一阶段直接归因，是最常见的误用：

**第一阶段 · 数据判断（先排除"假异动"，否则白忙）**
- **样本量闸**：模块 n<20、系统 n<100 → 不对单期波动归因，只标"样本不足、仅作趋势监控"。
- **口径闸**：确认本期与对比期计算口径一致，避免把"换了口径"误读成"体验变化"。
- **显著性闸**：有样本量就判显著——**卡方**（P<0.05 才算显著差异）或**置信区间**（两区间不重叠=显著、大量重叠=波动、轻微重叠=灰区回到卡方）。2×2 这类简单卡方/置信区间**能算就真算**；算不了就标"需显著性检验工具"。**不显著 → 明确收尾"以波动为主，无需深度归因"，不要硬编根因。**
- **结构因素闸**：分数变化可能来自**填答人群结构变了**（如新/低频用户占比升高拉低均值）而非体验劣化。按"假设用户结构不变"反推结构贡献（带数字算例见 b-end 场景正典），**先扣除结构因素，再对"同人群内部下降"归因**。

**第二阶段 · 定位（谁在哪里不满）**
- 看整体趋势是**阴跌**（长期积累）还是**跳水**（特定事件如改版/故障）。
- **用户维度下钻**（新老/会员/RFM/部门岗位…）+ **模块维度下钻**，交叉出"哪个群体在哪个模块跌最狠"，形成**问题假设**（如"采销新员工在比价查询模块满意度暴跌，疑与改版有关"）。下钻全靠分组算均值/占比，**能算就真算**。

**第三阶段 · 根因（为什么不满）**
- **不满原因占比**（吃二级归因多选的占比及环比）→ 聚焦贡献最大的原因。
- **负面关键词 / 开放题线索**（深做文本转 voc-analysis / code-open-feedback）。
- **多团队协作**：推断掉分时间点是否对齐版本发布/系统故障/外部舆情。
- **关联行为数据**：任务成功率、性能、错误日志、路径退出率（用户给了才用，没有则标"建议补行为数据复核"）。
- **低分回访**：问法要**具体场景化**（不要问"为什么打低分"，要问"您在【某步骤】打了低分，是哪一步卡住了"）。

**输出按「事实—定位—根因—总结」四段叙事**：事实（本期分数+环比幅度）→ 定位（哪个群体在哪个模块）→ 根因（数据+原声佐证的可行动原因）→ 总结（给产研的优先优化建议）。

## 第 5 步：路线 C — 上线后期望落差（ECT，常叠加在 A/B 上）

**当数据含"上线前后 / 逐价值点的期望 E 与感知 P"**，或要解释 A/B 里"为什么这个属性表现差"时，用 ECT：

- 把迭代拆成可独立评价的**价值点**，逐点填四层表（认知 / 期望E / 感知P / 落差Gap=E−P）。Gap 是**简单减法，务必真算**。
- **Gap 显著为正 = 失望点（候选 P0，进迭代待办）；显著为负 = 惊喜点（沉淀卖点）；≈0 = 确认**。
- 注意测量偏差：理想是使用前测 E、使用后测 P 或独立分组；只有事后数据则**承认近因效应**会让事后评分主导，审慎解读并写进局限。
- 认知层=否的价值点（用户根本不知道有此功能）→ 标"先补认知"，不算落差。
- ECT 是 A/B 的"为什么"层：把失望点的落差数据回流给路线 A 的驱动分析、把根因落到路线 B 的第三阶段。

## 第 6 步：组织交付

> **若第 1 步数据准入闸判定为硬缺口** → 本次交付就是一份**《数据补充请求》**（按骨架对应模板），不是诊断报告：说清判定的路线与缺口、列出待补数据清单与理由、给降级退路，附简版运行说明即可。下面的路线产出**待数据补齐后再做**，不要先硬出一份。

按 `references/diagnosis-skeleton.md` 的顺序输出**本次实际走了的路线**对应的块（没走的路线不强行凑）。交付**分两层**——先「发现层」写干净，再「依据与边界层」集中收口（呼应北极星「分层交付」）：

**【发现层，写干净、能扫读】**
1. **诊断结论摘要（结论先行，放最前）**：一句话回答本次目标——**关键驱动 Top 几 / 重点改进属性 / 掉分根因或 P0 失望点** + 3–4 条关键发现要点，让产研第一眼看到结论、不用先读方法；**别写成一大段**。（若第 1 步判定为硬缺口，本节改为一句话"数据不足以出诊断结论，见下方《数据补充请求》"。）
2. **路线 A**：驱动权重排序（标明是相对权重/Shapley/回归 还是 相关近似）+ IPA 四象限（逐属性落点）+ 重点改进清单（每条点明象限理由）。
3. **路线 B**：四段叙事（事实-定位-根因-总结）+ 三道闸的判定结果（样本量/口径/显著性/结构因素）。
4. **路线 C**：逐价值点 Gap 表 + 失望点(P0)/惊喜点清单。

**【依据与边界层，集中放后面】**
5. **数据研判小结**：判定的数据形态、所选路线、样本量与可下钻维度、能算到什么程度（**算力声明**）。（方法论与算力边界集中在此，不要放在结论摘要之前、也不要拆散夹进各路线发现里。）
6. **《本次运行说明》**（五部分，见骨架，逐项填实，主文档最后一节）。

**局限归口，不逐句夹进发现里**：基于 draft 正典、样本不足/无样本量、未真跑统计软件而给方向估计、相关非因果、无下钻维度无法定位——这些系统性口径与边界**集中写在第 5 步数据研判小结 + 第 6 步运行说明**；发现层只在真正影响某条结论时带一句轻量 ⚠️。

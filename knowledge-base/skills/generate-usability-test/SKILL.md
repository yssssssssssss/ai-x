# Generate Usability Test — 测试对象 + 研究目标 → 可用性测试执行包（编排 wiki 正典）

吃一个测试对象与研究目标，产出一份**这次专用**的可用性测试执行包。本 skill 不发明测试方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的方法正典——可用性测试本体、标准化体验量表（SEQ/SUS…）、受访者偏误规范、抽样与招募规范——把它们组装成贴合本次对象、阶段与人群的任务脚本、记录表与度量方案，并拿偏误规范回头自检。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** 任务设计原则、量表题数·计分·基准、Nielsen 样本量、偏误清单的唯一出处都在 wiki 正典里。你的工作是按本次 brief **读对的正典、抽对的要点、填进执行包骨架**，而不是凭记忆复述「可用性测试怎么做」。产出的价值在于贴合 *这一个* 产品、*这一个* 测试阶段、*这一类* 用户，不是又一份通用模板。
- **任务是测试的根基，且是定制品。** 可用性测试的结论好坏，首先取决于任务设计的好坏。同样测「搜索」，测草图原型和测灰度 demo、测新用户和测资深用户，要拆的核心流程、起止页面、要不要给场景都不同。务必让测试对象、阶段、用户类型真正改变任务内容——任务必须落在产品**最核心、最高频**的流程上，给目标场景而**不给操作步骤**。
- **观察纪律 = 数据质量。** 一句「你觉得好不好用？」、一次中途插话、一个被观察就刻意表演的用户（霍桑效应），都会污染整场测试。脚本落笔后必须拿偏误规范回头自检——这是本 skill 区别于「随手列几个任务」的关键一环。
- **边界：给任务、看操作 → 本 skill；纯开放访谈 / 大规模问卷 / 选方法 → 隔壁。** 本 skill 面向**布置任务、观察操作、度量可用性**的测试（虽含事后访谈，但主轴是任务与观察）。若用户真正要的是不带任务的 1v1 深访 / 焦点小组挖动机，用 `generate-interview-guide`；要的是可投放、可统计的问卷，用 `generate-survey`；需求还在「这事该用什么方法、整体怎么排」阶段，用 `generate-research-plan`。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把任务原则、量表题干/点数/计分公式/基准、Nielsen 样本量、偏误清单写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——规范可能已从 draft 改为 reviewed、量表可能已扩充。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这份执行包建立在什么之上、哪里可能不牢。结构见 `references/usability-test-skeleton.md`，五个部分一个都不能少。
3. **在执行包正文里就地声明局限**：基于 draft（未评审）正典的部分、因量表库未覆盖该度量而现写题的部分、因未找到贴合场景而用通用方法兜底的部分，都在该处直接标注，别让使用者误以为处处有正典或标准量表背书。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的任务原则、看似标准量表的题干/计分、或冒充本库规范的样本量数字。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 工具实际核验该路径确实不存在。**绝不靠目录列举、shell 报错或印象就断定"它不存在"——把真实存在的正典误报成缺失和编造内容一样有害。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，所以先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若当前已在该树内，向上找到含 `methods/ models/ assets/` 的那一层；③ 都不确定时，用 glob 搜锚文件 `**/methods/toolbox/collection/usability-testing.md`，其所在的 `.../research-wiki/` 即根。
- 工具选择：定位与「文件是否存在」的核验**一律用 Glob/Read**。本环境 bash 的 `ls`/`find` 可能失效，别用 shell 列目录来判断有没有某个文件。
- **找不到 wiki**：不要硬编一份测试方案糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用经验」，且**不得**伪造任何「依据可用性测试方法第 X 条 / 标准 SUS 计分」的说法。

## 第 1 步：解析 brief

从用户输入里抽出四个要素，决定后面读什么、执行包长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **测试对象（产品/功能 + 形态）** | 决定任务取哪几条核心流程；形态（纸/草图原型、可点 demo、已上线）决定测试阶段 | 完全缺失 → 先问清，这是不能假设的核心 |
| **研究目标** | 决定测什么（验证方案是否可用 / 定位可用性问题 / 竞品对比 / 看视觉关注点），决定要不要量表、要哪种度量 | 完全缺失 → 先问清 |
| **测试阶段** | 设计初始（探索性，纸原型+定性深访）/ 灰度前（评估性，面对面+SUS）/ 上线后（对比性，A/B+问卷），决定方法搭配与量表 | 缺 → 按对象形态推断（草图→探索性、可点 demo→评估性）并声明 |
| **目标用户 + 形式/时长（现场/远程、单场时长）** | 决定招募甄别、样本量、脚本节奏 | 缺 → 默认 1v1、现场或远程任一并声明、单场 45–60 分钟、5–8 名用户，并在运行说明声明 |

判断原则：**核心目标缺了就问，次要参数缺了就合理假设 + 声明**——不要因为缺一个时长或形式就卡住不动，也不要在测试对象或目标都没有时硬编一份脚本。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**。读每篇时带着「我要从这抽什么填进执行包」的目的，而不是通读。读完记下每篇的相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于执行包 |
|---|---|---|
| `methods/toolbox/collection/usability-testing.md` | **必读（主线）** | 可用性五要素；**三阶段测试**（探索性/评估性/对比性）的方法搭配；**STEP1 任务设计六原则**（选最核心流程、符合常规操作、为任务创建应用场景、明确起止页面、不过于简单、不给线索与操作步骤）；STEP3 物料（数据收集表 / 访谈提纲）；STEP4 试点前测要点；STEP5 观察不打断 + 事后访谈三层（认知习惯 → 需求关注 → 再追"为什么"，别只问"好不好"）；STEP6 报告结构 |

## 第 3 步：起草执行包（五件套）

骨架与各段写法见 `references/usability-test-skeleton.md`，**起草时打开它**。先按第 1 步定的测试阶段选好方法搭配，再落五件套：

1. **任务场景**：把研究目标拆成几条**核心、高频**的操作流程，每条写成一个**带应用场景**的任务（"你关注的双 11 报名开始了，请上后台完成报名"），明确**起点页面 → 终点页面**作为成功判据；**只给目标、不写操作步骤或按钮名**（否则会错过用户真实卡点）；任务不过于简单；多任务按常规操作流串联、不冲突。
2. **出声思考脚本**：开场含**自我介绍 + think-aloud 训练（请边操作边说出心里想法）+ 减负说明 + 录屏许可**；逐任务的**引导语**（念什么、给什么场景、何时开始计时）；过程中**不打断、不提示**的提醒；**事后访谈追问梯度**——先问认知习惯/需求关注，再就卡点追"当时在想什么、为什么这么操作"，**不止于"好不好用"**。
3. **观察记录表**：每个任务一行/块，含**是否完成（到没到终点页）、完成时间、出错/求助次数、关键事件与体验问题、SEQ 评分、观察备注**，给执行者现场即填。
4. **度量计划**：定义**任务成功率**（起点→终点的判定口径）；**SEQ** 每个任务做完即问（点数与低分追问口径取自量表库）；整场用 **SUS**（或短测用 UMUX-Lite）测整体可用性，写明计分与**对标基准/百分位**而非看绝对值；说明**样本量**与其依据；凡现写或库未覆盖的量表，就地标注。
5. **招募甄别**：列**关键特征 + 配额表**（覆盖关键差异维度）；给**甄别问卷**（含干扰项、开放题）；据抽样规范写**样本量**（Nielsen/基准 + 未出席率上浮、信息饱和、可分阶段招募）；涉及眼动追加视力等测试条件，B 端按利益相关者角色选样。

**就地标注**：任一处基于 draft、现写量表、或无贴合场景而通用兜底，在该处标一句，呼应纪律 3。

## 第 4 步：偏误与可用性自检（拿规范回头扫一遍）

**实时读 `methods/toolbox/collection/respondent-bias.md`**（并借 `usability-testing.md` STEP4 试点视角）对脚本与任务**逐条**回扫，这是实质修改、不是走过场：

- **任务表述**是否泄露了操作步骤 / 按钮名 → 改为只给目标场景（呼应任务设计原则，否则错过真实卡点）；
- **霍桑效应**：是否会诱导用户"表演" → 用熟悉环境、明确"我们测的是产品不是你"、引导复现平时真实操作；
- **友善偏见 / 峰终定律**：是否只在结尾问"整体好不好" → 改为逐任务、逐环节深挖细节，高分也追问；
- **顺序 / 锚定效应**：多任务或多方案是否固定顺序 → 随机化 / counterbalance，避免前置任务给后续埋锚点；
- **减负说明 / 求证**：开场减负是否到位、事后是否有复述求证环节。

把改动落到脚本与任务文本上，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明「基于未评审正典，定稿后需复核」。

## 第 5 步：组装交付

按 `references/usability-test-skeleton.md` 的顺序输出三块：

1. **可用性测试执行包**（任务场景 · 出声思考脚本 · 观察记录表 · 成功率/SEQ/SUS 度量计划 · 招募甄别）
2. **偏误与可用性自检结果**（对照偏误规范逐条 ✅/⚠️/❌ + 证据/怎么改）
3. **《本次运行说明》**（五部分，见骨架文件，逐项填实）

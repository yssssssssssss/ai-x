# Build Experience Metrics — 产品/系统 + 业务目标 → HEART 度量体系（北极星·护栏 + 采集口径 + 纵向追踪）（编排 wiki 正典）

吃一个产品/系统与一个业务目标，产出一套**这次专用、可落地的体验度量体系**。本 skill 不发明度量方法，而是**实时编排**用研知识库（`research-wiki/`）里既有的度量正典——HEART/GSM 体验度量、跨版本纵向追踪、数据标准化、B 端系统体验度量打法、标准化体验量表——把它们组装成贴合本产品、本阶段、本目标的**维度选择、北极星/护栏指标、采集口径、拆解树与纵向追踪方案**，并拿"指标好坏"的规范回头自检。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** HEART 五维、GSM 三步、基准值（SUS=68、SEQ≈5.5…）、量表口径、纵向追踪协议、B 端北极星四候选与频控规则，唯一出处都在 wiki 正典里。你的工作是按本次 brief **读对正典、抽对要点、填进体系骨架**，而不是凭记忆复述「HEART 是哪五维」「北极星怎么定」。产出的价值在于贴合 *这一个* 产品、*这一个* 阶段、*这一个* 目标，不是又一份通用度量模板。
- **GSM 是脊梁，不能跳过目标直接堆指标。** 一套度量体系最常见的塌方是：上来就把能埋的点全埋了、能问的分全问了，攒出一张谁也不看的大看板。正确顺序是**目标 G→信号 S→指标 M**——先讲清这个产品/版本要达成什么，再找能反映进展的可观测信号，最后才落成口径清晰的指标。没有目标牵引的指标，多到看不过来也定位不了问题。
- **区分核心与护栏，且度量要做减法。** 北极星是**一个**全局指标、定方向；护栏是**不可恶化的底线**（猛拉一个维度常会暗伤另一个——HEART 明确举例"猛拉采纳可能拉低愉悦度/留存"，必须给配护栏）。维度按目标选 **3–4 个**、不五维硬凑；核心追踪指标 NN/g 建议收敛到 **2–4 个**覆盖不同侧面。指标越多越像在测，其实越稀释重点。
- **度量只回答"变没变、变多少"，不回答"为什么"。** 本 skill 的产出是一套**能持续追踪**的体系，不是一次归因。指标下滑要转定性诊断、要验证某次改动是否有效要做 A/B——这些是**下游**。体系设计时只需**预留入口**（挂下钻维度、后台标签、把愉悦度/采纳/任务成功接到 KDA/feature-adoption/conversion-funnel），不在此展开归因。
- **分层交付：体系写干净，依据集中到后面。** 使用者要的是一套能直接拿去埋点、汇报的度量体系，不是方法论实况。正文**先把体系主体（GSM 推导、维度、北极星/护栏、口径看板、拆解树、采集方案）说清楚、能扫读**；而 draft 背书、当前无埋点/无基线落不了地、量表须对标百分位、通用兜底这些「依据与边界」**集中放到《本次运行说明》**，不逐句夹进每个指标下把体系淹没。判据：读者只读体系主体就应拿到一份干净、可落地的方案；体系层只在真正影响某指标能否落地时留一句轻量 ⚠️。
- **边界：建体系 → 本 skill；分析数据 / 出问卷 / 测可用性 → 隔壁。**
  - 手里**已经有**满意度/NSS/NPS 分数、要诊断"先改哪、为什么掉、上线涨没涨" → `analyze-satisfaction`。本 skill 是更前一步：**还没指标，先把体系和口径建起来**；建好后采集到的数据，正是 analyze-satisfaction 的输入。
  - 要把某个维度**落成可投放的问卷题目** → `generate-survey`。本 skill 产出"用哪个量表、口径是什么"，成稿问卷（封面信/题目/跳转）走那边。
  - 要一轮**可用性测试执行包**（任务脚本、现场 SEQ/SUS 度量、观察记录）→ `generate-usability-test`。
  - 要的是"**B 端整条线整体研究怎么打**"的场景方法论 → 直接读 `b-end-experience-measurement` 场景正典。本 skill 正是把那套打法**落成可交付的度量体系**，二者呼应、不冲突。

## 敏感素材处理（先读再动手）

本 skill 的输入多为产品描述与业务目标（不敏感），但常夹带**内部路线图、现有埋点/调研数据、竞品基准、用户分群口径**等内部材料：

- **会话内临时处理，不写入 Wiki、不持久化。** 不要把用户提供的内部数据、路线图、竞品情报写进 `research-wiki/` 或落盘到 skill 目录。本 skill 只**读** wiki 正典、**产出**度量体系，从不回写用户素材。
- **引用现有数据/竞品时做匿名化与脱敏**（正文、采集口径、《本次运行说明》都不出现真实联系方式、订单号、个人身份信息）。涉及 PII 字段尤其当心，既是隐私要求，也避免触发内容网关拦截。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把 HEART 五维定义、GSM 步骤、基准值、量表题干/计分、纵向追踪协议、B 端频控规则、标准化公式写死在这里或 references 里**。每次都从当前 wiki 现读，确保始终是最新版——正典可能已从 draft 改为 reviewed、基准或口径可能已修订。
2. **产出末尾固定附《本次运行说明》**，让使用者知道这套体系建立在什么之上、哪里可能不牢。结构见 `references/metrics-system-skeleton.md`，六个部分一个都不能少。
3. **声明局限，但集中归口、不淹没体系（分层交付）**：基于 draft（未评审）正典的部分、因当前无埋点/无基线而暂时落不了地的指标、因未找到贴合场景而用通用框架兜底的部分、用标准量表必须对标百分位而非绝对值的部分，都要交代清楚、别让使用者误以为处处有正典背书、或误以为指标已经可落地——但**集中写到《本次运行说明》**；体系主体只在真正影响某指标能否落地时留一句轻量 ⚠️（呼应北极星「分层交付」）。
4. **缺内容不编造；"未找到"必须先核验。** 读不到的正典、空目录、走空链接——标「未找到 / 待补充」，绝不凭空生成看似正典的维度定义、看似标准量表的计分、冒充本库的基准值或频控数字。**反过来同样重要：写下任何"未找到 / 走空链接"之前，先用 Glob 或 Read 实际核验该路径确实不存在。** 把真实存在的正典误报成缺失和编造内容一样有害。本环境 bash 的 `ls`/`find` 可能不可靠，**核验一律用 Glob/Read，不要用 shell 列目录**。度量体系会真的拿去埋点、定 KPI、向上汇报，宁可少给、也不可拿假指标误导。

## 第 1 步：解析 brief

从用户输入里抽出五个要素，决定后面读什么、体系长什么样：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **产品/系统 + 形态（C 端 / B 端）** | 决定选哪些维度、北极星候选、拆解方式。**B 端**（采销/供应商/中后台等"不得不用"的系统）侧重满意度/费力度/任务效率/性能，**别照搬 C 端活跃类指标**；**C 端**功能相近、差异化更重要，侧重满意度/NPS/忠诚度 | 完全缺失 → 先问清，这是不能假设的核心 |
| **业务目标（核心目标 + 护栏底线）** | GSM 的源头 G：决定信号与指标，区分"要提升的核心"与"不可恶化的护栏"（如"提升采纳"且"任务成功率不下降"） | 完全缺失 → **先问清**，没有目标牵引的指标体系等于乱堆数，这是不能假设的命门 |
| **产品阶段** | 冷启动/探索期→不堆多指标、借机做定性发现为主；深耕/打磨期→完善指标体系、做埋点取数；突破期→长期数据洞察趋势。阶段决定体系的详略与"先主观后客观"的节奏 | 缺 → 按形态/描述推断（新产品→探索期；成熟产品迭代→深耕期）并在运行说明声明 |
| **可用数据源 / 采集条件** | 有无行为埋点日志、能否投问卷、有无历史基线/竞品基准、有无后台标签可下钻——决定指标可落地性、采集方式与能否设基准 | 缺 → 标注"建议补埋点/基线"，并按 HEART 从 0 到 1 经验**优先给主观指标方案**（一期多以 2–3 轮访谈 + 大样本问卷拿满意度/易用性，客观指标二三期再系统化） |
| **是否跨版本 / 跨平台 / 跨人群比较** | 决定要不要配**纵向追踪方案**（口径冻结，第 4 步）、要不要做**数据标准化**（量纲不一/跨平台/客观指标赋分时的前处理） | 缺 → 默认体系将用于"长期跨版本追踪"（度量体系的常态），配纵向追踪方案并声明；跨平台/赋分需求等用户确认后再加标准化 |

判断原则：**核心缺了（产品、目标）就问，次要参数缺了就合理假设 + 声明**——不要因为缺一个产品阶段或数据源就卡住不动，也不要在产品或目标都没有时硬编一套体系。

## 第 2 步：读正典（本 skill 的核心——cross-walk）

按下表读 wiki，**只读相关的**，带着「我要从这抽什么填进度量体系」的目的读，而不是通读。先快速看相关 `index.md`（`methods/toolbox/analysis/index.md`、`assets/` 下索引、`methods/scenarios/index.md`、`methods/standards/index.md`）确认条目与各条 `status`，再决定打开哪几篇。读完记下每篇相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于度量体系 |
|---|---|---|

## 第 3 步：搭建度量体系主体（GSM → 维度 → 北极星/护栏 → 采集口径 → 拆解树 → 采集方案）

骨架与各段写法见 `references/metrics-system-skeleton.md`，**起草时打开它**。按 HEART/GSM 与 B 端打法做，顺序就是 GSM 的顺序——先目标、再信号、最后指标：

1. **GSM 推导链**：把第 1 步的业务目标写成清晰的 **Goal**，为每个目标找能反映进展的 **Signal**（可观测的行为或态度，优先选对体验变化敏感、不易被刷的），再落成口径清晰的 **Metric**。这条 G→S→M 链是整套体系的脊梁，必须显式写出来，别跳过。
2. **选 HEART 维度**：按目标挑 **3–4 个**真正相关的维度（Happiness/Engagement/Adoption/Retention/Task success），**不五维硬凑**；每维标清是"**要提升（核心）**"还是"**要守住（护栏）**"。**C 端/B 端选型要分清**：B 端用户"不得不用"，参与度/活跃度不反映体验，优先满意度/费力度/任务成功/性能；C 端侧重满意度/NPS/忠诚度。
3. **定北极星指标 + 护栏指标**：
   - **北极星 = 一个全局体验指标**，定方向。B 端在 b 端正典的**四候选**（满意度/费力度/继续合作意愿/推荐意愿）里按场景选一个并说明理由；C 端常用满意度/NPS。
   - **护栏 = 不可恶化的底线**。重点给那些"猛拉北极星会暗伤"的维度配护栏（HEART 明确举例：猛拉采纳可能拉低愉悦度/留存）——逐条写明"守住什么、跌破即预警"。
4. **写采集口径表（看板雏形）**：逐指标一行，写清 **G/S/M、口径（分子分母 / 统计周期 / 样本来源）、量表与计分、基准/目标值、核心 or 护栏**。愉悦度/任务成功/忠诚维度**直接取标准量表口径**（SUS/SEQ/NPS/CSAT/CES/UMUX-Lite），并写明**对标百分位/基准而非看绝对值**（SUS=68、SEQ≈5.5）。客观指标无固定满分的，按数据标准化的**基准值测定/赋分**思路给折分方案。
5. **拆解度量体系（拆解树）**：用 b 端正典的两法之一把北极星拆到可定位问题的颗粒度——**因子分解法**（子系统相互独立时，逐层拆满意度）或**客户旅程法**（任务导向时，按"首页→搜索→…→评价"等触点拆、任务结束即测）。拆解树是后续"指标涨跌了，问题出在哪个模块/触点"的定位地基。
6. **采集方案**：选采集方式（**问卷调研**全面但回收率低 / **场景化弹窗**即时但维度单一 / **可用性测试**系统但成本高），给最小题型单元（**体验评分 + 不满意原因 + 开放题**）；**B 端必配频控三层**（单题 1–3 次/周期、全局每人每天≤1 份、填答累计≤3 次）；按抽样规范/ b 端公式给**最小样本量**估算（95% 置信、3% 误差，总体比例未知设 50%）；**跨平台/跨量纲/客观指标赋分**时，标注需先做数据标准化前处理。

> **边界归口**：任一处基于 draft、当前无埋点/无基线落不了地、或无贴合场景而通用兜底，集中写到《本次运行说明》（呼应纪律 3 的分层交付），体系主体只在真正影响该指标落地时留一句轻量 ⚠️。

## 第 4 步：设计纵向追踪方案（让体系能"持续怎么量、跟谁比"）

**度量体系的价值建立在"持续追踪、可对标"之上**——单期数据没有参照系等于没有意义。除非用户明确只要一次性快照，否则**实时读 `longitudinal-benchmark-tracking.md`** 配一套纵向追踪方案：

- **固定关键任务集**：从最重要的任务里选约 **5–10 个**，行为类与态度类各覆盖，作为每个版本都原样重测的对象。
- **口径冻结**：任务表述、量表题项、人群与设备口径一旦定好就**冻结**——这是趋势线成立的前提，宁可指标不完美也不中途改口径（改了就断了与历史的可比性）。
- **立基线**：用历史版本立基线；**首次无历史数据，就先测竞品 / 行业基准 / 利益相关者目标**（如"表单填写 < 3 分钟"），立起第一条参照线。
- **采集时点**：改版后**留 2–3 周（高频产品）/ 4–5 周（低频产品）再测**，剥离变化厌恶；反向也别在上线 D1 新鲜感峰值取数。每波次**登记外部干扰因素**（营销活动、季节、大促、经济波动）入附录。
- **指标分层**：把指标按**领先（任务成功/难易感受）/ 同步（任务满意度/耗时）/ 滞后（NPS/整体满意度/SUS）**分层呈现，便于区分早期信号与长期价值。
- **追踪频率与（可选）ROI**：定多久重跑一波；需要向上证明价值时，把 UX 指标与组织 KPI（客服成本/营收/线索）挂钩换算 ROI。

> 用户若明确"只要这一版的快照、不追踪"，则跳过本步并在运行说明声明"未配纵向追踪方案（用户仅要单期快照）"——但要提示：无基线的单期分数难以解读。

## 第 5 步：指标质量自检（拿"指标好坏"的规范回头扫一遍）

**实时读 `experience-metrics-heart.md` 的「局限与误用」+「好/坏指标」两节、并借 b 端正典「该场景特有注意点」**，对刚搭好的体系**逐条**回扫——这是实质修改、不是走过场，是本 skill 区别于"随手列几个指标"的关键一环：

- **多维校验**：是否至少覆盖"**态度(H) + 行为(E/A/R) + 任务(T)**"三类？只剩单维 → 补，防单指标误导决策。
- **北极星唯一 + 护栏配对**：北极星是不是**一个**全局指标？有没有为"猛拉 X 会暗伤 Y"配上护栏（采纳↔愉悦度/留存）？
- **信号抗操纵 + 好/坏指标**：信号是否易被刷量、与体验弱相关？指标是否踩中**虚荣指标**（阅读数式好看无用）/ **后验指标**（只能复盘如活动 ROI）/ **复杂指标**（活跃率没界定日/周/月活）？尽量用**比率**而非裸量。
- **口径冻结性**：分子分母/周期/样本来源是否写清？能否在下个版本**原样重测**（跨版本可比）？
- **预留归因入口**：是否挂了下钻维度 / 后台标签（身份角色、高/低频、产品模块），以便指标涨跌时能定位"**谁在哪个模块**"？
- **维度/指标做减法**：有没有五维硬凑、核心追踪指标堆超过 4 个？
- **C/B 端选型**：B 端有没有误用活跃类指标？是否优先了满意度/费力度等主观体验指标？
- **采集不打扰 + 够量**：B 端**频控三层**在不在？**最小样本量**算了没？
- **量表对标**：愉悦度/任务成功是否用了**标准量表 + 对标百分位/基准**（SUS=68、SEQ≈5.5），而非看绝对值、把问卷分当百分比？

逐条给 ✅ 通过 / ⚠️ 存疑 / ❌ 不通过 + 证据或怎么改，把改动落回体系正文，并把自检结果作为交付的一部分输出（见 skeleton）。若依赖的正典为 `status: draft`：自检照做，但在自检结果与运行说明里声明"基于未评审正典，定稿后需复核"。

## 第 6 步：组装交付

按 `references/metrics-system-skeleton.md` 的顺序输出：

1. **brief 研判小结**（产品形态 C/B 端、阶段、核心目标+护栏、可用数据源、是否跨版本比较、能落地到什么程度的采集可行性声明）。
2. **体验度量体系**（GSM 推导链 · HEART 维度选取 · 北极星+护栏指标 · 采集口径看板 · 拆解树 · 采集方案/频控/最小样本量）。
3. **纵向追踪方案**（固定任务集 · 口径冻结 · 基线 · 适应期再测 · 领先/同步/滞后分层 · 外部干扰登记 · 频率/ROI）。
4. **指标质量自检结果**（对照"好/坏指标"与 HEART 局限逐条 ✅/⚠️/❌ + 证据）。
5. **《本次运行说明》**（六部分，见骨架，逐项填实，主文档最后一节）。

局限归口，不逐句夹进体系里：基于 draft 正典的部分、当前无埋点/无基线落不了地的指标、用标准量表必须对标百分位的部分、无贴合场景而通用兜底的部分——集中写到《本次运行说明》，体系主体只在真正影响该指标落地时留一句轻量 ⚠️。

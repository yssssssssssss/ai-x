# Code Open Feedback — 海量短反馈 → 编码标签库 + 主题频次/情感 + Top 痛点（编排 wiki 正典）

吃一批海量、非结构化的用户短反馈（评价 / 工单 / 开放题 / 社媒），产出**这一批专用**的结构化分析。本 skill 不发明分析方法，而是**实时编排**用研知识库（`research-wiki/`）里的 VOC 用户之声分析正典——文本分析流水线、标签体系模板、洞察输出模板、主题分析六步——按本批数据的来源与目标，把海量原话转成可量化的编码标签库、主题频次/情感表与 Top 痛点清单。

## 北极星与边界（先理解，再动手）

- **编排，不重造。** VOC 的流水线、标签体系、洞察输出模板的唯一出处在 wiki 正典里。你的工作是按本批反馈**读对正典、照它的流水线做、把结果填进产出骨架**，而不是凭记忆复述「反馈分析该怎么做」。产出的价值在于贴合 *这一批* 数据的真实来源与主题分布，不是又生成一张通用模板。
- **先圈范围，再编码。** 当这批反馈是冲着**某个具体产品 / 页面 / 功能**来分析时（如"活动首页""某次改版"），务必**先界定范围、把无关记录过滤掉，再编码**——否则整表噪声会稀释结论，让人一眼觉得"反馈和目标的关联度低"。过滤要**透明**（纳入 / 剔除各多少、按什么口径剔、剔除项仍留痕可复核），范围边界拿不准时**先与用户确认**，别自作主张地宽窄。若这批数据本身就是全范围（没有更小的目标透镜），跳过此步、直接全量编码即可。
- **结论先行，过程后置。** 产品 / 设计 / 运营第一眼要看到的是"**有哪些具体问题、多严重、该怎么办**"，不是编码本。交付按"**关键问题摘要 → Top 痛点 → 主题表**"先行；编码标签库、编码流程、《运行说明》这些**方法论痕迹收进后置附录**。把编码本铺在开头，等于用工作稿糊了业务方一脸——这是本 skill 最常见的返工来源。
- **量化收口，但只是计数。** 频次、占比、占负面% 是对编码结果的简单计数——它让「先修哪个」有据可依，但**不是统计显著性检验**。需要显著性、平台优劣势这类带检验/建模的计算，不在本 skill 范围（见下方边界）。
- **定性 + 定量两条腿走路。** VOC 最值钱的是「为什么」。只给频次不给原话，就丢掉了可读性与说服力；每条 Top 痛点必须配真实代表原话，把抽象的占比还原成有血有肉的故事。
- **一套坏标签污染整批数据。** 标签无定义、口径不一致，统计就失真。必须先沉淀一份带定义和正反例的**编码本（codebook）**，再据它打标——这是本 skill 区别于「随手归个类」的关键一环。
- **边界：海量短反馈打标 → 本 skill；其它走隔壁。**
  - 结构化问卷**交叉表 / 列联表**要做显著性、平台优劣势对比 → `survey-crosstab-analysis`（本 skill 只计数、不做检验）。
  - 少量**访谈逐字稿**要做深读、三进程编码与意义诠释（小样本、定性纵深）→ 那是 `qualitative-coding` 的活，不是本 skill。判据：**海量、被动、自发的短文本**用本 skill；**深度对话逐字稿**用 qualitative-coding。
  - **设计**问卷 / 访谈题目（采集阶段，手里还没有数据）→ `generate-survey` / `generate-interview-guide`。

## 四条纪律（本 skill 必须遵守，产出据此被信任）

1. **数据源 = 当前 master 版 wiki，运行时实时读取。** 本 SKILL 只告诉你「去读哪些路径、抽什么」，**绝不把 VOC 流水线、标签体系模板、洞察输出模板、主题分析六步写死在这里或 references 里**。每次都从当前 wiki 现读，确保是最新版——正典可能已从 draft 改为 reviewed、模板可能已扩充。
2. **产出末尾固定附《本次运行说明》**，五部分齐全（结构见 `references/coding-output-skeleton.md`），让使用者知道这份分析建立在什么之上、哪里可能不牢。
3. **在交付物正文里就地声明局限**：基于 draft（未评审）正典的部分、因未找到贴合内容而用通用方法兜底的部分、因数据量过大而只覆盖了部分样本的部分，都在该处直接标注，别让使用者误以为处处有正典背书、或误以为是全量分析。
4. **缺内容不编造。** 读不到的正典、空目录、走空的链接——标「未找到 / 待补充」。**更要紧的是不无中生有地造数据**：频次、占比、情感分必须来自对真实反馈的编码统计；代表原话必须是这批反馈里**真实出现过**的句子，绝不杜撰。分析结论会真的拿去排修复优先级、分派责任团队，宁可少给、也不可误导。

## 敏感素材处理（本 skill 特有，先读再动手）

研究员上传的评价 / 工单 / 会话 / 开放题常含个人信息（姓名、电话、订单号、住址、账号等）。处理原则：

- **会话内临时处理，不落 wiki、不持久化。** 这批反馈是用户原始数据，只在本次分析里用，**绝不写进 `research-wiki/`** 或任何长期文件。
- **产出与代表原话做匿名化。** 引用代表原话时，删掉与痛点无关的个人信息（把「我尾号 1234 的号收不到验证码」写成「我（的号）收不到验证码」），在**不损失痛点含义**的前提下脱敏。
- 若用户要把脱敏后的逐条打标结果存档，存到**本次工作目录**下的文件并提示用户，不要混进知识库。

## 第 0 步：定位 wiki 根目录

本 skill 安装后可能脱离仓库运行，先确认能读到正典：

- 约定：下文所有正典路径都**相对 wiki 根**（即 `research-wiki/` 那一层，其下有 `methods/ models/ assets/`）。
- 定位顺序：① 看当前工作目录下或其内是否有 `research-wiki/`；② 若已在该树内，向上找到含 `methods/ models/ assets/` 的那层；③ 不确定时用 glob 搜锚文件 `**/methods/toolbox/analysis/voc-analysis.md`，其所在的 `.../research-wiki/` 即根。
- **找不到 wiki**：不要硬编一套流程糊弄。如实告诉用户「未能访问 research-wiki 正典」，请其确认知识库位置后再跑；确需先给草稿时，明确声明「未基于正典、仅通用 VOC 经验」，且**不得**伪造「依据 VOC 第 X 步」之类的说法。

## 第 1 步：接收并研判这批反馈

落标签前先搞清三件事，决定后面怎么读、怎么编码：

| 要素 | 作用 | 缺失怎么办 |
|---|---|---|
| **来源类型** | 评价 / 工单 / 开放题 / 社媒 / 会话…决定主动 vs 被动、噪声水平、要不要先去广告水军 | 缺 → 看文本特征推断，并在运行说明声明 |
| **分析目标** | 常态化扫描？某次改版复盘？客诉激增定位？决定以**演绎**（套已知分类）还是**归纳**（长新主题）为主 | 缺 → 默认「常态化扫描 + 归纳为主」，声明 |
| **分析范围 / 透镜** | 是不是冲着**某个具体产品 / 页面 / 功能**来看的（如"活动首页"）？决定要不要**先过滤再编码**、以及主题主轴该按什么切 | 缺 → 先问清"这次是看整批、还是看某个点"；用户已给透镜就据它划范围 |
| **可用元数据** | 渠道 / 时间 / 版本 / 评分——有则能做趋势对比与按版本下钻；没有则只能出截面分布 | 缺 → 只出截面，注明「无时间/版本元数据，趋势待补充」 |

落标签前先做三个判别：

- **这批数据是不是本 skill 该接的？** 若其实是几篇访谈逐字稿要做深读 → 转 `qualitative-coding`；若是结构化交叉表 → 转 `survey-crosstab-analysis`。判错了，后面分析得再漂亮也是错的。
- **要不要先过滤到目标范围？**（呼应北极星"先圈范围，再编码"）当有明确透镜（某产品/页面/功能）时，**先界定"什么算相关"、把无关记录滤掉，再编码**：靠已有分类列 + 关键词 + 语义判断划定纳入/剔除口径，**把纳入多少、剔除多少、按什么剔说清楚**，剔除项留痕可复核。边界拿不准（比如某个模糊类目算不算"活动首页"）**先与用户确认**，别默默宽窄。范围本就是全量时，跳过。
- **数据量能不能完整处理？** 量太大无法逐条可靠编码时，**不要假装跑了全量**：说明你覆盖了多少条、是全量还是抽样、抽样怎么抽，把这点写进运行说明并在正文标注（呼应纪律 3 / 4）。

## 第 2 步：读正典（本 skill 的核心——照 VOC 流水线做）

按下表读 wiki，**只读相关的**，带着「我要从这抽什么填进产出」的目的读，而不是通读。读完记下每篇相对路径与其 frontmatter 里的 `status`，运行说明要列。

| 读什么（相对 wiki 根） | 何时读 | 抽什么用于分析 |
|---|---|---|
| `methods/toolbox/analysis/voc-analysis.md` | **必读（主流程）** | 完整文本分析流水线（清洗→分词归一→编码打标→主题聚类/建模→情感→趋势）；**标签体系（编码表）模板**（建编码本用）；**洞察输出模板**（做主题表用）；主题分析（Braun & Clarke）六步与演绎/归纳之分；主动 vs 被动 VOC 来源清单；「结果解读 / 局限与误用」（自选择偏差、被高声量小众带偏、跨渠道要统一归集） |
| `methods/toolbox/analysis/voc-product-line-classification.md` | 反馈是**京东 App / 产品**相关、`模块`维度要打准时（应用商店评价/工单/应用内反馈大多如此） | 京东**封闭模块词库**（产品/服务 × B2C/O2O × level3 模块）+ 11 个**体验维度** + **范围×严重度→A/B/C 优先级** + **防劫持路由规则**。用它把"模块"维度归得准、不臆造、跨界正确归属——直接提升编码质量。**非京东反馈则跳过，用上一行的通用模块维度兜底。** |
| `methods/toolbox/analysis/affinity-diagram.md` | 要把零散 code **自下而上聚成主题**、或归纳发现未预设主题时 | 其归并逻辑：初始不打标 → 按相似纵向成列 → 写 level1/level2 标签 → 复查。借它把开放式编码产出的 code 聚成主题（注意它本是线下团队协作法，这里只取其归并思路） |
| `methods/toolbox/analysis/text-analysis.md` | 可选**概念参考** | 文本技术本体（分词、TF-IDF 区分性关键词、情感语料库、语义网络）与工具（jieba / KH Coder）。它**无产出模板，主流程以 VOC 为准、不依赖它**；当你要解释「为何高频词≠关键词」这类概念时再看 |
| `methods/toolbox/analysis/issue-prioritization.md` | 可选**下游** | 若用户还要把 Top 痛点正式排优先级（RICE/ICE 等），抽其排序口径；否则本 skill 只按「高频 × 高负面」给清单 |
| `methods/toolbox/analysis/index.md` | 起步快览 | 确认分析技术清单与各条 status，避免空读；正典 `related:` 指向不存在的文件 → 记为「走空链接」写进运行说明 |

> `qualitative-coding.md` 一般**不读来做分析**——它只在第 1 步帮你判边界（输入是不是逐字稿、该不该转给它）。

## 第 3 步：建编码体系 + 编码打标

骨架见 `references/coding-output-skeleton.md`，**起草时打开它**。照 VOC 的流水线做：

1. **清洗预处理**：去重、剔除广告 / 水军 / 与主题无关的内容；按来源做必要的同义归一（呼应 VOC「清洗与预处理」）。**剔了什么、剔了多少**，记一笔备查。
2. **建 / 复用编码本**：用 VOC 的**标签体系模板**搭一份带**定义 + 正反例**的编码本——典型是「一级模块 × 问题类型 × 情感 × 严重度 × 来源渠道」这种多维结构（**维度与取值以运行时读到的 VOC 模板为准**，按本批数据增删）。先**演绎**套用已知分类，再**归纳**补充新主题。编码本是多人 / 多批打标口径一致的前提，必须先立、不能边打边随意改义。
   - **京东 App / 产品反馈**：`模块`维度优先改用 `voc-product-line-classification.md` 的**封闭模块词库**（产品/服务 × B2C/O2O × level3 模块），`问题类型`改用其 **11 个体验维度**，严重度/优先级可用其 **范围 × 严重度 → A/B/C** 口径，并按其**防劫持路由**打标——模块归得准、不臆造、跨界正确归属，编码质量更高、口径更稳。**非京东反馈仍用 VOC 通用模板兜底。** 取用了就在运行说明①列出该正典与其 status。
3. **逐条打标**：对每条反馈打「模块 + 问题类型 + 情感（+ 严重度）」标签。一条里「功能挺好但加载太慢」含两个相反方面 → 可拆成两个方面分别标（aspect-based，VOC 进阶项）。**逐条结果不要在正文铺开**——汇总成一张「逐条编码明细」表，按数据规模放进**附录**或导成**独立文件**（位置与分流规则见第 5 步「附录·逐条编码明细」）。
4. **聚类成主题**：用 affinity-diagram 的归并思路把零散 code 自下而上聚成主题；演绎时套预设分类，归纳时让数据长出未预设的新主题（往往正是新引入的体验问题）。

> 这一步是 LLM 的**语义判断**，不是跑公式。打标口径要自洽：同一类反馈每次都归到同一标签。打标量大时，先在小样本上把编码本试稳，再铺开全量。

## 第 4 步：量化 + 主题表 + Top 痛点

照 VOC 的「量化与趋势分析 / 生成洞察清单」做，把结果填进骨架：

- **主题表**：按 VOC **洞察输出模板**的字段逐主题出表（典型含 主题 / 频次 / 占负面% / 趋势 / 情感分 / 代表原话——**列以运行时读到的模板为准**）。**只有拿到时间 / 版本元数据时才填「趋势」**；没有就标「无元数据 / 待补充」，别编趋势。把模板里的"责任团队"这一列**拆成产品 / 设计 / 运营三方职责**（见下），并**加一列「机会点」**。
- **每条痛点配三方职责 + 机会点**（本 skill 的落点，别省）：
  - **产品 / 设计 / 运营三分**——同一个问题往往三方都有份，别笼统写一个"责任团队"：**产品**=功能能力/逻辑/流程/权限；**设计**=信息架构、入口可发现性、操作路径与效率、状态反馈、事前告知与可解释性、文案；**运营**=规则制定、审核时效、活动力度、客服解决率与告知触达。**设计视角最容易被漏**——"找不到入口""退了没反馈""静默失败没提示""客服复制粘贴话术"这类都是设计问题，务必单列出来。
  - **机会点 = 针对这个问题能做什么**：一句话给出可落地的改进方向（如"已报管理支持一键/批量退出+状态可视"）。机会点是**基于反馈的改进建议、非已验证结论**——落地/交互类建议标"待原型/灰度验证"。
- **Top 痛点清单**：按「高频 × 高负面（× 影响面）」排序，每条配 **2-3 句真实代表原话**（已脱敏）+ 三方职责 + 机会点。提醒：**高声量小众诉求 ≠ 多数人核心痛点**，要结合占比和影响面判断，别被极端情绪带偏（VOC「局限与误用」明确点了这条）。
- **计数即计数**：所有频次 / 占比都是对你编码结果的统计，**标清分母**（占总反馈、还是占负面反馈）。**不做显著性**；用户若要显著性 / 平台对比，指给 `survey-crosstab-analysis`。

## 第 5 步：组装交付（结论先行，编码过程后置）

按 `references/coding-output-skeleton.md` 的顺序输出。**交付稿在前当正文、编码底稿在后当附录**——产品/设计/运营先看到结论，要追溯的人再翻附录：

**正文（结论先行）**
1. **分析范围与过滤**（做了范围过滤才写）：一句话讲清"活动首页=什么"、纳入多少 / 剔除多少 / 按什么口径剔——让读者信任后面的分母。全范围就省略。
2. **关键问题摘要（TL;DR）**：开篇就给"第一眼要看的几个具体问题"——一张扫读表（问题 / 规模 / 不满率 / 主责产·设·运 / 机会点一句话）+ 一句话主线 + 最关键变量结论。
3. **Top 痛点清单**（高频 × 高负面排序）：每条配脱敏代表原话 + 产品/设计/运营三方职责 + 机会点。
4. **主题表**（按 VOC 洞察输出模板逐主题：频次 / 占负面% / 趋势 / 情感 / 主责产·设·运 / 机会点）。
5. **关键变量 / 下钻分析**（当范围内有可切分维度时）：哪个变量最集中地聚集了不满；次要维度（如用户分群）作方向性观察，标"计数非显著性检验"。

**附录（方法论后置）**
6. **附录·编码标签库**（带定义 / 正反例的结构化标签体系）——这是编码口径，放附录供审稿，别铺在开头。
7. **附录·逐条编码明细**——按数据规模 N 分流：
   - **N < 30**：附录里出一张「逐条编码明细」表，每条一行（列含 # / 原话节选 / 模块 / 问题类型 / 情感 / 严重度；京东场景加 level1-2 / 优先级）。
   - **N ≥ 30**：**不放进文档**，单独导出到本次工作目录下的 `coding-detail.csv`（或 `.md`），文档此处只留一句指引「逐条明细见 `coding-detail.csv`，共 N=… 条」。做了范围过滤时，csv **保留全量并加"是否相关+剔除原因"列**，便于复核边界。
8. **《本次运行说明》**（五部分，见骨架文件，逐项填实，**文档最后一节**）。

正文里把局限就地标注：基于 draft 正典的部分、只覆盖了抽样/过滤后范围的部分、无元数据无法出趋势的部分。

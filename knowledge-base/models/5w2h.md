# 5W2H 模型

> 一句话：用 7 个固定提问维度（Who/What/When/Where/Why/How/How much）系统化拆解一个用户行为或需求场景，把"用户说"还原成"用户在什么情境下做什么"。
> 落点：`models/`。判据——它**贯穿研究全程**：问题界定、访谈提纲、行为分析、需求定义、流程优化都用到，拆开任一环就不构成 5W2H 思维方式。
> 边界：与 `methods/toolbox/analysis/` 的差别——5W2H 不是某种"算法/出图步骤"，而是一种**思考透镜**和**信息结构化框架**，因此入 `models/`。
>
> ⚠️ 待补充(缺源)：本条目正文整理自原 Joyspace 文档；该文档中大量"在用户研究中的应用：" / "用户购买行为示例："条目下的列表内容未在导出正文中可见（疑为图片或被折叠的列表块），需 PR 时由 owner 对照原文补齐。本条目仅纳入正文中可读的内容，不做推断填充。

## 核心概念

- **5W2H 又称"七何分析法"**：以 5 个 W 开头的英文词（Who、What、When、Where、Why）和 2 个 H 开头的英文词（How、How much）系统化提问，从回答中发现解决问题的线索。
- **核心思想**：通过追问获得项目背景和业务目标，**强迫分析者从多个维度全面考察问题，避免思考盲区**。
- **场景化思维公式**：

  > 某种类型的用户（**Who**）在某时间（**When**）某地点（**Where**），因为周围出现了什么事物（**What**）而产生某种需求（**Why**），并希望通过某种方法（**How**）来满足这种需求，投入的成本是（**How much**）。

## 理论来源

- 起源：1932 年由美国人提出，后经不断运用和总结，发展为企业管理、产品设计、用户研究领域通用的分析框架。
- 本条目整理自：[Joyspace - 5W2H情境洞察用户需求研究知识文档](https://joyspace.jd.com/pages/FtUlxMV0NQUAYxhx1fW4)（李笑欣）。

## 关键构念 / 维度

| 维度 | 核心问题 |
|---|---|
| **Who** | 目标用户是谁？需求的提出者、使用者、影响者分别是谁？ |
| **What** | 用户要完成什么任务？需要解决的问题或实现的目标是什么？ |
| **When** | 需求何时发生？用户在什么时候使用产品/功能？ |
| **Where** | 用户在哪里使用产品？需求在什么地点/场景下产生？ |
| **Why** | 用户为什么产生这个需求？问题产生的背景和根本原因是什么？ |
| **How** | 用户如何完成目标？如何解决问题？具体执行方式是什么？ |
| **How much** | 用户的消费金额？解决问题的成本/效益？做到什么程度？ |

**关键原则（来自原文）**：**「为什么」比「是什么」更重要。** 仅关注"是什么"是舍本逐末；只有知道"为什么"才能有效判断流程节点的必要性，避免盲目照搬现有流程。

## 如何作为透镜使用

### 问题定义阶段

- 用 5W2H 把模糊的研究问题（"为什么用户流失"）拆成 7 维度子问题，识别哪些维度数据已有、哪些缺失。
- 针对缺失维度补设研究动作。

### 采集阶段（最直接的应用）

5W2H **可以直接作为访谈提纲框架**，确保覆盖全维度信息：

| 访谈问题 | 5W2H 维度 | 示例 |
|---|---|---|
| 你是做什么工作的？ | Who | 了解职业背景和使用场景 |
| 你通常什么时间使用这款 App？ | When | 发现使用时机规律 |
| 你在什么情况下会打开这个功能？ | Where/When | 还原使用场景 |
| 你用这款 App 做什么事情？ | What | 发现核心任务 |
| 你为什么选择这个而不是其他方式？ | Why | 揭示动机和偏好 |
| 你通常怎么完成这个任务？ | How | 了解行为路径 |
| 你愿意为这个功能付多少钱？ | How much | 了解价值感知 |

### 分析 / 结论阶段

- **从场景出发的需求发现逻辑**：传统需求收集容易陷入"用户说什么就做什么"的陷阱。5W2H 通过情境还原，帮助分析者跳出表层主诉，定位真实需求与机会。
- **情境化需求描述公式**：用 5W2H 把低质量的需求描述（"用户希望能更快地找到商品"）改写成有 Who/When/Where/Why 的具体情境化描述。
- **从 5W2H 到需求定义**：完成 7 个维度的情境还原后，需进一步做"需求升华"——⚠️ 待补充(缺源)：原文未给出升华的具体步骤，需 owner 补。

### 应用场景速查

- **用户行为分析框架**：用户购买行为分析、用户流失分析（如餐饮业流失分析）。
- **产品经理需求筛选/排优先级**：面对大量需求时按 5W2H 7 维度评估，B 端需求筛选实践。
- **用户访谈提纲设计**：见上表。
- **业务流程梳理与优化**：检验每个流程节点的必要性。
- **活动策划**（如拉新活动）。

## 与方法 / 分析的衔接

- **5W2H 与情境洞察（Contextual Inquiry）**：情境包含四个核心要素⚠️待补充(缺源)；5W2H 中的 **Where + When + Who** 三个维度直接映射情境洞察的核心要素，是情境调研结构化分析的天然框架。
- **洞察链路**：

  ```
  用户行为观察
   ↓ Who（谁在做）+ When（什么时候）+ Where（在哪里）
   ↓ What（做了什么）+ Why（为什么这么做）
   ↓ How（怎么做的）+ How much（付出了多少努力/成本）
   ↓ 洞察：真实需求 + 痛点 + 机会点
  ```

- **5W2H 是问题诊断工具**：输出的是"场景和洞察"，**不是最终答案**——还要结合 KANO、Y 模型等做解决方案设计和优先级决策。
  - 见 `models/kano.md`（待建）。

## 局限与误用

> 五个误区，全部直接来自原文。

1. **停留在"是什么"，不追问"为什么"**——Why 是 7 维度中最容易被忽略但最有价值的，只收表面行为数据无法发现真实需求。
2. **把 5W2H 当检查清单机械填写**——它是一种情境化思维方式，不同场景各维度权重不同，要灵活运用。
3. **混淆不同用户角色的数据**——不同 Who（新用户/老用户）的 When/Where/Why 往往完全不同，必须分别建立情境。
4. **只做定性不做定量（或反之）**——最佳是定性 + 定量结合：定量回答 How much，定性回答 Why。
5. **把分析结果当最终答案**——5W2H 是诊断工具，需结合 KANO、Y 模型等做方案与优先级。

## 实操步骤（直接来自原文）

| Step | 动作 |
|---|---|
| 1 | **明确分析目标**：确定本次 5W2H 要回答的核心问题（如"为什么用户在结算页流失"） |
| 2 | **收集多源数据** |
| 3 | **按 7 个维度逐一填写**：建立 5W2H 分析表，每个维度填已知信息和待验证假设 |
| 4 | **识别信息空白**：标记哪些维度信息不足，设计补充研究方案（追加访谈/数据分析） |
| 5 | **还原情境场景**：将 7 维度信息整合成完整的用户情境叙述（用户故事形式） |
| 6 | **提炼需求洞察**：从情境叙述中识别真实需求/痛点/机会点 |
| 7 | **输出分析结论**：形成有数据支撑的需求报告，每条需求附带完整 5W2H 场景说明 |

# 用户需求的本质（三层模型）

> 一句话：**用户需求不是"用户说了什么"或"用户抱怨什么"，而是在特定场景下，用户为达成目标而必须被满足的任务、动机与结果。**
> 落点：`models/`。判据——这是用研最底层的概念透镜，贯穿"问题定义→采集→分析→结论"全程，且与各种采集/分析方法的结论解读绑死，拆开任一环就讲不通。
> 边界：JTBD 单独成篇 → `models/jtbd.md`（待建）；KANO 把需求按强度分基本/期望/兴奋型 → `models/kano.md`（待建）；本篇是**用研者拉齐"什么是用户需求"基本理解**的总论性透镜。

## 核心概念

> 用户需求 ≠ 用户说什么；用户需求 ≠ 用户抱怨什么。
> 用户需求 = **特定场景下，用户为达成目标而必须被满足的任务、动机与结果。**

启动"需求挖掘"类项目前，团队需先**拉齐对"用户需求"的理解**，规避常见的理解误区，基于对"用户需求"的本质来进行后续的洞察分析。这是一切需求洞察工作的最低共识。

## 关键构念 / 维度

### 三条核心原则

#### 1. 洞察用户需求先回归用户场景

- **场景不同，用户需求也不同**——只有看懂背后场景的细节才能完整理解需求。
- 不要直接问用户"想要什么 / 不要什么"——要先**退后一步**回归到用户真实生活，理解行为背后的**生活场景**和**需求动机**，再分析推导真正的需求是什么。

#### 2. Job To Be Done 原则

- 用户购买的不是"产品 / 服务"——他们是"**雇佣**你的产品 / 服务来完成他们需要的某种任务（Job）"，**这个任务才是用户真正的需求**。
- 站在 Job 的基础上找需求；看懂这个任务才能看懂真正的需求，才能有针对性地改进产品。

#### 3. 用户痛点不完全等于用户需求

- 痛点只说明"现状有多糟糕"——**无法告诉我们"怎么做会更好"**。
- 有时候用户需求**并不站在痛点之上**，而是站在场景和动机里。
- 看到痛点的同时要追问：这个痛点背后的**用户场景**和**真实需求**到底是什么？痛点常常只是**表象需求**，洞察到深层需求才能得出真正的解决方案。

### 三层理解（核心模型）

| 层 | 名称 | 内容 | 典型问法 |
|---|---|---|---|
| **表层** | **显性诉求** | 用户可直接表达的内容 | 行为习惯 / 决策关注 / 用户痛点 / 功能诉求 |
| **中层** | **任务场景** | 用户行为背后的任务场景 | 什么情况下发生 / 具体如何发生 |
| **深层** | **目标动机** | 任务场景背后的动机 | 用户为什么这么做 / 深层心理动机 |

- **三层都看到才算理解需求**——尤其**不能只看到表层显性诉求就认定是用户需求**。
- 必须进一步分析：显性需求背后是否有**真实存在的场景**？用户的**深层动机**是什么？
- 综合三层才能研判什么是真正的用户需求。

## 如何作为透镜使用

### 问题定义阶段

- 把研究问题改写成"用户在什么场景下要完成什么 Job"——而不是"用户喜欢哪种功能"。
- 同一个表层问题（"用户想要什么样的购物 App"）可以拆出十几个 Job，每个 Job 是一个独立研究问题。

### 采集阶段

- 访谈题目从"你想要什么 / 你建议什么"转为"行为描述题"：请描述你最近一次做 X 的过程。
- 至少要采集到三层的全部信息：用户**做了什么（行为）**+ **在什么情境下做（场景）**+ **为什么这么做（动机）**。

### 分析 / 结论阶段

- 三层逐层判断：表层显性诉求是否有中层场景支撑？中层场景背后的深层动机是否真实成立？
- 痛点出来时**追问到深层**——把"现状有多糟"翻译成"用户需要什么"。
- 把诉求标注为"个性需求"还是"普遍需求"——见下方误区三。

## 局限与误用（三个常见错误认知）

| 错误认知 | 正确认知 |
|---|---|
| **只听态度不看行为** | 用户需求不是"用户怎么说"——用户的**主动表达态度**并不等于用户需求；要分析态度背后的**行为发生场景**和场景背后的**真正目标和动机** |
| **把用户建议当用户需求** | 用户不是专业的，**不能替代产品经理**提出解决方案。听到"我想要 X / 我建议 X"时，先思考：这个建议是否有合理的场景？是表层还是深层？是真需求还是伪需求？ |
| **把个性需求当普遍需求** | 要甄别哪些需求是**过于个性化的**——受用户自己过于个性化的行为/场景影响，属于该用户**非常个人化**的需求，不能推广为普适的用户需求 |

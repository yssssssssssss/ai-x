# 用研需求分析与沟通规范

> 来源：神灯圈子·翁鸿诚《用研项目需求分析沟通中的五个"坑"》（原作者徐凯，写于 2019-05-16） 原文 http://xingyun.jd.com/shendeng/article/detail/46608

> 内部「必须遵守」的权威规则。本规范在本库**单一出处**，其他文件引用而非复制。
> 落点：`methods/standards/`。本规范约束"用研项目接到需求后，如何分析与沟通需求"这一环节。

## 适用范围

- 适用于所有用户研究项目在**立项/接需求阶段**对需求的梳理、分析与沟通。
- "需求"既是一个项目的起点，也是评价一个项目是否圆满完成的终点标准；需求分析是否清晰明确，直接影响需求优先级的确定、研究方法的选取、研究结论的方向以及研究价值的体现。
- 作为产品部门或业务部门的"乙方"，用研需将粗犷式、精细式、泛滥式的需求清晰呈现在甲乙双方之间，建立需求传递的"坦途"。下述"五个坑"即需求分析沟通中需要填平的五个关键点。

## 必须遵守（MUST）

- 本规范由经验总结沉淀而来，未规定强制性硬规则；具体做法见「推荐（SHOULD）」。

## 推荐（SHOULD）

需求的分析与沟通需要围绕五个关键点不断迭代、补充，通过反复梳理、分析、沟通、迭代，最终把一个需求完整、透彻地呈现在甲乙双方面前。下列为针对"五个坑"的推荐做法。

- 建议对信息在传递过程中的失真保持警惕：直接对接人不等于需求的真正提出者，沟通前宜先识别需求的真正提出者及关注者，不只凭直接对接人的转述就确定需求。
- 建议接到需求后不要急于对需求进行拆解；最好先在了解提出者/关注者、产品/业务现状与背景的基础上，重新审视并与需求方沟通需求的提出原因与目标，再进入拆解。

### 坑一：需求的提出者并非需求真正的提出者

一个需求普遍会有一个直接对接人，需求由直接对接人传达，但直接对接人可能并非需求的真正提出者。识别需求的真正提出者及关注者，建议从以下几个方面完成：

- 了解需求直接提出人是谁、所在部门、部门的主要职责、提出人的主要职责。
- 了解需求直接提出人的上级是谁、上级是否知晓此需求、上级的态度如何、是否是需求的真正提出人。
- 了解需求的主要利益关系人有哪些：围绕此需求的产品、设计、业务都有哪些人，他们是否知晓此需求，对需求有没有补充。

### 坑二：需求相关的产品/业务现状或背景不清

产品或业务是用户研究人员与需求方的连接桥梁；对产品/业务现状的了解，不仅影响沟通的顺畅性，而且影响研究过程的广度和深度，影响最终研究建议的可行性、建设性。建议按以下步骤梳理现有产品/业务状况：

- **首先，熟悉产品/业务。** 可对产品/业务做体验走查（不需要输出严格的体验走查报告，但建议梳理走查中看到或想到的各种问题及疑问，在后续需求沟通时做确认及解答）；也可通过相应的培训或讲解来了解产品/业务。
- **其次，了解产品/业务现状。** 对产品/业务的现有数据表现做相应了解；通过对现有数据的了解，一方面帮助分析需求的提出背景及原因，另一方面拓宽需求研究的内容，从数据中发现值得研究、对产品/业务有价值的新内容。
- **最后，沟通分析需求提出背景。** 结合现状及行业数据，以及前两步中的疑问或洞察，综合看待需求的提出背景，分析需求提出者对现状及背景的理解是否准确、在当前现状及背景下提出需求是否合理，帮助需求方提出更准确的需求或问题。

![](https://img14.360buyimg.com/ling/jfs/t28552/321/1362828585/36586/352f53b/5cdcfa38N604b702b.png)

### 坑三：需求原因/目标理解不准确

需求的原因/目标理解不准确，不仅带来资源浪费，而且易导致研究结论本末倒置、无法提供有效洞察参考的风险。同样的研究需求，从不同的提出者口中、或在不同的现状/背景下提出，主要目标多数不同。

以"老板在周会上提出需要频道页的产品找出频道页上的体验问题、优化频道页的体验"为例：

- **结合需求提出者/关注者来看**：如果提出此需求的老板是频道页产品部门负责人，需求的主要目标可能是提升用户在频道页功能应用、交互操作方面的体验；如果是频道页运营部门负责人，主要目的可能是提升频道页的订单转化率。
- **结合产品/业务现状及需求背景来看**：如果当前频道页计划进行全面改版，那么产品部门及运营部门的需求目标不仅是找出现有体验问题，还应扩展到频道页的改版方向探索上。

因此，建议接到需求时不要急于拆解，基于对提出者/关注者及产品/业务现状、背景的了解，重新审视并沟通需求的提出原因及目标，判断需求是否合理、完善、方向正确。

![](https://img30.360buyimg.com/ling/jfs/t29701/51/1338523591/33297/879e9b99/5cdcfa46N2519d580.png)

### 坑四：需求内容拆解不细致

100 分的研究不仅要让需求方得到他们想知道的内容，还要让需求方获取到他们应该知道的内容。建议从需求的**纵向和横向**两个方面做发散梳理，既保证研究的深度，又保证研究的广度：

- **纵向发散**：通过对研究目标/目的的逐层分解对研究内容做纵向发散，如按产品功能模块拆解、按业务流程步骤拆解、按更细致的人群划分拆解等。
- **横向发散**：通过对需求方角色及主要工作内容的分析对研究内容做横向发散。例如，对产品经理的需求，可从需求挖掘、功能创新、策略逻辑完善、交互流程改进、视觉效果呈现等维度发散；对运营同事的需求，可从用户群体细分，以及各群体的拉新、促活、留存、转化、营销传播等典型工作内容发散。

仍以优化频道页体验的需求为例：

- 纵向发散：频道页的产品体验可拆解为频道页导航的功能及操作体验、首页焦点图的浏览及切换体验、特色活动板块的功能及操作体验、活动内容的浏览体验等。
- 横向发散：产品部门在频道页上的主要职责是发现用户需求并把用户需求产品化，因此可将频道页体验优化需求向目标用户定义、用户潜在需求挖掘、现有功能的用户需求验证等方向横向发散。

研究内容发散并非没有边界，发散的边界在于用研的研究能力和资源。

### 坑五：方法/资源/时间规划缺少风险预见

根据前述梳理、分析、沟通，选择合适的研究方法，梳理开展研究所需的资源投入（人力资源和物质资源），预估项目工时，对项目流程及时间计划做排期，并与需求方沟通确认。这一环节容易出现的问题是对项目风险缺乏预见及应急方案。预见项目风险建议：

- 结合以往类似项目的经验审视自己的项目，假设以前项目执行中的问题在本次项目中依然会遇到，并据此设想应急方案。
- 对项目执行步骤及各步骤上的工作内容、所需资源、资源获取方式、涉及人员等做整体梳理，并设想各步骤上可能出现的问题。
- 与需求方沟通，了解他们所能预见的风险问题。

## 禁止（MUST NOT）

- 本规范为经验总结，未规定禁止性硬约束；上面「推荐（SHOULD）」中的"不只凭转述就确定需求""不要急于拆解"等为建议性提醒，并非强制。

## 检查清单

- [ ] 已识别需求的真正提出者及关注者（直接提出人及其部门职责、上级态度、利益关系人是否知晓并有无补充）。
- [ ] 已了解需求相关的产品/业务现状与背景（完成体验走查、了解现状数据、沟通确认需求提出背景）。
- [ ] 已与需求方对齐需求的真实原因/目标，并判断需求是否合理、完善、方向正确。
- [ ] 已从纵向（目标逐层分解）与横向（角色/工作内容）两个方向对需求内容做发散拆解。
- [ ] 已完成方法、资源（人力/物质）、工时与时间排期规划，并与需求方沟通确认。
- [ ] 已对项目风险做预见并准备应急方案（参考历史项目、逐步骤设想问题、收集需求方预见的风险）。

# 无障碍产品体验与适配规范

> 来源：神灯圈子·金菁《无障碍产品体验与适配规范》原文 http://xingyun.jd.com/shendeng/article/detail/59315

> 内部「必须遵守」的权威规则。本规范在本库**单一出处**，其他文件引用而非复制。
> 落点：`methods/standards/`。边界：`toolbox/` 讲「无障碍怎么做/怎么测」（知识）；本规范讲「我们的产品在无障碍模式下必须遵守什么」（规则）。

## 适用范围

- 面向**产研测团队**（产品、研发、测试、设计）的无障碍模式适配准则，用于统一无障碍体验标准、提升交付质量与一致性。
- 服务对象：**主要面向视障人士**，通过适配 App 的无障碍模式提升其信息获取与操作的便利性，使其使用体验与普通用户趋于一致。
- 适配环境：无障碍模式为手机系统内置功能——iOS 设备可通过 [设置]-[旁白]（VoiceOver）、Android 设备可通过 [设置]-[TalkBack] 启用。
- 设计理念：引导每一次需求设计融入「包容性思维（Inclusive Design Mindset）」，确保特殊群体用户亦能顺畅、便捷地完成购物体验。
- 国际标准依据：万维网联盟（W3C）[《Web 内容无障碍指南（WCAG）2.1》](https://www.w3.org/TR/WCAG21/)所提出的数字无障碍国际通用标准，结合移动 App 使用场景落地。

## 必须遵守（MUST）

### 适配总则（四大核心原则）

基于 WCAG 2.1，结合移动 App 场景制定四大核心适配总则，确保所有用户都能平等访问产品功能：

| 原则 | 设计要义 | 实现要点 |
| --- | --- | --- |
| **可感知 (Perceivable)** | 用户能感知界面信息 | ✔ 控件有名称、标签、提示音、语义描述 |
| **可操作 (Operable)** | 所有操作均可被无障碍方式触发 | ✔ 所有可交互元素有焦点、焦点顺序正确、手势可达 |
| **可理解 (Understandable)** | 内容与操作一致、反馈清晰 | ✔ 朗读准确、状态明确、状态变化有提示 |
| **可兼容 (Robust)** | 与多设备、系统做兼容 | ✔ 支持 TalkBack、VoiceOver、性能稳定 |

### 感知性要求

1. 所有可交互控件必须具备可识别**标签（Label）**，能被读屏工具识别并朗读。
2. 图片、图标必须添加描述性**替代文本（alt）**，如「商品图片：苹果 iPhone 15」。
3. 颜色不可作为区分信息的唯一手段；需同时辅以文案/图标/形状。
4. 动效可关闭或不阻断朗读；避免高频闪烁（防癫痫风险）。

**真实案例 1：控件无标签** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025450010000983615)
在商详页底部的加购按钮应添加「加入购物车」标签，以便用户**明确了解该控件的用途**，标签可朗读为「加入购物车，按钮」。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-12g9MKZM12eXBHhWoO.png)

**真实案例 2：图片需描述关键信息** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025120010001400055)
在商详选规格弹窗中，无货商品的左上角商品图片朗读时，应增加「无货」信息的朗读，使**视障用户了解商品的库存情况**。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-12tpN10S8lg122q2wax.png)

### 可操作性要求

1. 所有交互元素（按钮、输入框、切换、弹窗）必须**可聚焦（focusable）**。
2. **焦点顺序应符合视觉阅读顺序**（上→下、左→右）。
3. 键盘/物理按键可操作（如方向键、回车、返回）。
4. 手势与系统读屏手势不冲突，保留系统手势优先级。

**真实案例 1：未设置焦点** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025300010001375159)
登录短信验证页面的「无法接收短信」按钮未设置焦点，**阻断用户访问操作**。应确保所有交互元素可访问，能被扫动及触摸访问。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-14qbcDanjI6PoapNM.png)

**真实案例 2：焦点顺序不符合逻辑** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025220010000782924)
忘记密码-身份验证界面焦点顺序异常，无法仅靠单指扫动浏览所有页面元素，应设置合理焦点顺序，使其**满足从左到右、从上到下的聚焦顺序**。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-1470QfWc4IL8Yzkma.png)

### 可理解性要求

1. **控件状态变化**（选中/禁用/成功/失败）需有朗读提示（如「已添加至购物车」）。
2. **控件类型与功能语义正确**（Button/Link/Switch/EditText）。
3. 错误提示明确可理解，并提供修复建议（如「号码格式错误，请输入 11 位数字」）。

**真实案例 1：控件状态缺失** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025080010001710490)
首页左上角 tab 标签控件状态缺失，致使用户**无法明确识别当前所处状态**，从而缺乏对操作的掌控认知感，应增加「已选中」状态的提示信息。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-142C8EnbRyQhuNXAQ.png)

**真实案例 2：控件类型错误** —— [工单](https://new-ux.jd.com/workbench?issueNo=2025420010001267207)
客户服务右上角「搜索」按钮的控件类型设置为图像，导致触摸朗读为「搜索，图像」，**影响用户对按钮功能的认知**，应将控件类型调整为「搜索，按钮」。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2026-02-04-22-14g0RbTYCWHtr2au4.png)

### 兼容性与性能

1. 支持 TalkBack、VoiceOver。
2. TalkBack 与 VoiceOver 环境下稳定，无明显卡顿、崩溃与明显延迟。
3. 控件层级合理，避免过深导致读屏性能问题。

### 各控件类型详细规范

| 控件类型 | 规范要点 | 示例 |
| --- | --- | --- |
| **按钮** | - 必须包含明确文本或可访问标签；避免仅图标<br>- 点击后若状态变化，需朗读反馈（如「已加入购物车」）<br>- 禁用态需表意清晰，并朗读「已禁用」 | ✅ **Good**：有明确文本「加入购物车」，朗读：「加入购物车，按钮」<br>❌ **Bad**：仅有购物车图标，无 contentDescription |
| **弹窗** | - 弹出后焦点自动移入弹窗首个元素，关闭后返回触发点<br>- 提供标题/作用说明，如「选择支付方式」<br>- 焦点仅能聚焦弹窗内的元素<br>- 提供可达的「关闭」控件并可朗读 | ✅ **Good**：弹窗弹出后朗读当前聚焦的首个元素，如「选择支付方式」，关闭后焦点回到触发按钮<br>❌ **Bad**：弹窗无标题，焦点留在背景区域，扫动/触摸能聚焦朗读弹窗外的元素 |
| **输入框** | - 必须有可读的 Label 与占位符区分；错误提示可被朗读<br>- 密码框等敏感信息注意朗读策略（不读出具体内容） | ✅ **Good**：朗读「请输入号码，输入框」并在错误时朗读「号码格式错误」<br>❌ **Bad**：无提示 label，仅占位符 |
| **切换控件** | - 切换后需朗读当前状态（已开启/已关闭/已选中/未选中）<br>- 成组 Radio 需朗读：标签+控件类型+控件状态 | ✅ **Good**：朗读「通知, 开关, 已开启」「通知, 开关, 已关闭」<br>❌ **Bad**：切换状态后无朗读反馈 |
| **列表** | - 每项朗读包含关键信息（标题+价格+状态）<br>- 分页/懒加载需提示加载状态 | ✅ **Good**：朗读「商品：iPhone 15，价格 7999 元，加入购物车, 按钮」<br>❌ **Bad**：商品信息朗读不完整，缺少价格信息 |
| **Banner/轮播** | - 提供暂停播放；自动轮播不打断朗读<br>- 朗读当前序号与总数 | ✅ **Good**：朗读「第 1 张，共 3 张，京东秒杀」<br>❌ **Bad**：自动轮播打断朗读 |
| **Tab/底部导航** | - 焦点顺序与视觉一致<br>- 切换后提示当前 Tab | ✅ **Good**：朗读「我的京东，选中」<br>❌ **Bad**：仅朗读「我的京东」，不提示选中状态 |
| **提示** | - 提示应被朗读，不阻断主要操作 | ✅ **Good**：朗读「下单成功」<br>❌ **Bad**：提示弹出但读屏无反馈 |
| **图片/图标** | - 装饰性图像可标记为忽略<br>- 承载信息的图像必须有说明 | ✅ **Good**：「银联」图标配置标签为「银联」，朗读「银联, 图片」<br>❌ **Bad**：「银联」图标朗读「未加标签, 图片」 |

## 推荐（SHOULD）

- 错误提示在告知错误的同时提供**修复建议**（如「号码格式错误，请输入 11 位数字」），而非仅提示「错误」。
- 列表分页/懒加载场景**提示加载状态**，避免视障用户对内容是否更新无感知。
- Banner/轮播**提供暂停播放**能力，自动轮播不打断朗读。
- 控件层级尽量扁平，避免层级过深影响读屏性能。

## 禁止（MUST NOT）

聚焦类（可操作性）需避免：

- **无焦点**：可交互元素无法聚焦，用户根本不知道有这个元素。
- **焦点陷阱**：焦点被限制在一个区域无法跳出，导致操作中断。
- **底层焦点**：焦点停留在底层控件上，而用户实际想操作的是上层控件。
- **大焦点覆盖**：多个控件焦点区域重叠，焦点跳转混乱，难以准确操作。
- **焦点过细**：焦点拆分过细，用户使用效率低下。

朗读类（可理解性）需避免：

- **冗余朗读**：重复内容、多个无意义或重复的焦点区域。
- **不完整朗读**：关键信息缺失。
- **错误朗读**：朗读内容与控件实际内容或功能不符（语义错误）。

其他：

- 颜色**不可**作为区分信息的唯一手段。
- 动效**不可**阻断朗读；**避免**高频闪烁（防癫痫风险）。
- 自定义手势**不可**与系统读屏手势冲突或拦截系统手势。

## 检查清单

无障碍适配 checklist 📝（交付前逐项核对）：

- [ ] 1. 所有控件有可读标签
- [ ] 2. 所有可操作元素可被聚焦
- [ ] 3. 所有图片有替代文本
- [ ] 4. 所有表单有标签和错误提示
- [ ] 5. 所有弹窗有标题和关闭按钮，关闭后返回触发点
- [ ] 6. 焦点顺序与视觉一致，无陷阱、无底层焦点、无大焦点覆盖
- [ ] 7. 关键状态变化有朗读提示；错误提示可理解且可定位
- [ ] 8. TalkBack/VoiceOver 实测通过；读屏性能稳定无卡顿

## 例外与审批

- **装饰性图像**可标记为忽略（无需替代文本）；但凡**承载信息**的图像必须有说明。
- 密码框等**敏感信息**采用特殊朗读策略：不读出具体内容。
- 问题定级与解决机制：具体定级标准与处理流程参见 [无障碍体验问题定级标准与解决机制](https://joyspace.jd.com/pages/n5zryd5V4atBC69j6wHB)。

## 问题定级（京东常见无障碍问题类型与定级标准）

> 按问题对无障碍用户的阻断程度分 A/B/C 级，A 级最严重（直接阻断访问与操作）。

| 等级 | 问题类型 | 场景说明 | 实现方式 |
| --- | --- | --- | --- |
| **A 级** | **无标签** | 界面中的某个功能元素（如按钮、输入框、图标等）没有提供明确的文字说明或语义标签，导致辅助技术（如屏幕阅读器）无法正确识别和朗读该元素的内容或功能。 | Android: `contentDescription`；iOS: `accessibilityLabel`；H5: `aria-label` |
| | **无焦点** | 某个可交互元素（如按钮、输入框、菜单项等）在使用键盘或辅助技术时无法聚焦，导致用户无法通过键盘或屏幕阅读器进行操作（用户根本不知道有这个元素）。 | Android: `focusable=true`；iOS: `isAccessibilityElement=true`；H5: `tabindex=0` |
| | **大焦点覆盖** | 多个控件的焦点区域重叠，导致用户无法准确选择目标控件。用户操作时焦点跳转混乱，难以准确操作。 | 调整点击区域，仅为主按钮分配焦点；避免父层级 clickable 过大 |
| | **底层焦点** | 焦点停留在底层控件上，而用户实际想操作的是上层控件。用户操作时焦点不准确，导致误操作或无法操作目标控件。 | 为遮挡层设置 `importantForAccessibility='noHideDescendants'` 隐藏底层控件 |
| | **焦点陷阱** | 焦点被限制在一个区域，无法跳出，用户无法继续操作其他控件。用户无法继续使用 App，导致操作中断。 | 保存触发点并 `restore focus` |
| | **朗读完全错误** | 屏幕阅读器读出的内容与控件实际内容或功能不符。用户接收到错误信息，导致操作错误。 | 检查控件的描述属性与类型绑定；保持朗读文本与视觉一致 |
| | **手势异常** | 手势操作在无障碍模式下行为异常，如滑动、双击、长按等不生效或误触发。用户无法通过手势完成操作，或操作被错误执行。 | 避免全屏拦截；在 View 中重写 onTouch 时优先保留系统手势 |
| | **闪停卡等性能异常** | App 在无障碍模式下出现卡顿、闪屏、加载慢等性能问题。 | / |
| **B 级** | **控件类型缺失/错误** | 控件的类型未正确声明或缺失，导致屏幕阅读器无法识别其功能。用户无法理解控件的用途或操作方式。 | Android 使用 `Button` 类；H5 `role="button"` |
| | **控件状态缺失/错误** | 控件的状态（如选中、禁用、展开等）未正确声明或缺失。用户无法知道控件当前的状态，导致误操作。 | 状态变化时触发 `sendAccessibilityEvent` 或 `announce` 更新朗读 |
| | **焦点过细** | 控件的焦点拆分过细，用户使用起来效率低下。 | 合并无效子控件，聚焦到主要操作控件 |
| | **朗读不完整** | 屏幕阅读器朗读的内容不完整，缺少关键信息。 | 确认朗读顺序与文本拼接逻辑完整性；保持一对一描述 |
| **C 级** | **焦点过大/过小** | 控件的焦点区域与实际可操作区域不匹配，导致用户在使用辅助技术（如屏幕阅读器、键盘导航）时操作困难。 | / |
| | **焦点/朗读冗余** | 页面中存在多个无意义或重复的焦点区域，或屏幕阅读器重复朗读相同内容，造成用户操作效率低下或理解困难。 | 避免 label 和 hint 重复设置；仅保留核心信息 |
| | **其他体验问题** | 指不属于上述分类，但仍然影响无障碍用户体验的问题。这类问题可能涉及信息/错误提示、操作时间不足等。 | / |

# 无障碍走查 / 无障碍审查（Accessibility Audit）

> 一句话：由**评估员（借助读屏工具）**逐屏、逐流程地检视界面是否满足无障碍规范，找出"违反了哪条、读屏实际怎么表现"的无障碍问题，按等级排序、给出**分平台整改方式**——一种以**专家检视为主、不必先招真实视障用户**的无障碍检查（accessibility inspection）。
> 落点：`methods/toolbox/collection/`。方法本体在此**唯一出处**，scenarios/skills 引用而非复制。
> 标尺出处：无障碍"**必须遵守什么**"（POUR 四原则、各控件类型规范、MUST/SHOULD/MUST NOT、A/B/C 问题定级、交付前 checklist）见 [`methods/standards/accessibility-adaptation.md`](../../standards/accessibility-adaptation.md)；本篇讲**怎么组织一次走查**（怎么开读屏、走查节奏、量化阈值核查、汇总与整改），不复述规范条目。它是 [启发式评估](heuristic-evaluation.md) 的**无障碍专项变体**。

## 何时用 / 不适合

- **适用**：
  - 上线前 / 改版前想**低成本、早期**扫出无障碍适配问题；无障碍专项验收或治理前的**预扫**；
  - 原型、灰度、已上线版本都能查；App（iOS / Android / 鸿蒙）/ H5 / 小程序皆适用；
  - 团队想借机**养无障碍意识**（对照规范与读屏表现练眼力）。
- **不适合 / 慎用**：
  - **替代真实视障用户验证**——走查找的是"评估员眼里的疑似问题"，读屏真实体验高度依赖真人操作与个人辅助设置，结论仍须真实视障用户的任务测试验证（见 [`usability-testing.md`](usability-testing.md)）；
  - 需要**量化无障碍指标**（无障碍模式下的任务成功率 / 耗时）时——那是任务测试的活；
  - 判断高度依赖特定辅助技术版本或个体使用习惯时（评估员未必代表得了真实视障用户）。

## 前置准备

- **读屏工具（本方法的关键"仪器"）**：iOS 打开 [设置]-[辅助功能]-[旁白（VoiceOver）]；Android 打开 [设置]-[无障碍]-[TalkBack]。走查前**先练熟基本手势**（单指右滑＝下一个元素、单指双击＝激活、两指上下滑＝滚动、三指滑＝翻页等），否则容易把"自己不会用读屏"误当成"产品有问题"。
- **评估员**：无障碍走查可单人做初扫，但**问题覆盖同样受益于多名独立评估员**（依据见 [`sampling.md`](../../standards/sampling.md) 与 [`models/nielsen-heuristics.md`](../../../models/nielsen-heuristics.md)——任一评估员都会漏检一部分）；条件允许时 **2–3 名评估员各自独立走查再合并**。评估员最好懂 POUR 与读屏；不熟时先对照规范培训对齐"每类控件该读成什么、什么算问题"。
- **收窄范围（重要）**：明确评哪个端 / 版本号 / AB 版本；聚焦到**一个核心任务、一个板块、一类页面**；覆盖主流机型与系统（iOS / Android / 鸿蒙），必要时区分登录态、新老用户态。范围太宽会走查不深。
- **量测工具**：对比度用取色器 / 对比度检查工具；触达区、字号用设计标注或开发者工具核对。
- **记录物料**：统一问题记录表，字段建议：**位置**（页面·路径·截图）/ **读屏实际朗读或表现** / **违反的规范条目**（原则 + 控件类型）/ **等级** / **分平台整改方式** / **复测验证点**。

## 执行步骤

1. **准备**：开读屏、收窄范围、备记录表；评估员先熟悉读屏手势与本次评估对象。
2. **多遍走查（三遍法 + 走查节奏，先熟悉再挑刺，互不影响先各自独立）**：
   - **第一遍 · 视觉整体浏览（先不开读屏）**：整体过一遍熟悉流程；同时核**不依赖读屏就能看到的**——颜色是否被当作区分信息的唯一手段、文本 / 图形**对比度**、**触达区**大小、是否有高频闪烁或自动动效、**字号放大**后是否可读。
   - **第二遍 · 读屏线性遍历（开 VoiceOver / TalkBack）**：单指从左到右、从上到下逐个滑动遍历，**听每个元素怎么朗读**，逐项核：① 可交互控件有无可读**标签**；② 图片 / 图标有无**替代文本**（装饰性应标记忽略、承载信息必须朗读）；③ **控件类型**是否正确（按钮读"按钮"而非"图像"）；④ **状态变化**是否朗读（选中 / 禁用 / 加载 / 成功失败）；⑤ **焦点顺序**是否符合视觉阅读顺序，有无**焦点陷阱 / 底层焦点 / 大焦点覆盖 / 焦点过细**；⑥ **弹窗**打开后焦点是否移入首元素、关闭后是否回到触发点、能否误聚焦到弹窗外；⑦ 朗读有无**冗余 / 不完整 / 错误**。
   - **第三遍 · 键盘 / 手势与边界**：核自定义手势是否与系统读屏手势冲突、是否拦截系统手势；刻意试**逻辑边界 / 异常态**（空态、错误态、加载中、无货、回退、超时）——状态朗读缺失最易在边界暴露；核读屏下有无明显卡顿 / 闪退 / 延迟（兼容性与性能）。
   - **查漏补缺**：补前几遍没覆盖到的细节。
   - 每发现一个问题就记一条：**写清在哪、读屏实际怎么表现、违反哪条规范、分平台怎么改**。
3. **对照规范逐项核**：拿 `accessibility-adaptation.md` 的 **8 项交付前 checklist + 9 类控件详规**回扫，确保按钮 / 弹窗 / 输入框 / 切换 / 列表 / Banner·轮播 / Tab·底部导航 / 提示 / 图片各类型都覆盖到，别漏类型。
4. **定级**：按 `accessibility-adaptation.md` 自带的 **A / B / C 问题定级**（问题类型 → 等级，A 级最严重、直接阻断访问与操作）给每条打级；需要跨问题排序、或与通用体验问题对齐口径时，接 [`issue-prioritization.md`](../analysis/issue-prioritization.md)（NN/g 0–4 或广度×深度 ABCD、P0–P3、RICE/ICE）。
5. **汇总输出**：无障碍问题清单 + **整改清单**（每条含**分平台整改方式** + **复测验证点**）+ 按等级排序的修复优先级。多人走查时先**亲和图聚类去重**再合并。

## 量化核查基准（WCAG，供第一遍视觉核查用）

> 规范强调"颜色不可作为区分信息的唯一手段""避免高频闪烁（防癫痫）"，但**未给具体数值**。以下阈值取自 W3C《WCAG 2.1》，可作走查时的量化基准（规范定稿时可考虑并入 MUST）：

- **文本对比度**：正文 ≥ **4.5:1**；大字号（≥18pt 或 14pt 粗体，约 24px / 18.5px）≥ **3:1**（WCAG 1.4.3；logo、纯装饰、失效控件文本可豁免）。
- **图形 / UI 控件对比度**：图标、控件边界、图表等与相邻色 ≥ **3:1**（WCAG 1.4.11 非文本对比度）。
- **触达区**：建议 ≥ **44×44 pt / CSS px**（WCAG 2.5.5；iOS HIG 44pt、Android Material 48dp 为平台基线）；句中行内链接等可豁免。
- **文本缩放**：放大到 **200%** 时内容不丢失、不重叠、不截断，无需横向滚动即可阅读（WCAG 1.4.4）。
- **移动 / 闪烁内容**：自动播放且持续 > **5 秒**的内容需提供暂停 / 停止 / 隐藏（WCAG 2.2.2）；任何内容 1 秒内闪烁**不超过 3 次**（WCAG 2.3.1，防癫痫）。

## 关键技巧与注意

- **会用读屏是前提**：先练熟基本手势，别把"自己不会用读屏"当成"产品的问题"。
- **违反 ≠ 一定是问题**：结合场景与替代方案判断（如装饰性图像本就该被忽略朗读、层级弹窗焦点本就该锁定），别把"提到某条规范"当"违反"堆伪问题；低等级也如实标低，别为凑数注水。
- **分平台给整改方式**：无标签 → Android `contentDescription` / iOS `accessibilityLabel` / H5 `aria-label`；无焦点 → `focusable=true` / `isAccessibilityElement=true` / `tabindex=0`；状态变化 → `sendAccessibilityEvent` / `announce` 更新朗读……具体对照规范 A/B/C 定级表的"实现方式"列，别默写。
- **每条要有证据**：引用读屏"**实际朗读了什么**"或截图，不脑补界面。
- **定级 / 优先级用单一出处**：每条无障碍问题的等级用规范自带的 A/B/C；跨问题排序、排期用 `issue-prioritization.md`。
- **真机实测**：模拟器与真机的读屏行为常有出入，关键结论以**真机 VoiceOver / TalkBack 复测**为准。

## 常见坑

- **评估员不会用读屏**：手势不熟，误报一堆"读不出来"其实是操作不当。
- **只在视觉层挑毛病，没真开读屏走一遍**：无标签 / 焦点顺序 / 状态朗读这类问题只有开读屏才暴露。
- **把走查当"已验证无障碍"**：跳过真实视障用户——它只是预扫 / 检视，不能替代真人验证。
- **找完不定级、不给分平台整改方式**：研发拿到一堆"读不对"却无从下手。
- **只查正常态**：没试空态 / 错误态 / 加载态——状态朗读缺失恰在这些边界暴露。
- **范围太宽**：一次想查完整个 App，结果每处浅尝辄止；应收窄到核心任务 / 板块。

## 产出与交付

- 直接产出：**无障碍问题清单**（位置 / 读屏表现 / 违反的规范条目 / 等级 / 整改方式 / 复测点）+ **整改清单**（可直接派给研发的分平台改法 + 复测验证点）+ 按等级排序的**修复优先级** + **走查范围与方法说明**（评了哪个端 / 版本、几名评估员、覆盖哪些机型与状态）。
- 可接续的分析技术（写进 `related`）：[`issue-prioritization.md`](../analysis/issue-prioritization.md)（问题分层与优先级排期）；真实视障用户验证接 [`usability-testing.md`](usability-testing.md)。

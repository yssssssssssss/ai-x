# 启发式评估 / 体验走查（Heuristic Evaluation）

> 一句话：由**少数专家评估员**拿一套公认的可用性准则（默认 [尼尔森十大启发式](../../../models/nielsen-heuristics.md)）逐屏、逐流程地检视界面，找出"违反了哪条"的可用性问题并按严重度排序——一种**不需要真实用户**的可用性检查（usability inspection）法。
> 落点：`methods/toolbox/collection/`。方法本体在此**唯一出处**，scenarios/skills 引用而非复制。
> 标尺出处：10 条准则的含义见 `models/nielsen-heuristics.md`；本篇讲**怎么组织一次评估**（评估员、流程、严重度、汇总），不复述准则内涵。

## 何时用 / 不适合

- **适用**：
  - 想**低成本、早期**地扫出明显可用性问题（没预算/没时间招真实用户时尤其值）；
  - **正式可用性测试之前**先预扫一遍，锁定该重点测什么；
  - 原型、灰度、已上线版本都能评；也适配 App / Web / 小程序 / 大屏 / VR / 语音等多种界面；
  - 团队想借机**养可用性直觉**（新人对照启发式练眼力）。
- **不适合 / 慎用**：
  - **替代用户研究**——启发式评估找的是"专家眼里的疑似问题"，UX 高度依赖真实情境，结论仍须用真人验证（见 `usability-testing.md`）；
  - 需要**量化指标**（成功率、SUS 分、转化）时——那是任务测试/问卷的活；
  - 高度依赖领域知识或特定人群心智的判断（专家也未必代表得了目标用户）。

## 前置准备

- **评估员（本方法的"样本"）**：**3–5 名独立评估员**。依据见 `methods/standards/sampling.md` 与 `models/nielsen-heuristics.md`——任一评估员都会漏掉一部分问题，多人合并显著提升覆盖；单人不可靠。评估员最好懂可用性/这套启发式；条件有限时，先培训对齐"每条启发式什么意思、严重度怎么打"。
- **收窄范围（重要）**：明确**评估对象与边界**——评哪个端 / 版本号 / AB 版本；聚焦到**一个核心任务、一个板块、一类用户或一种设备**，范围太宽会走查不深。<!-- 走查版本/硬件准备：京东《产品体验问题走查指南》 -->
- **硬件 / 环境**：按对象覆盖主流机型与系统（iOS / Android / 鸿蒙），必要时区分网络、登录态、新老用户态。
- **记录物料**：统一的问题记录表（NN/g 工作表、表格或在线白板均可），字段建议：问题描述 / 位置（页面·路径·截图）/ 违反的启发式 / 严重度 / 改进建议。

## 执行步骤

1. **准备**：选定并培训评估员；约定记录方式与问题字段；把评估范围收窄到具体任务/板块/设备。
2. **独立评估（每人 1–2 小时，互不看彼此的发现）**——保持独立是数据质量的根本，看了别人的会被带跑。采用**两遍法 + 走查节奏**：
   - **第一遍**：先把目标流程整体走一遍，只**熟悉系统**、不评判；先做**整体浏览**（从上到下看内容/布局/视觉等直观问题）。
   - **第二遍**：再回头**逐条比对 10 条启发式**，找违反点。落到操作层就是**模拟使用**——扮演真实用户，按重要程度逐一体验具体功能；**尤其注意逻辑边界**（如购物车显示"仅剩 1 件"却去点"+2 件"，看反馈与链路是否合理）。
   - 最后**查漏补缺**：补走前两步没覆盖到的细节。<!-- 整体浏览/模拟使用/查漏补缺：京东走查指南 -->
   - 每发现一个问题就记一条：**写清在哪、违反哪条启发式、怎么改**。
3. **汇总合并（小组）**：把各人发现**用亲和图聚类**去重；集体讨论：哪里看法一致/分歧、哪些问题最伤体验或业务目标、哪些还需更多数据、下一轮可用性测试该优先测什么、短期与长期分别怎么修。
4. **严重度评级（独立打分后取均值）**：现场只顾着找问题、严重度判断不可靠，故**评估结束后**再单独评级——把**全部已发现问题**列成清单（每条配足够描述、必要时附截图），让**每位评估员独立**对整张清单打 **0–4 分**（约 30 分钟/人），再**取均值**（约 3 人均值即足够可靠）。评级看三因素的综合：
   - **频率**：常见还是罕见；
   - **影响**：用户能否轻松克服；
   - **持续性**：一次性障碍（学会就好）还是反复困扰；
   - （另可叠加**市场影响**——有些问题技术上轻微却严重伤口碑。）
   严重度评级的标准（NN/g 0–4 与京东《体验问题评级标准》影响范围×严重程度）以 [`issue-prioritization.md`](../analysis/issue-prioritization.md) 为**唯一出处**（本篇不复制档位表）；评级用于**排修复优先级**与**上线决策**（一堆"灾难级"基本不该上；剩下都只是"装饰级"则可放行）。
5. **输出**：问题清单（问题 × 违反的启发式 × 严重度 × 位置/证据 × 改进建议）+ 按严重度排序的修复优先级 + 评估说明。接 `analysis/issue-prioritization.md` 排期。

## 关键技巧与注意

- **严重度评级用单一出处**：发现的问题如何定级，遵循 [`issue-prioritization.md`](../analysis/issue-prioritization.md)（评估发现的问题定级/排序的唯一出处）——含 **NN/g 0–4**（频率×影响×持续性，凭专家判断即可打、无需数据）与**京东《体验问题评级标准》**（影响范围×严重程度→等级，需 UV占比/点击率/曝光率/反馈量数据）两套；无影响范围数据用 0–4，有则可并用京东标准。**本篇不复制档位表，以该文为准。**
- **独立性**优先于效率：评估、严重度评级都要求评估员**先各自独立完成**再合并/取均值，避免互相影响与从众。
- **两遍法别省**：不先熟悉就逐条挑刺，容易把"还没看懂"当成"设计有问题"。
- **给场景、模拟真实用户**：像真实用户那样按重要度走核心流程，并刻意试**逻辑边界 / 异常路径**——边界处最容易暴露问题。
- **违反 ≠ 一定是问题**：判断要结合场景与替代方案（见 `models/nielsen-heuristics.md` "局限与误用"），但也别轻易把自己当例外。

## 常见坑

- **只用 1 名评估员**：漏检严重，务必凑够 3–5 人独立评。
- **把启发式当硬性规范**逐条扣分，忽略场景权衡，产出一堆"伪问题"。
- **拿它替代真人测试**：当成"已经验证过可用性"，跳过用户研究——它只是预扫。
- **找完问题不评严重度**：没有 0–4 分就无法排优先级，团队会"什么都想改、什么都改不动"。
- **范围太宽**：一次想评完整个 App，结果每处都浅尝辄止；应收窄到核心任务/板块。
- **评估员互相串**：边评边讨论，独立性被破坏，覆盖率反而下降。

## 产出与交付

- 直接产出：**可用性问题清单**（每条含 位置/证据、违反的启发式、严重度 0–4、改进建议）+ 按严重度排序的**修复优先级清单** + **评估范围与方法说明**（评了哪个端/版本、几名评估员、用的哪套启发式）。
- 可接续的分析技术（写进 `related`）：`methods/toolbox/analysis/issue-prioritization.md`（问题分层与优先级排期）；竞品横评可接 `experience-evaluation-rating.md`（分档打分 / GSB / 优势占比）。

# 满意度异动归因（满意度下降如何分析）

> 来源：神灯圈子·夏凡舒《NSS介绍：用户满意度下降时如何归因》 原文 http://xingyun.jd.com/shendeng/article/detail/54834（Joyspace 镜像 https://joyspace.jd.com/pages/2rD0fd6djkqgWUZjNHwa）

> 一句话：当满意度分数（尤其 NSS）出现下降时，用「数据判断 → 定位问题 → 诊断根因 → 叙事输出」的三阶段流程，把整体异动收敛到具体用户群、功能模块与可行动根因。
> 落点：`methods/toolbox/analysis/`。判据——这套归因流程可脱离 NSS 单独复用于任何「分数型结果指标」的异动分析（CSAT、转化率、留存率等同样适用）。

## 何时用
- 适合的问题类型：某个结果性满意度指标（NSS / CSAT / NPS / CES）环比或同比出现下降，需要判断这是真异常还是噪音波动，并进一步定位「谁在哪里为什么不满」，最终给产研一份可执行的优化建议。
- 在产品迭代期的定位：常用于版本更新后或定期调研后的复盘环节，是「发现分数掉了」之后的标准动作。
- 与近似技术的区别：
  - 本篇是**异动归因流程**（分数为什么变化、变化是否真实、根因在哪），不是常态化的驱动因素建模——后者用「满意度驱动因素分析 KDA」`methods/toolbox/analysis/key-driver-analysis.md`，回答「平时哪些属性最影响满意度」。
  - 本篇消费的开放题文本/关键词分析属 VOC 范畴，深入做主题/情感编码见 `methods/toolbox/analysis/voc-analysis.md`。
  - 指标本身的定义与计分口径、问卷如何设计与清洗，见下文「四种度量方式（背景速览）」与 `methods/toolbox/collection/satisfaction-survey.md`，本篇不展开。

### 四种度量方式（背景速览）
归因前先明确在追的是哪个指标。本篇默认对象是 NSS（净满意度，集团内统一汇报口径），其余三种作为对照：

| 指标 | 关注点 | 典型场景 | 计算方式 |
|---|---|---|---|
| 费力度 CES | 完成单个任务所需努力（解决效率） | 注册、搜索、下单、修改信息、退款等流程型任务 | CES = 所有填答得分的平均值（或分析低分比例）|
| 满意度 CSAT | 对某一次具体交互的即时感受（触点型） | 客服对话、页面设计、物流通知、支付等微观单次触点 | CSAT = (4-5 分高分数量) / 总填答量的占比 |
| 满意度 NSS | 用户对一段时间内**所有体验**的整体评价（关系型） | 版本更新后或定期调研，评估产品综合健康度 | NSS = %满意者(9-10 分) − %不满意者(0-6 分) |
| 推荐度 NPS | 用户忠诚与主动推荐意愿（粘性） | 战略层面，衡量口碑与长期增长潜力 | NPS = %推荐者(9-10 分) − %贬损者(0-6 分) |

> 京东内部由于横跨数十乃至上百条产品线，若 B 端用 CES、C 端用 NPS、客服用 CSAT，管理层无法横向比较体验水位。NSS 作为包容性极强的「结果性指标」，无论 B/C 端、无论经历何种流程，最终都收敛为一个整体满意与否的判断，故**统一采用 NSS**，并配套标准化问卷、计算口径与汇报模板，降低沟通与教育成本。指标选择与问卷三段式设计详见 `methods/toolbox/collection/satisfaction-survey.md`。

## 输入数据要求
- **分数与样本量**：本期与对比期（上月/上季/上版本）的满意度分值，以及对应的**填答样本量 n**（显著性检验必需）。
- **可下钻的维度字段**：用户属性（B 端：部门/岗位/职级/品类/新老/使用频次；C 端：RFM 分层/会员等级/活跃度/地区）与功能模块标签，用于交叉拆分。
- **定性反馈**：开放题文本（改进建议）、低分用户的二级归因多选（不满意原因选项及占比）。
- **关联证据（按需）**：用户行为日志（任务成功率、性能数据、错误日志、路径退出率）、产研/技术侧的版本与故障记录、外部舆情。
- **最小样本（口径见下）**：模块 n<20、系统 n<100 时不对单期波动做归因，只作监控；是否达到置信样本量用 NPS 样本量计算器估算。

## 分析步骤

### 第一步：数据判断——区分是分数波动还是确有异常
**目的**：排除数据噪音，确保投入精力分析的波动是有意义的。

1. **判断样本量是否充足**
   - **样本量 < 20 的模块、样本量 < 100 的系统，忽略单期波动**，仅作观测指标，监控趋势是否持续走低（大于 3 个月），或合并季度数据再分析。
   - 如何判断是否达到置信样本量：用 NPS 样本量计算器，输入用户量级及置信区间（通常为 95%）即可得出所需置信样本量；也可通过预估填答率，反推所需投放样本量。

2. **检查统计口径**
   - 确认计算口径是否与往期一致，避免实际是因替换新口径导致分数变动（而非用户体验真实变化）。

3. **检查统计显著性**（二选一或灰区组合）
   - **方法一 · 假设检验（卡方）**：用卡方检验计算器，输入本月及上月分值，
     - **P < 0.05** → 分数环比有显著性差异，需进一步归因；
     - **P ≥ 0.05** → 无显著性差异，以波动为主。
   - **方法二 · 区间检验（置信区间）**：用置信区间计算器输入本月和上月的满意度分数及样本数量，得出 95% 置信区间。判断三种情形：
     - 两区间**没有重叠** → 强烈暗示存在**显著差异**，需进一步下探归因；
     - 两区间**有大量重叠或基本重叠** → 很可能**没有显著差异，以波动为主**；
     - 两区间**只是轻微重叠**（如 NPS 本月 60% 区间与上月 75% 区间仅在 65~70% 小部分重叠）→ 处于**灰色地带**，需回到方法一做卡方检验，测算 P 值得出置信结论。

> **阶段小结**：如果波动不显著，无需启动深度归因，避免人力浪费。

### 第二阶段：定位问题——「谁」在「哪里」感受到了不满？
**目标**：将整体的满意度下降，精准定位到具体的用户群和功能点。

1. **整体趋势分析**
   - **持续阴跌**：通常意味着长期积累的问题（如产品迭代方向偏离需求、使用体验持续劣化）。
   - **突然跳水**：通常与近期发生的特定事件有关（如改版 / 系统重大 bug 等，根因诊断见第三阶段）。

2. **用户维度下钻（「谁」在不满？）**
   - **B 端用户属性**：按部门、岗位、职级、品类、新老用户、使用高/低频用户等维度拆分满意度。
   - **C 端用户属性**：按 RFM 分层（是否高价值）、付费会员/普通用户、高/低频活跃度、地区等拆分。
   - **关键分析**：下降是由**全体用户**引起的，还是由**「采销部门的新员工」**或**「某地区的低频用户」**等特定群体驱动的？

3. **模块维度下钻（「哪里」不满？）**
   - 查看各功能模块的满意度变化，哪个模块跌幅最大？
   - **关联分析**：结合用户维度，看是**哪个群体**在**哪个模块**的满意度下降最厉害。例如：「是采销人员在『比价查询』模块的满意度暴跌」。

> **阶段小结**：此时可形成初步的**问题假设**，如「假设『直播操控台』在『直播列表』模块的满意度下降，可能与版本改版后该模块的界面变化有关」。

### 第三阶段：诊断根因——为什么他们会不满？
**目标**：验证问题假设，找到导致不满的具体、可行动的根因。

1. **分析不满意原因——最宏观的数据**
   - 通过不满意各选项占比及趋势变化，聚焦导致低分最多的原因。例：直播工具中「美颜效果不满意」占比最高（72%），且环比上涨 10% → 体验持续劣化。

2. **挖掘用户反馈——最直接的线索**
   - **关键词分析**：看用户负面情绪是否集中在某个关键词上（如卡顿、封禁、缺少 xx 功能等）及其环比趋势。（文本主题/情感深入分析见 `methods/toolbox/analysis/voc-analysis.md`。）

3. **多方团队协作——最高效的输入**
   - 主动跟相关产品、技术等部门了解最近改动，推断满意度下降时间点是否与以下事件关联：
     - **版本发布**：是否在某个新版本上线后开始下降？
     - **系统故障**：近期是否有知名的性能下降或 BUG？
     - **市场舆情**：外部是否有重大舆论事故或竞品市场有变？（to C）

4. **关联行为数据——最客观的证据**
   - 针对已定位到的「问题模块」和「问题用户群」，通过数据/研发提供的**用户行为日志**分析，重点关注：
     - **任务成功率**：核心任务的失败率是否升高？（如提交采购单、提交订单等）
     - **性能数据**：页面加载时间、接口响应时间是否显著变高？
     - **错误日志**：是否集中触发某类错误（如「查询超时」「提交失败」）？
     - **路径分析**：在哪个具体页面或步骤的退出率异常升高？

5. **建立回访机制——最全面的了解**
   - 针对低分用户，主动联系，了解深层不满意原因。提问要**具体化、场景化**：

     | | 提问方式 |
     |---|---|
     | **不要问** | 「您为什么打低分？」 |
     | **而要问** | 「我们看到您昨天在使用【采购单查询】后打了低分，能分享一下是哪个具体步骤让您觉得卡住了吗？」 |

   - 必要时进行深度用户访谈，**搭建用户旅程地图**：快速邀请几位「问题用户群」代表，观察其操作流程，重点关注「问题模块」，全面了解使用全流程卡点并梳理总结。
   - **联系不到用户怎么办？**
     - 在问卷开放题中增设联系方式问讯（如「请留下您的联系方式，参与后续深度访谈，获取奖励——礼品卡、积分等」）。
     - 建联其他团队（如由采销引荐其负责的供应商、商家运营引荐 BP 的外卖商家等）。
     - 从工单/客服系统回捞相关反馈关键词：经常提交相关工单/客诉的用户，很可能就是给出低分但未留名的用户。

> **阶段小结**：此时可验证初步假设是否全面合理，并补充信息，得出最终分析结论。

## 结果解读
**目标**：将「满意度下降」归因转化为清晰的、可供团队参考的优化方向建议及具体执行清单。最终输出按「事实—定位—根因—总结」四段叙事呈现：

- **事实**：本月整体 NSS 得分为 xx，环比下降 xx%。
- **定位**：下降主要由【某用户群体】在【某功能模块】的体验恶化导致。
- **根因**：通过数据分析，该群体在该模块的【响应时间延长至 XX 秒】/【任务失败率提升 xx%】，结合用户反馈，核心导致低分原因为【系统卡顿（占比 xx%）】，具体表现问题是【……】并引用用户原声。
- **总结**：建议产研团队优先优化【xx 模块】的【性能/交互设计】，减少用户等待时间，同时针对用户提到的细节操作卡点【……】进行优化，提升用户使用流畅度。

> 得出结论后，及时和产品沟通确认具体优化方案及落地时间，并补充进文档，使优化进程一目了然。

### 用户结构因素的拆解（识别「不是体验变差，而是人群变了」）
分数下降有时并非体验劣化，而是填答**用户结构**发生了变化（如低频 / 新用户占比上升，把整体均值拉低）。归因前应把整体变动拆为「各人群（分层）满意度变化」与「人群占比变化」两部分，**先扣除结构因素**，再对剩余的「同人群内部下降」做第二、三阶段归因；否则会把"人群构成变化"误读为"产品体验变差"。

> 带数字的结构因素拆解示例（按"假设用户结构不变"反推结构贡献占比）见 `methods/scenarios/comprehensive-business/cross-line-topics/b-end-experience-measurement.md` 的「归因与应用 · 趋势波动分析（用户结构）」。

## 可视化 / 出图
- **趋势线图**：整体 NSS 的逐期走势，叠加版本发布/故障时间点标注，用于区分「阴跌 vs 跳水」。
- **维度下钻表/热力图**：用户群 × 模块的满意度矩阵，高亮跌幅最大的格子（定位「哪个群体在哪个模块」）。
- **置信区间对比图**：本期 vs 对比期分数及 95% 置信区间，直观看两区间是否重叠（支撑第一步显著性判断）。
- **不满意原因占比图**：低分二级归因各选项的占比及环比变化（条形/帕累托）。
- **关键词/词频图**：开放题负面反馈的关键词聚集与趋势。

## 工具 / 实现
源文提供的在线计算工具，照搬为链接：

- **NPS 样本量计算器（Sample Size Calculator）**：https://www.npscalculator.com/en/Significance ——输入用户量级与置信区间（通常 95%），得出所需置信样本量。
- **卡方检验计算器**：https://www.evanmiller.org/ab-testing/chi-squared.html ——输入本月及上月分值，得出 P 值，判断是否 P < 0.05。
- **置信区间计算器**：输入本月/上月满意度分数及样本量，得出 95% 置信区间，判断两区间是否重叠。⚠️ 待补充（缺源）：源文未给出该计算器的具体链接。

## 局限与误用

- **样本量不足时不要归因单期波动**：模块 n<20、系统 n<100 应忽略单期波动，只作趋势观测（持续走低 >3 个月或合并季度再分析），否则会把噪音当信号。
- **口径不一致会造成假异动**：归因前必须确认计算口径与往期一致，避免把「换了新口径」误读为「体验变化」。
- **灰色地带不可武断**：置信区间仅轻微重叠时既不能直接判为显著、也不能判为波动，须回到卡方检验测算 P 值再下结论。
- **低分回访问法不当会拿不到有效信息**：直接问「为什么打低分」难以获得可行动线索，应把问题具体化、场景化到「哪个步骤卡住」。
- **结构因素**：分数下降可能来自填答用户结构变化而非体验劣化，需先拆出结构因素再归因（带数字的拆解示例见 `methods/scenarios/comprehensive-business/cross-line-topics/b-end-experience-measurement.md`）。
- ⚠️ 待补充（缺源）：源文未直接展开「非等比例抽样需谨慎」等抽样代表性议题；如需补充须另寻来源，勿自造。

# 定性洞察分析框架

> 来源：神灯圈子·邱伟雄《从“听见”到“看见”：定性研究中的洞察方法论》 原文 http://xingyun.jd.com/shendeng/article/detail/56773
<!-- 合并自 Joyspace「访谈技巧知识库」外部文章，2026-06-25 -->
> 补充来源（「定性分析的方法论原则」一节）：《访谈后如何得出有价值的访谈结论——用户访谈系列之三》@玄清，人人都是产品经理　https://www.woshipm.com/share/6033016.html

> 一句话：把碎片化的访谈资料，系统地“生产”成能指导产品行动的用户洞察。
> 落点：`methods/toolbox/analysis/`。判据——这四大框架（用户旅程地图 / 价值主张画布 / 同理心地图 / A·B·C 模型）能脱离单个研究被反复复用于不同定性资料的结构化分析。

## 何时用
- 适合的问题类型：完成一轮深度访谈/观察后，原始记录散乱、结论难以拼成清晰图景，需要从“收集到的信息”里有目的地提炼出能改变产品方向、触动用户内心的洞察。
- 解决的痛点（“洞察焦虑”）：源于“我们熟练地‘收集’信息，却未系统地‘生产’洞察”这一差距——满眼用户吐槽与故事片段，却差一口气难以精准、有力地回答“我们最应该解决的根本问题是什么”。
- 与近似技术的区别：
  - 观察 vs 洞察——观察是对表面现象的直接感知，主要回答“发生了什么”；洞察穿透言语表象，回答“为什么”。数据告诉我们“是什么”，洞察告诉我们“为什么”，不知道“为什么”就只能被动修补“是什么”的问题。
- **洞察的定义**：洞察 = 用户 在 某个具体场景下，想要完成某个任务，因为他希望达成某种情感需求/价值，但却面临着某个具体的矛盾或障碍。

## 输入数据要求
- 定性原始资料：用户深度访谈的录音转写、现场观察记录、用户原话引述（Quotes）。
- 高质量素材依赖访谈采集环节（见下“访谈场景的洞察小技巧”）——尤其是用户的故事、细节、场景、情绪，以及语调/表情/肢体语言等非言语信号。

## 分析步骤

### 前置：访谈场景的洞察小技巧（提高素材含金量）
1. **短暂沉默之后追问**：访谈中最有价值的信息，往往出现在短暂沉默之后。用户答完一个问题、停顿时，不要急于抛出下一个问题填补空白；保持微笑和眼神接触，多等待 3–5 秒，传递“我对你刚才说的很感兴趣，请继续 / 我正在认真思考”的信号，鼓励用户说出未经修饰的、更深层次的想法。
2. **读身体语言（用户的身体不会说谎）**：用户在聊不同话题时的语调、语速、表情、眼神乃至坐姿变化都是宝贵数据。
   - 语调和音量：谈到兴奋点时音量不自觉提高、语速加快——用于**高价值场景的识别**（哪怕口头都说“一样”，语气更兴奋的场景对 TA 价值更高）。
   - 表情和眼神：回忆操作时眼神是飘忽（可能在努力回忆或编造）还是笃定自信——用于**真伪需求的识别**。
   - 肢体语言：身体前倾说明投入、愿意沟通；若从前倾转为双臂交叉，说明已开启戒备状态，你的问题或提问方式让用户不舒服，此时拿不到更多细节——**还原度不高，当下就要缓解**。
3. **引导讲故事，而非发表观点**：人天生喜欢讲故事，故事里有细节、有场景、有情绪，正是洞察的土壤。
   - 平庸的提问：“您觉得我们的洗地机好用吗？”（只会得到 好用/不好用/还行）。
   - 挖矿式提问：“能和我聊聊，您最近一次用洗地机，是想清理什么东西吗？当时是在什么情况下？后来怎么样了？”
   - 技巧：多用“讲讲上次……”“回忆一下当时……”“带我回到那个场景……”引导用户进入**“故事模式”**，从中发现真实动机、遇到的障碍，以及用户自己摸索出的“野生”解决方案（Workaround）。
4. **区分“事实区”与“观点区”**：用户不是产品经理，“我认为你应该……”“这个功能可以改成……”是用户的**观点**，可以听但不能全信；要透过观点挖掘背后支撑它的**事实**。
   - 技巧：当用户提出一个解决方案时，永远多问一句“您为什么会这么想呢？”或“如果有了这个功能，会对您有什么帮助呢？”
   - 例：用户说（观点）“你们应该在首页加一个‘一键签到’按钮！”→ 追问（挖掘事实）“您是希望每天都能方便地签到吗？签到对您来说意味着什么呢？”→ 经挖掘（深层动机）发现用户并非执着于“签到”动作，而是担心错过“连续签到 7 天送优惠券”的奖励——他把“一键签到”当作解决“不想错过奖励”的方法，于是解决方案未必是加按钮，也可能是优化提醒机制。

### 框架一：用户旅程地图
用户旅程地图就是一张“故事图”：把用户为实现一个目标所经历的完整过程（从开始到结束）像讲故事一样画在一张图上，不仅讲用户做了什么（步骤），更同时揭示用户在每个环节的感受、想法、痛点以及背后的机会。
1. **定义阶段**：将用户与产品/服务交互的过程分为几个大阶段（如：需求触发、寻找入口、浏览与筛选、对比、购买决策）。
2. **填充用户行为与想法**：在每个阶段下列出用户的具体行为（Actions）和引用的原话/想法（Quotes/Thoughts）。
3. **标注情绪曲线**：画一条情绪曲线，标注用户在各环节的情绪高低，直观找到“低谷”（痛点）和“高峰”（爽点）。
4. **识别机会**：在痛点和爽点下方，直接推导出产品或服务可以如何改进或创新。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2025-12-25-11-27cPizo0F7c8LoCXj.png)

*案例：来自排行榜的用户调研，梳理不同看榜场景的用户旅程地图*

### 框架二：价值主张画布
价值主张画布就是一张“配对图”，用来检验和设计产品/服务是否和目标顾客内心真正想要的东西“严丝合缝”。核心思想：价值不是由你决定的，而是由“产品提供的”和“顾客需要的”这两者匹配的程度决定的。这张画布分为左右两部分，就像一把锁和一把钥匙：

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2025-12-25-11-28Poy8qjkpkgUJuTf.png)

1. **建立用户/客户档案**
   - 用户任务：用户试图在生活中完成什么工作？（功能性、社交性、情感性）
   - 痛点：在完成任务前后，用户遇到的困难、风险和不希望发生的负面结果。
   - 收益：用户希望得到的结果、好处和渴望的正面体验。
2. **价值地图**
   - 产品与服务：你提供的解决方案列表。
   - 痛点缓解剂：你的产品如何具体缓解上述痛点？
   - 收益创造器：你的产品如何具体创造上述收益？

### 框架三：同理心地图
同理心地图就是一张“用户内心世界的透视地图”：它强迫我们暂时放下自己的假设，站到用户立场，去推测他在某个特定情境或接触点下的所见、所想、所感和所为。

Say-Do-Think-Feel 模型：
- 他说 Say：用户在访谈中直接说出口的言论。
- 他做 Do：用户表现出的实际行为（包括观察到的或他自述的）。
- 他想 Think：用户可能没有明说但潜藏的想法、顾虑、期望。
- 他感 Feel：用户在整个体验中的情绪和感受。

产出：最终提炼出用户的“痛点”（Pains）和“收益点”（Gains）。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2025-12-25-11-288TXxxPiReVFwfdw.png)

### 框架四：A/B/C 模型
A/B/C 模型是一个心理学框架，像“拆解机器”一样拆解用户的内心活动；洞察就是通过观察这三大部件（情感、行为、认知）是否协调运作，发现那些“反常”或“矛盾”的时刻——这些时刻往往揭示了用户的深层真相。
- 情感 (Affect)：“我感觉如何？”
- 行为倾向 (Behavior Tendency)：“我会怎么做？”
- 认知 (Cognition)：“我怎样认为？”

基于 A/B/C 的分析，发现关于用户的深层真相或矛盾。

![](https://s3.cn-north-1.jdcloud-oss.com/shendengbucket1/2025-12-25-11-28A28Jq1212Yd728orD0I.png)

*案例：“我知道运动对身体好（认知-C），但我就是觉得跑步很痛苦（情感-A），所以一直没法坚持去健身房（行为-B）”*

## 定性分析的方法论原则（过程纪律）
<!-- 合并自 Joyspace「访谈技巧知识库」外部文章，2026-06-25 -->

> 上面四大框架解决“用什么结构呈现洞察”；这一节解决“分析这件事在流程上怎么做才对”——是凌驾于具体框架之上的过程论原则，决定了你能不能、以及在什么时候用好这些框架。

1. **分析非线性、从访谈伊始即开始**：定性分析不是访谈全部做完才动手的线性环节。访谈前就用现有产品数据/二手资料形成初步假设；访谈过几位用户后即形成阶段性结论，带着假设继续访谈、边采边验、不断填补漏洞。拖到项目后期才统一分析风险太高（记忆衰减、漏洞来不及补、前期假设没机会在后续访谈中验证）。
2. **分析框架在调研初期就选定**：先基于研究目标确定研究方法与分析策略（例如决定用同理心地图或用户旅程地图呈现），再据此采集相应维度的数据——确定用同理心地图，采集时就主动去问用户“说的/做的/想的/感受的”。避免数据采完才发现缺了关键维度、框架填不满。
3. **「信息对等原则」——降低信息差以提升洞察深度**：同一份访谈资料经不同人分析，结论可能天差地别，差距取决于研究者对领域与商业的理解深度。研究开始前应主动补足领域/市场、竞品、政策、商业模式、历史材料、业务规则与当前瓶颈等背景，把视角拉高到与业务决策人同等高度——唯有如此，才能从同一份访谈里洞察出直指决策的机会。

> 数据整理与报告结构不在此展开，避免与下列文档重复：
> - 访谈后的数据整理三步（转录 → 汇总 → 编码）与三进程编码细节，见 `methods/toolbox/analysis/qualitative-coding.md`；该篇“关键原则与技巧”一节对上述三条过程原则有更完整的论述（含演绎/归纳、三种效度）。
> - 研究报告的证据链结构（研究目标 → 思路与方法 → 结论汇总 → 分论点“结论+证据+建议” → 附件）与受众定位，见 `methods/standards/research-report-writing.md`；引用用户原话以增强代入感、降低沟通阻力的做法亦在该规范内。

## 结果解读
- 输出长什么样、怎么读、怎么下结论：四大框架分别产出不同形态的结构化洞察——
  - 用户旅程地图：完整服务流程 + 情绪曲线，读出“低谷=痛点 / 高峰=爽点”，并在其下推导改进/创新机会。
  - 价值主张画布：呈现“产品提供”与“顾客需要”的匹配关系，判断方案是否击中用户需求。
  - 同理心地图：从 Say/Do/Think/Feel 提炼出用户的痛点（Pains）与收益点（Gains），构建立体用户画像。
  - A/B/C 模型：聚焦单一现象/矛盾，把情感-行为-认知的“不协调”处结构化为一个深层洞察。
- 共同目标：将碎片化的信息拼接成能指导行动的“洞察蓝图”，并据此推动团队形成共识、激发创意、让产品更贴合用户需求。

## 工具 / 实现
- 这四大框架均为纸上/白板/在线协作画布可绘制的分析模板，不依赖特定软件。
- 上游配套：高质量定性资料来自深度访谈与观察，详见关联中的采集方法。

## 局限与误用
- ⚠️待补充(缺源)：原文未系统列出各框架的常见误用与边界；如需“高风险坑”清单，待补充权威来源，不在此编造。
- 选型提示（来自原文“四大模型的应用差异”）——选错框架会让庞杂的用户故事难以转化为精准、可行动的洞察：
  - 若想梳理完整的服务流程与情绪起伏，就绘制**用户体验地图**，产出旅程与痛点。
  - 若要深度共情并定义一类典型用户，就使用**同理心地图**，构建立体的用户画像。
  - 若需检验产品方案是否击中用户需求，就借助**价值主张画布**，清晰呈现匹配关系。
  - 若想对一个具体现象或矛盾进行深度解构，就运用**A/B/C 模型**，形成结构化的单一洞察。

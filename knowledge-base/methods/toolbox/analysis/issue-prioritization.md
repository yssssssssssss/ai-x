# 问题分层与优先级评估（Issue Severity & Prioritization）

> 来源：Joyspace《问题分层与优先级评估（Issue Severity & Prioritization）》 https://joyspace.jd.com/pages/VMTNZ98phOT5zxVeubKF

> 一句话：把迭代评估中分散发现的体验/可用性/转化问题归类、定级、排序，用统一标准回答"先修哪个"，让有限的迭代资源投向收益最高的修复点。

## 速览卡

| 字段 | 内容 |
| --- | --- |
| 定义 | 对一批已发现的问题做归类（分层）、定级（严重度）、排序（优先级）的结构化方法，输出可执行的修复顺序 |
| 核心逻辑 | 严重度 = 影响（impact）× 频率（frequency）× 持续性（persistence）；优先级 = 价值/影响 ÷ 成本/难度，把定性严重度与定量影响（受影响用户量、转化损失）结合后排序 |
| 适用场景 | 一轮可用性测试/访谈/VOC/漏斗诊断后积累了几十条问题，需求与缺陷挤在一起、团队对"先做什么"有分歧时 |
| 输入 | 问题清单（来自可用性测试、深度访谈、VOC、转化漏斗、Adoption 分析）、各问题的受影响用户量与业务指标损失、研发/设计工时估算 |
| 输出 | 带分层标签（P0–P3）、严重度分（0–4）、优先级分（RICE/ICE）的问题排序表，以及一张影响×难度矩阵 |
| 局限·前提 | 评分含主观成分，需校准与多人评估；严重度高≠优先级高（需叠加成本与战略）；定量数据缺失时易退化为拍脑袋 |

## 何时用

问题分层与优先级评估包含三个递进动作：**分层、定级、排序**。

适合的问题类型：一轮可用性测试/访谈/VOC/漏斗诊断后积累了几十条问题，需求与缺陷挤在一起、团队对"先做什么"有分歧时。

本方法属于【诊断归因】类的"收口"环节。前序方法（[用户深度访谈](../collection/interviews.md)、[转化漏斗分析](conversion-funnel.md)、[功能采纳 Adoption 分析](feature-adoption.md)、[VOC 用户之声分析](voc-analysis.md)、[体验度量方法](experience-metrics-heart.md)）会**产出大量分散问题**；本方法负责把它们**收敛成一份带优先级的待办**，直接对接下一轮迭代的需求池。

使用时机有三：

- 一是**评估收尾时**——一轮研究做完、问题攒够时集中分层定级；
- 二是**版本规划前**——把研究问题与产品需求放进同一张优先级表，让资源分配有据；
- 三是**线上事故后**——结合 VOC 与监控数据快速判定 P0/P1，决定是否热修。

与近似技术的区别：严重度评级只刻画"问题有多糟"，优先级框架（RICE/ICE/矩阵）才决定"先修哪个"——两者递进互补，不可互相替代。

## 输入数据要求

- 问题清单（来自可用性测试、深度访谈、VOC、转化漏斗、Adoption 分析）；
- 各问题的受影响用户量与业务指标损失（转化流失、客诉量）；
- 研发/设计工时估算（Effort，人月）。

## 分析步骤

1. **汇总去重**：把各渠道（测试、访谈、VOC、漏斗）的问题归并到一张清单，合并同因异象的重复项，每条写清"现象—影响的任务—证据来源"。
2. **分层打标**：按 P0–P3 给每条问题贴层级标签，统一团队语言。
3. **严重度定级**：按 NN/g 三因素（频率×影响×持续性）给每条打 0–4 分；关键问题由 3–5 名评估者独立打分后取均值或讨论收敛，降低主观偏差。
4. **补定量证据**：为每条问题补上 Reach（受影响用户数/占比）与业务损失（转化流失、客诉量），把定性严重度锚定到定量影响上。
5. **估算成本**：与研发/设计共同估 Effort（人月）或难度档。
6. **跑优先级模型**：对候选问题计算 RICE 或 ICE 分；问题量大时先用价值×成本矩阵做粗筛，再对高价值项做 RICE 精排。
7. **画矩阵与定线**：在影响×难度矩阵上落点，圈出"高影响×低难度"的快赢（Quick Wins）优先排期。
8. **评审定档**：与产品/业务对齐，按依赖关系和战略"必备项"做人工微调（RICE 分不应作为铁律），输出最终修复顺序。

### 分层（按问题性质归类）

先把问题按对用户任务的破坏程度归入层级，便于沟通与统一处置标准：

- **P0 阻断性**：用户无法完成核心任务（崩溃、流程走不通、关键按钮失效）——必须立即修复；
- **P1 功能性**：核心功能可用但结果错误或严重偏差，多数用户受影响；
- **P2 易用性**：能完成但费力、易出错、需绕路，影响效率与信心；
- **P3 愉悦/外观性**：文案、视觉、微交互瑕疵，不影响完成，仅影响体验质感。

### 定级（严重度评级）

业界基准来自 Nielsen Norman Group：单条问题的严重度由三个因素合成——**频率**（常见还是罕见）、**影响**（用户克服它的难易）、**持续性**（一次性受阻还是反复被困扰），并叠加对产品口碑的市场影响。NN/g 给出 0–4 五级量表（原文）：

| 分值 | 含义（NN/g 原文释义） |
| --- | --- |
| 0 | 我不认为这是一个可用性问题 |
| 1 | 仅外观问题：除非有富余时间，否则无需修复 |
| 2 | 轻微可用性问题：修复优先级低 |
| 3 | 严重可用性问题：重要、应高优先级修复 |
| 4 | 可用性灾难：产品发布前必须修复 |

合成口径：

$$\text{严重度} = \text{频率（frequency）} \times \text{影响（impact）} \times \text{持续性（persistence）}$$

### 排序（优先级框架）

严重度只刻画了"问题有多糟"，但要决定修复顺序还须引入**成本**与**触达范围**。常用框架：

- **RICE**：

  $$\text{RICE} = \frac{\text{Reach} \times \text{Impact} \times \text{Confidence}}{\text{Effort}}$$

  Reach 用真实产品指标度量"每段时间受影响的人/事件数"；Impact 取离散档（巨大 3／高 2／中 1／低 0.5／极小 0.25）；Confidence 用百分比给估计打折（高 100%／中 80%／低 50%）以抑制对"听起来很美但说不清"想法的冲动；Effort 以"人月"计，更高的成本会摊薄总收益。结果代表"单位投入产出的总影响"。

- **ICE**：

  $$\text{ICE} = \text{Impact} \times \text{Confidence} \times \text{Ease}$$

  比 RICE 少一个 Reach 维度，更轻、更快，适合早期或小批量决策。

- **价值×成本 / 影响×难度二维矩阵**：每个问题在两轴上打分形成四象限，无需公式即可快速可视化排序。

### 体验问题 ABCD 定级（广度×深度）

<!-- 合并自 神灯46470 -->

NN/g 的 0–4 量表偏"用户感受"侧的深度刻画。另有一套面向**产品体验问题**、可与 NN/g 并用的二维定级法（京东内部联合资深设计/产品及总监共建，经上千条体验问题验证）：把问题拆成**广度（影响范围）× 深度（严重程度）**两维分别打分，相乘得"产品体验指数"，再映射到 **A/B/C/D 四级**。它弥补了"研究偏深度、产品偏广度，两角色定级口径不一"的分歧，用统一的思考框架降低主观误差。

**维度一·广度（影响范围）**

$$\text{影响范围} = \frac{\text{进入该任务的人数}}{\text{页面总 UV}} \times 100\%$$

按占比划分边界值赋 **1/2/3 分**，占比越高分越高（边界百分比可按自家产品实际调整）。这是相对可量化的指标。

**维度二·深度（严重程度）**

参照产品设计流程把"严重程度"拆为**需求层**与**设计层**，再细分为四个评判维度，按"需求→设计"逐层、依次评分，**只对存在问题的维度打分，多个维度都有分时只计最高分**：

- **功能定义**（需求层）：需求是否合理——是否属于页面核心价值主张内的基础功能，或竞争趋势下应扩充的功能；
- **策略逻辑**（需求层）：业务/产品逻辑是否合理——是否有利于传递产品核心信息、满足核心用户利益；
- **信息展示**（设计层）：信息布局问题——是否符合用户认知习惯、是否清晰有主次；
- **操作流程**（设计层）：交互操作问题——操作是否符合用户预期、是否有明确指引、产品与用户互动是否高效。

每个细分维度另有更具体的分档标准（原文以表给出）以进一步缩小误差。

**合成与定级**

$$\text{产品体验指数} = \text{广度（影响范围）} \times \text{深度（严重程度）}$$

按指数区间映射到 **A / B / C / D** 四个体验问题等级（区间映射见原文）。该框架不止用于用户反馈跟进，也适用于产品走查、用户调研等各渠道搜集的体验问题管理，并可向其他问题类型延展。

> 与上文 NN/g 量表的关系：两者均可用于"定级"环节——NN/g 重单条问题的用户感受严重度，本法把"广度"显式纳入并落到 ABCD，便于研究与产品对齐口径；可择一或并用。

## 结果解读

输出是一份带分层标签（P0–P3）、严重度分（0–4）、优先级分（RICE/ICE）的问题排序表，以及一张影响×难度矩阵。

读法：分层标签统一团队语言（P0 立即修、P3 有空再修）；严重度分刻画"问题有多糟"；RICE/ICE 分代表"单位投入的总影响"，分越高越该优先排期；矩阵四象限给出无需公式的快速可视化结论——"高影响×低难度"的快赢（Quick Wins）优先排期。最终顺序须经评审，结合依赖关系与战略"必备项"做人工微调。

## 局限与误用

- **只数频次**：把"出现次数最多"等同于"最该修"，忽略单次影响极大的阻断性问题（低频高影响往往是 P0）。
- **只凭主观**：单人拍脑袋打严重度，缺乏多人校准与定量证据，结论难服众。
- **严重度=优先级**：误把严重度直接当修复顺序，忽略 Effort 与 Reach——一个 4 分但需 8 人月的问题，可能要让位于多个 3 分快赢。
- **公式当铁律**：照搬 RICE/ICE 排名而无视依赖关系与"必备项"，导致漏排前置条件。
- **Confidence 缺失**：对数据薄弱的估计不打折，让"听起来诱人"的点虚高占位。
- **分层标准飘移**：团队对 P0/P1 没有统一定义，同一问题不同人定级不一，沟通成本激增。

- **上游供料**：[转化漏斗分析](conversion-funnel.md)、[功能采纳 Adoption 分析](feature-adoption.md)、[VOC 用户之声分析](voc-analysis.md)、[用户深度访谈](../collection/interviews.md)、[体验度量方法](experience-metrics-heart.md) 持续产出问题清单，本方法负责归类定级排序。

# 转化漏斗分析（Conversion Funnel Analysis）

> 来源：Joyspace《转化漏斗分析（Conversion Funnel Analysis）》 https://joyspace.jd.com/pages/FGs77Vxb2KcnLvUSCPAD

> 一句话：把一条关键转化路径拆成有序步骤、逐步算转化率与流失率，定位最大流失环节，是迭代期"线上转化问题"诊断归因的主力量化方法。

## 何时用

转化漏斗分析属于【诊断归因】一类，核心任务是定位"线上转化问题"和"迭代后新引入的体验问题"。它的本质是把"完成一件事"看成一条有序事件序列，逐级衰减、定位最大流失环节。

- **适合的问题类型**：注册开通、下单支付、新功能首次使用、内容发布等有明确目标且路径相对固定的关键流程；版本迭代后评估线上转化是否变差或被新体验问题拖累。

- **典型时机有三个**：
  - **版本上线后的监控期**：把核心路径漏斗作为常规看板，一旦某步转化率较基线明显下滑，立即预警，判断是否为新版本引入的退化。
  - **明确知道"转化变差"但不知病灶时**：用漏斗快速收敛到具体环节与具体人群，再决定投入哪种定性方法深挖。
  - **改版评估闭环**：迭代针对某一步做了优化后，用漏斗对比改动前后的分步转化率，量化收益（与[体验度量方法](experience-metrics-heart.md)配合形成度量闭环）。

- **与近似技术的区别**：漏斗只回答"在哪流失、流失多少"，不直接给"体验好不好"的主观结论——那是[满意度问卷设计与数据清洗](../collection/satisfaction-survey.md)和[期望确认理论 ECT](../../../models/ect.md)的职责。漏斗也不直接回答"为什么"，归因须接定性方法。新功能的"发现→首次使用→持续使用"本身就是一条漏斗，与[功能采纳 Adoption 分析](feature-adoption.md)方法同源、常配合使用；当用户实际走法发散时应先做路径分析、再决定是否建漏斗。漏斗是诊断的入口，不是终点。

## 输入数据要求

- 埋点事件序列、用户/会话标识、时间戳、分群维度（渠道、设备、新老用户、版本）。
- 依赖埋点完整准确、步骤口径清晰：确认每一步都有对应事件、参数完整、无重复触发、无漏报，否则漏斗会失真。
- 路径相对固定且可线性化；强行把发散路径塞进线性漏斗会失真。

## 分析步骤

1. **定义转化目标与路径口径**：明确宏转化目标，约定开放/封闭漏斗、时间窗、每用户/每会话计一次、是否允许中间穿插其他行为（GA4 的"直接跟随/间接跟随"）。
2. **拆解有序步骤**：把路径拆成 3–7 个互斥且必经的步骤，给每步起清晰名字，用中性步骤名而非具体字段名（示例：访问落地页 → 填写注册信息 → 验证身份 → 完成开户 → 首次下单）。
3. **校验埋点**：确认每一步都有对应事件、参数完整、无重复触发、无漏报，否则漏斗会失真。
4. **计算分步转化率与流失率**：得到整体漏斗图，标出绝对人数与相对流失最大的环节。
5. **分群下钻**：按渠道、设备、新老用户、版本拆分，定位异常段，区分"普遍问题"与"特定人群问题"。
6. **形成归因假设**：对最大流失环节，按五类流失原因列出疑似根因。
7. **定性求证**：调录屏、做可用性测试、约[深访](../collection/interviews.md)、看 [VOC](voc-analysis.md)，验证或推翻假设。
8. **给出优化建议并回测**：上线改动后用同口径漏斗对比验证收益。

## 结果解读

漏斗分析的本质，是把"完成一件事"看成一条有序事件序列。以 Amplitude 的免费试用漏斗为例：打开邮件 → 点击 CTA → 创建账户 → 转为付费；每一步都会损失一部分用户，沉到底部的才是最终转化者。

**分步转化率与流失率**是两个最基本的量。设第 $n$ 步进入人数为 $U_n$，则该步转化率与流失率为：

$$\text{转化率}_n = \frac{U_{n+1}}{U_n}, \qquad \text{流失率}_n = 1 - \text{转化率}_n$$

整条路径的**总转化率**等于各步转化率的连乘：

$$\text{总转化率} = \prod_{n} \text{转化率}_n = \frac{U_{2}}{U_1} \times \frac{U_{3}}{U_2} \times \cdots$$

因此任何一步的小幅下滑都会被逐级放大。诊断时不能只盯绝对人数，而要看**相对流失**：哪一步的转化率明显低于其前后步骤、低于历史基线、低于同类流程的经验值，那里就是最值得动手的杠杆点。Amplitude 的说法很直接——若某一步出现严重流失，你就知道该把精力集中在哪。

**宏转化（macro）与微转化（micro）**要分层看：

| 类型 | 含义 | 特征 |
| --- | --- | --- |
| 宏转化（macro） | 真正有商业价值的目标动作（如完成下单、完成开通） | 路径终点，转化率通常较低 |
| 微转化（micro） | 路径上"做出增量进展"的次级动作（如点击链接、看完一段视频、滑过首屏） | 本身未必有价值，但能表明某种参与度，转化率通常高于宏转化 |

NN/g 指出，微转化"本身未必有价值，但能表明某种参与度"，且通常转化率高于宏转化。把微转化作为漏斗中间步骤，能让长链路获得更细的观测粒度，避免只看首尾两点而错失中段问题。

**分群/细分对比**是把"平均的谎言"拆穿的关键手段。整体转化率往往掩盖了结构差异：NN/g 举过一个典型例子——一次非目标人群的流量激增会把整体转化率拉低，原因不在产品而在流量结构。把漏斗按**渠道来源、设备、新老用户、版本号**等维度拆开（GA4 称为 Breakdown），常能定位到"只有某一段人群在某一步异常流失"，从而把问题收敛到具体场景。

漏斗只能告诉你"在哪流失、流失多少"，**回答不了"为什么"**。要做归因，必须把定量漏斗与定性证据结合：对最大流失环节调取**会话录屏**、做**可用性测试**、跑**用户深度访谈**、汇总 **VOC 用户之声**。Baymard 对结算流程的研究是量化+定性结合的范例：在线购物车平均放弃率高达 **70.22%**（取自 50 项研究），剔除"只是逛逛"后，可改善的主要原因占比如下：

| 主要放弃原因 | 占比 |
| --- | --- |
| 额外费用过高（运费/税费/手续费） | 39% |
| 不信任站点处理支付信息 | 19% |
| 被强制创建账户 | 19% |
| 结算流程过长过复杂 | 18% |

这些正好对应一组可复用的**流失原因清单**。

### Fogg 行为模型（B=MAP）：把流失原因结构化

借 BJ Fogg 行为模型可把流失原因结构化：一个行为要发生，**动机（Motivation）、能力（Ability）、提示（Prompt）**必须同时具备；缺一即不发生。映射到漏斗某一步的流失，常见根因有五类：

| 流失原因类别 | 对应 B=MAP | 含义 |
| --- | --- | --- |
| **认知不足** | 提示缺失（Prompt） | 不知道下一步要做什么 |
| **动机不够** | 动机（Motivation） | 不觉得值得继续 |
| **操作能力门槛** | 能力（Ability） | 步骤太难、字段太多、要切换设备 |
| **信任顾虑** | 动机/能力 | 对安全/隐私/资质存疑 |
| **技术故障** | 能力（Ability） | 报错、卡顿、加载失败 |

把每一步的疑似根因落到这五类上，定性研究就有了明确的求证靶子。

## 可视化 / 出图

- **漏斗图**：逐步展示进入人数与分步转化率/流失率，标出绝对人数与相对流失最大的环节。
- **分群对比矩阵**：以"步骤"为行、"分群维度值"为列，填入各格转化率，颜色标出明显低于行均值的格子，快速锁定异常人群×异常环节的交叉点。
- **到达耗时拆解**：按 GA4 的能力，可把步与步之间或整条漏斗的完成耗时按小时/天/周拆解，辅助判断时间窗设置是否合理。

## 工具 / 实现

- **GA4 Funnel exploration**：支持封闭漏斗（用户必须从第一步进入）与开放漏斗（允许从任意步骤进入）；用 "Within" 设定步间或整条漏斗的完成时限，并可按小时/天/周拆解到达耗时；分群归因和耗时计算按用户"首次"实例处理；提供 Breakdown 维度拆分。
- **Amplitude**：漏斗分析，定位严重流失步骤。
- **会话录屏 / 错误监控**：用于最大流失环节的定性下钻与技术故障排查。

> **漏斗定义清单（开工前逐项确认）**

| 项 | 取值/说明 |
| --- | --- |
| 宏转化目标 | 例：完成开户并首次下单 |
| 步骤序列 | 3–7 步，互斥、必经、用中性名 |
| 漏斗类型 | 开放 / 封闭 |
| 时间窗 | 整体或步间时限（如 7 天内完成） |
| 计数口径 | 每用户一次 / 每会话一次 |
| 顺序要求 | 直接跟随 / 间接跟随 |
| 分群维度 | 渠道、设备、新老、版本 |

漏斗口径需事先约定，否则结论可能相反：

- **开放 vs 封闭**：封闭漏斗要求用户必须从第一步进入，开放漏斗允许从任意步骤进入；两者算出的人数与转化率不同。
- **转化时间窗**：窗口设得太短会把"慢决策但终将转化"的用户误判为流失，太长则会把不相关行为也算进来。
- **重入**：同一用户在统计周期内多次走流程时，工具通常只取"首次"计入（GA4 在分群归因和耗时计算上都按首次实例处理）；分析时要明确每用户计一次还是每会话计一次。

> **诊断假设清单（按 Fogg 五类）**

| 流失原因类别 | 观测信号 | 求证方法 |
| --- | --- | --- |
| 认知不足（缺提示） | 在该步长时间停留后退出、反复返回上一步 | 录屏、可用性测试 |
| 动机不够 | 进入即跳出、跳过非必填 | 深访、VOC |
| 操作能力门槛 | 表单报错率高、需切换设备 | 录屏、表单字段分析 |
| 信任顾虑 | 在涉及隐私/资质处停滞 | 深访、问卷 |
| 技术故障 | 错误日志、特定设备/版本集中流失 | 错误监控、分群漏斗 |

## 局限与误用

- **只看绝对人数不看相对转化率**：基数大的步骤流失人数自然多，未必是病灶；要看相对流失。
- **口径未约定就开算**：开放/封闭、时间窗、计数方式不同，结论可能相反；同口径才能跨期对比。
- **把发散路径硬塞进线性漏斗**：用户实际走法多样时，强行线性化会制造假流失，应先做路径分析再决定是否建漏斗。
- **时间窗设置不当**：窗太短把慢决策用户误判为流失，窗太长混入无关行为。
- **忽视分群、被平均值误导**：整体平稳可能掩盖某一渠道/设备的严重退化。
- **只做漏斗不做归因**：知道"在哪流失"却不查"为什么"，无法形成可执行的优化动作。
- **埋点不可信仍强行解读**：漏报/重复触发会让漏斗失真，分析前必须校验数据。

## 关联

- 理论根源（model）：[Fogg 行为模型 B=MAP](../../../models/fogg-behavior-model.md) ⚠️ 待补充(缺源) —— 用于把流失原因结构化为动机/能力/提示三要素与五类根因。
- 常配合的采集方法：
  - [用户深度访谈](../collection/interviews.md)  —— 对最大流失环节做归因时，深访是验证 Fogg 五类根因的主力定性手段。
  - [VOC 用户之声分析](voc-analysis.md) —— 上线后从评价、客服、反馈中聚合流失抱怨，与漏斗异常段相互印证。
  - [问题分层与优先级评估](issue-prioritization.md) —— 漏斗定位出的多个流失环节，需按影响面与严重度排序，决定先修哪个。
  - [A/B 测试与在线对照实验](ab-testing.md) —— 针对某步的优化方案，用 A/B 实验严格验证对该步转化率的因果提升。
  - [体验度量方法 HEART](experience-metrics-heart.md) —— 把分步转化率纳入长期度量看板，量化迭代收益。
  - [功能采纳 Adoption 分析](feature-adoption.md) —— 新功能的"发现→首次使用→持续使用"本身就是一条漏斗，两者方法同源、常配合使用。
